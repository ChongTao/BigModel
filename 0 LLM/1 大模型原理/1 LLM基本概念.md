# 一 LLM 基本概念
## 1.1 什么是 LLM

LLM（Large Language Model，大语言模型）通常指：

- 在海量语料上训练
- 参数规模很大
- 以语言建模目标为核心
- 常常建立在 Transformer 或其变体之上

的通用语言模型。

它最基础的工作方式不是“先想完整答案”，而是：

**根据当前上下文，预测下一个最可能的 Token。**

## 1.2 Token

Token 是大模型处理文本时的最小离散单位。它既不完全等于“一个字”，也不完全等于“一个单词”，而是由分词器和词表共同决定的文本片段。

例如：

- 中文里，一个汉字有时是一个 Token。
- 常见词组、标点、空格，也可能被单独编码成 Token。
- 英文单词则可能被拆成多个子词片段。

可以把 Token 理解为模型眼中的“文本积木”。

### 1.2.1 Token 的三种常见形态

同一个 Token 在模型链路里通常会经历三种形态：

1. 文本片段：例如 `"我"`、`"模型"`、`"un"`、`"able"`
2. 整数编号：即 `token id`
3. 向量表示：进入神经网络后的浮点数向量

要特别区分：

- Token 本身不是向量。
- Token 本身也不是概率。
- Token 最初只是被切分出来的文本单位及其编号。

## 1.3 向量化

模型不能直接对文字本身做神经网络计算，所以需要先把 Token 变成向量。这一步通常叫做 Embedding，也就是向量化。

例如：

```text
token = "模型"
token_id = 18532
```

经过向量化后，可能变成：

```text
[0.12, -0.84, 1.07, ..., 0.33]
```

这串高维浮点数就是模型可以真正处理的输入表示。

如果只看概念层面，可以先记住：

- Token 解决“文本如何切开”
- 向量化解决“离散符号如何变成可计算对象”

更完整的向量化内容见 [3 Embedding.md](./3%20Embedding.md)。

## 1.4 上下文窗口

大模型不是无限读文本的。它一次调用能处理的内容上限，本质上是 **Token 上限**，这就是上下文窗口。

上下文窗口决定了：

- 一次最多能输入多少内容
- 多轮对话能保留多少历史
- 长文档和 RAG 信息能装进去多少
- 调用成本和延迟会不会快速上升

因此，很多工程问题表面上看是“模型不够聪明”，其实本质上是“上下文装不下”或“上下文太乱”。

## 1.5 下一个 Token 预测

LLM 的基础训练目标通常非常朴素：给定前文，预测下一个最可能的 Token。

例如输入：

```text
今天天气很
```

模型不会直接给出整句答案，而是先输出下一步的概率分布，例如：

```text
晴: 0.42
好: 0.31
热: 0.08
冷: 0.05
```

然后从这些候选里选出一个 Token，再继续预测下一个。

所以，大模型本质上是逐 Token 自回归生成的，不是一次性把整段答案“想完再输出”。

## 1.6 采样与输出风格

模型算出的是概率分布，但最终输出哪个 Token，还要经过采样策略。

这时候会涉及：

- greedy
- top-k
- top-p
- temperature

这些策略决定的是“怎么选”，而不是“模型知道多少”。

例如：

- 低温度更稳定、更保守
- 高温度更多样、更发散

更详细内容见 [4 采样与解码.md](./4%20%E9%87%87%E6%A0%B7%E4%B8%8E%E8%A7%A3%E7%A0%81.md)。

## 1.7 为什么 LLM 会显得“很聪明”

表面上看，大模型只是不断预测下一个 Token；但为了把这个预测做准，模型必须在训练中学习很多更深层的规律，例如：

- 语法结构
- 词语搭配
- 事实共现
- 常识关联
- 某些局部推理模式

因此，大模型表现出的“理解感”“知识感”“推理感”，很大程度上来自这些统计规律在大规模训练中的压缩表达。

## 1.8 LLM 不是什么

理解 LLM 时，也要同时知道它不是什么：

- 它不是数据库：
  不保证事实永远准确、最新。
- 它不是搜索引擎：
  没有外部检索时，不能天然获取新信息。
- 它不是规则系统：
  输出不是严格可证明的逻辑推导结果。

所以在应用层，LLM 往往还要结合：

- Prompt
- 上下文工程
- RAG
- 工具调用
- 运行时约束
