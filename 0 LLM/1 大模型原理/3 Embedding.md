# 一 Embedding 是什么

Embedding 可以理解为把离散对象映射到连续向量空间中的表示方法。对于大模型和检索系统来说，最常见的对象是文本，也可以是图像、音频、代码等。

它的目标不是直接生成答案，而是让系统能够在数学空间中表示“相似、相关、接近、可比较”的关系。

例如：

- 语义接近的句子，向量距离通常更近。
- 主题相近的文档，向量分布通常更接近。
- 同一个词在不同上下文中的表示，可能并不完全一样。

Embedding 是很多能力的基础组件，例如：

- 语义搜索
- 向量检索
- 聚类与分类
- 推荐系统
- 文本相似度计算
- RAG 中的语义召回

但要注意：**Embedding 是一个宽泛概念，既可以指最初的“查表向量”，也可以指面向检索输出的句向量或文档向量。**

如果不先把这几个层次分开，后面很容易把不同概念混在一起。

# 二 文本是如何被变成向量的

从大模型视角看，文本变成向量通常会经过这些步骤：

1. 分词：把文本切成 token。
2. 编号：每个 token 映射到词表中的 id。
3. 查表：每个 token id 对应一个初始向量表示。
4. 编码：经过模型层处理，得到上下文化表示。
5. 池化或投影：把 token 级表示变成句子级、段落级或文档级向量。

需要注意的是：

- `token embedding`：通常指第 3 步查表得到的初始向量。
- `hidden states`：通常指第 4 步经过 Transformer 等模型层后的上下文化表示。
- `text embedding`：通常指第 5 步得到的句子或文档级向量，常用于检索、匹配、相似度计算。

三者相关，但不完全等价。

## 2.1 三个容易混淆的概念

### 2.1.1 Token Embedding

这是最狭义的 embedding，意思是：

`token id -> 向量`

例如词表中的某个 token id，查表后得到一个固定维度的向量。这一步更像“把离散符号映射到可计算空间”。

### 2.1.2 上下文化表示

同一个 token 进入 Transformer 后，会结合上下文不断更新表示。

所以：

- 初始 `token embedding` 可以是静态查表结果
- 经过模型层后的表示则会受到上下文影响

这也是为什么同一个词在不同句子里，最终表示可能不同。

### 2.1.3 Text Embedding

很多检索模型不会直接拿单个 token 的表示做检索，而是会把整句话、整段文本压成一个向量。

这个向量通常就是大家在 RAG、向量数据库、语义检索里说的 `embedding`。

因此在工程语境里，大家口头说的“做 embedding”，很多时候其实是：

**把一段文本编码成一个可用于相似度检索的句向量或文档向量。**

# 三 稀疏向量与稠密向量

## 3.1 稀疏向量

稀疏向量（Sparse Vector）是指向量中大多数元素都为 0，只有少量位置非零。

例如，一个 1000 维向量里只有 10 个位置非零，它就是典型的稀疏向量。

稀疏向量常见于：

- One-hot 表示
- TF-IDF
- BM25 的扩展表示
- 一些基于词项权重的检索模型

由于大多数元素为 0，稀疏向量通常不会直接完整存储，而是只保存：

- 非零元素的值
- 非零元素的位置

常见存储方式包括：

1. 坐标形式：存值和索引。
2. 压缩稀疏行：常用于矩阵，也可扩展到向量。
3. 字典形式：直接存储索引到数值的映射。

例如，向量 `[0, 0, 1, 0, 0, 2]` 可以写成：

- 非零值：`[1, 2]`
- 索引：`[2, 5]`

或者：

```json
{
  "2": 1,
  "5": 2
}
```

## 3.2 稠密向量

稠密向量（Dense Vector）是指大多数维度都有非零值的向量表示。

在现代深度学习中，文本 embedding 通常就是稠密向量。相比稀疏表示，它的特点是：

- 维度通常较低但每一维都参与表达。
- 更适合表示语义相似度。
- 常用于向量检索、聚类、推荐、语义匹配。

稠密向量的质量高度依赖模型本身。如果任务是通用语义检索，可以使用通用 embedding 模型；如果任务是垂直领域检索，有时需要专门训练或微调。

要特别注意：**检索 embedding 追求的是“相似文本在向量空间里更接近”，它优化的重点不是生成质量，而是表示质量。**

# 四 相似度是怎么计算的

Embedding 的价值之一，是让系统可以直接计算“近不近”。

常见相似度计算方式包括：

- 点积（Dot Product）
- 余弦相似度（Cosine Similarity）
- 欧氏距离（Euclidean Distance）

在检索场景中，余弦相似度和点积都很常见：

- 余弦相似度更关注方向是否接近
- 点积既和方向有关，也和向量长度有关
- 如果向量预先做了归一化，点积和余弦相似度会非常接近

欧氏距离也能使用，但在文本语义检索中通常不如前两者常见。

## 4.1 为什么“距离近”可以代表“语义近”

Embedding 模型训练的目标之一，就是让语义上更接近的文本，在向量空间中也更接近。

例如：

- “苹果手机性能很好”
- “这部 iPhone 跑分很强”

如果模型训练得足够好，这两句话虽然字面不同，但向量可能会比较接近。

这就是语义检索能够工作的基础。

## 4.2 稀疏向量的计算特点

对于稀疏向量，也常会涉及：

- 向量加法
- 向量点积
- 稀疏矩阵与向量乘法

这类运算的重点在于：只计算非零部分，从而减少内存和计算开销。

# 五 Embedding 在大模型体系里的位置

从知识体系上看，Embedding 更偏底层表示学习能力，而不是生成能力本身。

它和以下内容密切相关：

- Transformer 中的 token 表示
- 上下文化编码
- 表征学习
- 检索与匹配

它和以下内容需要区分：

- Prompt：决定模型如何被指令化
- Generation：决定模型如何生成输出
- Fine-tuning：决定模型如何通过训练适配任务

所以更准确地说，Embedding 属于“大模型原理和表示层能力”的一部分，而不是纯粹的应用技巧。

# 六 Embedding 与 RAG 的关系

Embedding 本身是底层能力，而向量检索是它在 RAG 中的典型应用。

可以这样理解两者关系：

- Embedding 解决“文本如何表示成向量”。
- 向量检索解决“如何用这些向量找到相关内容”。
- RAG 解决“找到内容后，如何组织并交给模型回答问题”。

因此在知识组织上，把 `Embedding` 放在 `LLM` 或表示学习相关目录中，把 `向量检索` 放在 `RAG` 中，是比较清晰的划分。
