# 一 强化学习在大模型里是做什么的

在大模型训练流程里，强化学习（Reinforcement Learning, RL）通常不是最开始的那一步，而是出现在**预训练和监督微调之后**，用于进一步优化模型的行为方式。

可以先把大致流程理解成：

1. 预训练：让模型学会语言、知识和基本生成能力
2. SFT（监督微调）：让模型学会按指令回答
3. 强化学习或偏好优化：让模型更符合“什么样的回答更好”

所以，强化学习在 LLM 中更像是在回答下面这些问题：

- 回答是否更符合人类偏好
- 推理过程是否更稳定
- 工具调用是否更有效
- 长链任务中是否更容易得到高质量结果

它解决的不是“模型有没有基础知识”，而是**在已有能力之上，如何让模型更倾向于产生高质量行为。**

## 1.1 为什么有了 SFT 还需要强化学习

SFT 的做法，本质上是让模型模仿标注答案。

这很重要，但它有几个天然限制：

- 标准答案不总是唯一
- “更好”往往是相对概念，而不是唯一正确标签
- 很多任务更适合比较两个答案谁更好，而不是直接写出唯一参考答案

例如：

- 两个回答都基本正确，但一个更清晰
- 两个解题过程都能得出答案，但一个更稳
- 两个 Agent 轨迹都完成任务，但一个工具调用更少

这些情况里，“偏好”往往比“标准答案”更重要，这也是为什么后面会引入偏好学习、奖励建模和强化学习。

## 1.2 在 LLM 里如何理解强化学习

把传统强化学习概念映射到大模型里，可以粗略理解为：

- `policy`：当前模型本身，也就是“怎么生成 token / 回复”的策略
- `action`：模型生成的 token、整段回答，或一次工具调用
- `environment`：任务、上下文、工具反馈、评测器
- `reward`：这个回答或行为到底好不好

在机器人里，动作可能是“往前走一步”；在 LLM 里，动作更常见的是“生成一段回答”或“完成一条推理轨迹”。

## 1.3 Reward 为什么难

强化学习能不能有效，核心取决于奖励信号。

但在语言模型里，奖励往往并不容易定义：

- 正确性有时难自动判断
- 人类偏好本身可能主观
- 长回答里，问题可能出在局部，但奖励通常只在整体上给出

所以很多 LLM 强化学习方法，本质上都在解决两个问题：

1. 如何得到更可用的奖励
2. 如何更稳定地用这些奖励更新模型

# 二 RLHF 与偏好优化主线

## 2.1 RLHF 是什么

RLHF（Reinforcement Learning from Human Feedback）可以理解为：

**用人类反馈或其近似形式，来训练模型更偏向人类喜欢的回答。**

常见流程是：

1. 先做 SFT，得到一个能基本遵循指令的模型
2. 收集偏好数据，例如“A 回答比 B 回答更好”
3. 训练奖励模型，或直接利用偏好数据构造优化目标
4. 再用 PPO、GRPO 等方法更新模型策略

## 2.2 奖励模型是什么

奖励模型（Reward Model）不是负责生成答案的主模型，而是一个“打分器”。

它的作用是：给模型生成的候选回答打分，表示“这条回答有多好”。

例如同一个问题有两个回答：

- 回答 A：更清晰、更安全、更符合要求
- 回答 B：虽然也能答，但更啰嗦或更偏题

偏好数据会告诉系统“A 比 B 更好”，奖励模型就尝试学会这种打分规律。

## 2.3 Rollout 怎么理解

在 LLM 强化学习里，rollout 通常指：

**让当前模型针对一个输入实际生成回答，然后拿这个回答去评估奖励。**

可以把它理解为：

1. 给模型一个 prompt
2. 让模型生成一个或多个回答
3. 根据规则、奖励模型或任务结果给分
4. 再据此更新模型

这和前面讲的采样也有关系：如果模型生成缺乏多样性，很多 RL 方法就更难形成有效对比。

## 2.4 DPO 与 RLHF 的关系

DPO（Direct Preference Optimization）常和 RLHF 放在一起讨论，但它和 PPO / GRPO 不完全是一类方法。

可以简单理解为：

- PPO / GRPO：更像“显式强化学习”，会基于 reward 去优化策略
- DPO：更像“直接用偏好数据优化策略”，不一定显式训练奖励模型，也不一定走完整 RL 回路

所以：

- DPO 属于偏好优化方法
- PPO / GRPO 更接近严格意义上的策略优化方法

在工程实践里，这些方法常常被统一归入“对齐阶段”的方法栈。

# 三 常见方法怎么理解

## 3.1 PPO

PPO（Proximal Policy Optimization）是最经典的策略优化方法之一。

它的核心思想是：

**更新策略，但不要一次改得太猛。**

因为如果模型每次更新幅度过大，训练可能会变得非常不稳定。PPO 通过裁剪机制限制更新步幅，让训练更稳。

在 LLM 里，PPO 常被视为早期 RLHF 的代表性方法。它的优点是：

- 思路成熟
- 训练相对稳定
- 容易作为基线方法理解

但它的代价也比较明显：

- 训练链路更重
- 常需要 value / critic 相关组件
- 对大模型来说显存和工程成本较高

## 3.2 GRPO

GRPO（Group Relative Policy Optimization）可以看作对 PPO 类方法的一种简化和改进。

它的核心思路是：

**不给单个回答绝对打分，而是让一组回答彼此比较。**

例如针对同一道题，模型生成多条回答，再看哪些回答相对更好、哪些相对更差，然后基于这种组内相对优势来更新策略。

这样做的好处通常是：

- 不必像 PPO 那样强依赖 critic
- 训练流程更轻
- 对推理类任务尤其有吸引力

它的风险则在于：

- 很依赖采样多样性
- 如果同组样本都差不多，训练信号会变弱

## 3.3 GSPO

GSPO（Group Sequence Policy Optimization）可以理解为：在组内相对优化的基础上，更强调**序列整体质量**，而不只看局部 token。

这种思路适合的场景是：

- 长回答
- 长链推理
- 更关注整体结构、整体逻辑的任务

和 GRPO 相比，它更强调“整段输出是否更好”，而不是局部位置上的优化信号。

## 3.4 DAPO

DAPO（Decoupled Clip and Dynamic Sampling）更偏工程优化思路。

它关注的重点不是只换一个目标函数，而是：

- 如何让训练样本更有效
- 如何减少无效更新
- 如何把算力集中到真正有学习价值的样本上

所以它更像是在策略优化框架上的工程化增强。

## 3.5 BAPO

BAPO（Balanced Policy Optimization）更关注训练中的平衡问题，例如：

- 正负样本影响是否失衡
- 旧数据能否被更好利用
- 模型是否因为奖励偏置而变得过度保守

这类方法的重点通常是提升样本利用率，并避免训练后模型多样性明显下降。

## 3.6 ARPO

ARPO（Agentic Reinforced Policy Optimization）更偏向 Agent 场景。

它优化的不只是“一段文本好不好”，还会更关注：

- 工具调用是否合适
- 多轮决策是否有效
- 关键步骤是否做对

因此它更适合：

- Tool Use
- 多轮任务
- 长流程决策

也就是说，ARPO 这类方法已经不只是优化“说得好不好”，而是在优化“做得对不对”。

# 四 这些方法之间是什么关系

可以把它们先粗略分成三层：

## 4.1 偏好优化层

代表方法：DPO

重点是：直接利用偏好数据，让模型更倾向于输出“人类更喜欢的回答”。

## 4.2 策略优化层

代表方法：PPO、GRPO、GSPO

重点是：根据 reward 或相对优势信号，对当前策略做持续更新。

## 4.3 工程增强层

代表方法：DAPO、BAPO、ARPO

重点是：让训练更高效、更稳定，或者更贴合特定场景，例如 Agent。

所以不要把这些名字看成彼此完全平级的“替代品”。更准确地说，它们解决的问题层次并不完全一样：

- 有些方法在解决“怎么根据偏好更新模型”
- 有些方法在解决“怎么让策略更新更稳”
- 有些方法在解决“怎么把 RL 更好落到具体场景”

# 五 强化学习不会替代基础模型能力

要特别区分：

- 预训练决定模型有没有基础知识和语言能力
- SFT 决定模型会不会按指令做事
- 强化学习或偏好优化决定模型更偏向哪种行为

如果基础模型本身没有能力，后面的 RL 也不会凭空创造知识。

更准确地说，强化学习更像是在已有能力之上做：

- 对齐
- 行为塑形
- 推理强化
- 工具使用优化
