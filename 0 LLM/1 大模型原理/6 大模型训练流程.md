# 一 大模型完整训练链路

如果只看单个概念，`预训练`、`SFT`、`偏好对齐`、`部署` 很容易被理解成彼此独立；但从系统视角看，它们通常属于一条连续链路。

可以先用一句话概括：

`数据准备 -> 预训练 -> 指令微调 -> 偏好对齐 -> 评测 -> 推理部署`

这条链路里，每一阶段解决的问题并不一样：

- 预训练：让模型获得语言能力、知识和通用表示能力
- 指令微调：让模型学会“按要求回答”
- 偏好对齐：让模型更符合人类偏好和目标行为
- 推理部署：让模型在真实系统里可用、可控、可负担

## 1.1 为什么“预训练”最关键

对大模型来说，预训练通常是成本最高、数据量最大、决定能力上限的一步。

因为：

- 模型的大部分知识来自预训练阶段
- 语言流畅性和通用推理能力主要在这一阶段形成
- 后续 SFT 和对齐更多是在“塑形”，而不是从零创造能力

所以更准确地说：

- 预训练决定“模型会不会”
- 微调和对齐决定“模型更倾向于怎么做”

# 二 预训练流程在做什么

## 2.1 数据准备

预训练的第一步不是直接开始训练，而是先准备大规模语料。

常见数据来源包括：

- 网页文本
- 书籍
- 代码
- 论文
- 问答语料
- 多语言文本

数据准备通常要做的事情包括：

- 去重
- 清洗脏数据
- 过滤低质量文本
- 去除明显无意义内容
- 控制不同语料的配比

这一阶段非常重要，因为：

- 垃圾数据会直接污染模型能力
- 数据分布会影响模型的知识结构和风格
- 数据质量问题会在后面被成倍放大

## 2.2 Tokenization

原始文本不能直接送进神经网络，所以要先经过分词器，切成 token。

这一步和前面的 [1 LLM基本概念.md](./1%20LLM%E5%9F%BA%E6%9C%AC%E6%A6%82%E5%BF%B5.md)、[3 Embedding.md](./3%20Embedding.md) 是连起来的：

1. 文本先被切成 token
2. token 再映射成 id
3. id 再查表变成向量
4. 向量进入 Transformer 计算

所以 tokenization 是训练链路里很靠前、但影响很深的一步。

## 2.3 训练目标

对于自回归语言模型来说，最常见的预训练目标是：

**给定前文，预测下一个 token。**

例如：

```text
输入：今天天气很
目标：预测下一个 token
```

模型不会直接记一句完整标准答案，而是不断重复下面这个过程：

1. 读入一段上下文
2. 预测下一个 token 的概率分布
3. 计算预测和真实 token 之间的差距
4. 反向传播，更新参数

## 2.4 Loss 是怎么来的

预训练时，模型每一步都会输出一个对整个词表的预测分布。

如果真实下一个 token 是某个词，而模型给它的概率太低，就会产生更大的损失（loss）。

所以从直觉上看：

- 预测越准，loss 越低
- 预测越偏，loss 越高

训练的目标，就是不断让平均 loss 降下来。

## 2.5 训练为什么这么贵

预训练成本高，主要因为它同时消耗：

- 海量数据
- 超大参数量
- 极长训练时间
- 巨量算力和显存

而且 Transformer 的训练并不是“读一遍数据就会了”，而是要在海量 token 上持续迭代更新。

所以大模型预训练往往是：

- 最贵的阶段
- 最难复现的阶段
- 最决定能力上限的阶段

# 三 预训练之后发生什么

## 3.1 微调（Fine-tuning）在这里扮演什么角色

预训练后的模型，本质上更像一个强大的“续写器”。

它可能有知识，但不一定擅长：

- 按角色回答
- 按格式输出
- 遵循任务约束
- 以助手方式互动

所以通常需要进入“微调”阶段。

但要注意，**微调不是单一方法名，而是一类适配过程。**

在大模型实践里，微调常见有两种理解：

- 狭义微调：主要指 SFT（监督微调）
- 广义微调：泛指在基础模型上继续做任务适配或领域适配

因此当大家说“给模型做微调”时，可能实际指的是：

- 用指令数据做 SFT
- 用领域数据做任务适配
- 用 LoRA / QLoRA 这类参数高效方法训练

## 3.2 指令微调（SFT）

SFT（Supervised Fine-Tuning）的核心是：用“指令-回答”样本进一步训练，让模型学会更像助手一样工作。

这一步主要解决：

- 更听指令
- 更符合问答格式
- 更贴近具体任务
- 更接近助手式交互

可以把它理解为：

- 预训练解决“模型会不会语言和知识”
- SFT 解决“模型会不会按要求回答”

## 3.3 微调方法和训练阶段不是一回事

还要再区分一层：

- `SFT / 领域微调`：说的是训练目标和训练阶段
- `Full Fine-tuning / LoRA / QLoRA / Adapter`：说的是参数更新方式

也就是说：

- 你可以做一个 SFT 任务
- 但这个 SFT 既可以用全参数微调完成，也可以用 LoRA 完成

这也是为什么“微调”这个词特别容易混淆。

更具体的微调方法，可继续看 [7 微调.md](./7%20%E5%BE%AE%E8%B0%83.md)。

## 3.4 偏好对齐

SFT 之后，模型已经能回答问题，但“能回答”和“回答得更好”不是一回事。

所以很多模型还会继续做偏好对齐，目标通常包括：

- 更安全
- 更有帮助
- 更符合人类偏好
- 在推理和工具使用中表现更稳定

常见方法包括：

- 奖励模型 + RLHF
- DPO 等偏好优化方法
- PPO、GRPO 等策略优化方法

这部分和 [5 强化学习.md](./5%20%E5%BC%BA%E5%8C%96%E5%AD%A6%E4%B9%A0.md) 是直接相关的。

# 四 继续预训练、领域微调、RAG 的区别

这几个概念经常被混在一起，但它们不属于同一层。

## 4.1 继续预训练

继续预训练（continued pretraining）指的是：

在已有基础模型上，再用特定领域的大规模语料继续做预训练。

它更适合解决：

- 模型缺少某个领域的语言分布
- 模型对某类文本风格不熟
- 希望增强领域知识基础

例如：

- 法律文本
- 医学文献
- 金融报告
- 企业内部专业文档

## 4.2 领域微调

领域微调更像是：在特定任务样本上，让模型学会该领域里的具体输出方式。

它更偏：

- 任务适配
- 指令适配
- 输出风格适配

所以可以粗略理解为：

- 继续预训练：补领域基础知识和语言分布
- 领域微调：补领域任务表现

## 4.3 RAG

RAG（检索增强生成）通常不属于训练阶段本身，而是应用阶段的能力增强方式。

它的核心思路是：

- 先从外部知识库检索相关内容
- 再把这些内容放进上下文
- 最后让模型基于这些内容回答

所以 RAG 解决的是：

**模型不需要把所有知识都重新训练进参数里，也能在回答时访问外部知识。**

## 4.4 工具调用

工具调用也通常更偏运行时能力，而不是训练流程本身。

它解决的是：

- 查询外部系统
- 执行代码
- 访问数据库
- 调用 API

因此不要把：

- 继续预训练
- 微调
- RAG
- 工具调用

看成同一类事情。它们分属训练层、适配层和应用层。

# 五 训练完成后为什么还要部署优化

模型训练完成，不代表系统就能直接上线。

推理部署还要解决很多工程问题，例如：

- 延迟
- 成本
- 吞吐
- 显存占用
- 并发
- 安全与权限控制

这也是为什么一个模型“效果很好”，不一定意味着它“系统可用”。

训练阶段决定能力形成，部署阶段决定能力能否稳定提供给真实用户。
