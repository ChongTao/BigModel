# 一 微调在大模型训练链路里的位置

微调（Fine-tuning）是指：在已经训练好的基础模型上，再用特定数据继续训练，让模型更适合目标任务或目标场景。

如果放到完整训练链路里看，可以先粗略理解为：

`预训练 -> 微调 -> 偏好对齐 -> 部署`

其中：

- 预训练决定模型有没有基础语言能力和知识
- 微调决定模型会不会按任务要求输出
- 偏好对齐决定模型更偏向怎样的回答方式

所以，微调更像是“能力适配”阶段，而不是从零训练模型。

## 1.1 为什么需要微调

预训练后的模型通常更像一个通用续写器，它可能已经具备：

- 语言理解能力
- 基础知识
- 基础推理能力

但它未必天然擅长：

- 按指令回答
- 输出固定格式
- 适应某个具体业务任务
- 使用某个领域的话语风格

这就是微调的意义：**让通用模型更贴近实际任务。**

## 1.2 微调不等于所有“后续训练”

“微调”这个词经常被说得很宽，所以要先把几个概念拆开：

- `continued pretraining`：继续预训练，偏向补领域知识和语言分布
- `SFT`：监督微调，偏向让模型学会按指令完成任务
- `DPO / RLHF`：偏好对齐，偏向让模型更符合人类偏好

因此更准确地说：

- 微调通常主要指 `SFT` 和任务适配
- 继续预训练不完全等于微调
- 偏好对齐也不应直接混进“微调方法”里

# 二 微调主要在解决什么问题

## 2.1 任务适配

让模型在具体任务上表现更好，例如：

- 问答
- 摘要
- 分类
- 信息抽取
- 代码补全

## 2.2 指令适配

让模型更像一个“助手”，而不是只会机械续写文本。

例如：

- 更理解用户指令
- 更遵循输出格式
- 更稳定执行任务约束

## 2.3 领域适配

让模型更适应某个专业领域的表达方式、知识结构和输出风格，例如：

- 医疗
- 法律
- 金融
- 企业内部知识

## 2.4 风格适配

让模型输出更符合目标风格，例如：

- 更正式
- 更简洁
- 更像客服
- 更像内部助手

# 三 微调的两种主线

## 3.1 按训练目标分

如果按“想让模型学会什么”来分，常见有两类：

### 3.1.1 监督微调（SFT）

用“输入-输出”配对数据训练模型。

例如：

- 指令 -> 回答
- 问题 -> 答案
- 文本 -> 分类标签

这是当前最常见、最核心的微调方式。

### 3.1.2 领域任务微调

和 SFT 类似，但数据和目标更偏具体业务场景。

例如：

- 法律问答
- 医疗报告生成
- 企业知识助手

## 3.2 按参数更新方式分

如果按“到底更新多少参数”来分，常见有两类：

- 全参数微调（Full Fine-tuning）
- 参数高效微调（PEFT，Parameter-Efficient Fine-Tuning）

这里要特别区分：

- `SFT` 是训练目标
- `LoRA / QLoRA / Adapter / Prefix-Tuning` 是训练手段

所以：

- 可以做一个 SFT 任务
- 但这个任务既可以用全参数微调完成，也可以用 LoRA 完成

# 四 常见微调方法

## 4.1 全参数微调（Full Fine-tuning）

全参数微调的意思是：训练时更新模型的大部分或全部参数。

优点：

- 适配能力最强
- 对复杂任务通常上限更高

缺点：

- 计算成本高
- 显存占用大
- 训练和存储成本都更重

它更适合：

- 资源充足
- 模型规模相对可控
- 对任务效果要求很高

## 4.2 LoRA

LoRA（Low-Rank Adaptation）是最常见的参数高效微调方法之一。

它的核心思想不是直接改完整权重，而是：

- 冻结原模型大部分参数
- 只训练额外的低秩增量矩阵
- 用这些小矩阵去近似表示参数更新

所以 LoRA 的重点是：**用更少的可训练参数，达到接近全参数微调的适配效果。**

优点：

- 训练参数少
- 显存占用低
- 存储和分发更方便

缺点：

- 某些复杂任务上可能不如全参数微调
- 依赖超参设置和目标模块选择

## 4.3 QLoRA

QLoRA 可以看作 LoRA 的进一步工程化版本。

它的核心思路是：

- 把基础模型参数以更低精度方式加载
- 在低显存条件下做 LoRA 训练

因此它特别适合：

- 单卡或少卡场景
- 资源有限的实验环境
- 中小团队做任务适配

可以把它理解成：

- LoRA：减少可训练参数
- QLoRA：在 LoRA 基础上进一步减少显存压力

## 4.4 Prefix-Tuning / P-Tuning

这类方法的思路是：不直接大改模型内部权重，而是在输入侧加入可学习的前缀表示，让模型在这些前缀的引导下完成任务。

优点：

- 训练参数少
- 适合轻量实验

缺点：

- 对复杂任务的适配能力可能有限
- 实际工程采用率通常不如 LoRA 高

## 4.5 Adapter

Adapter 的思路是：在模型层之间插入小型适配模块，只训练这些新增模块。

优点：

- 参数量小
- 不同任务可以挂不同 Adapter
- 多任务共存更方便

缺点：

- 结构会比 LoRA 稍重
- 工程复杂度可能更高

# 五 微调的一般流程

## 5.1 数据准备

先准备高质量任务数据，例如：

- 指令-回答对
- 分类标注数据
- 问答对
- 结构化输出样本

这一阶段通常要做：

- 清洗
- 去重
- 格式统一
- 划分训练集和验证集

## 5.2 选择训练目标

先明确你到底要让模型学会什么：

- 更好执行指令
- 更好适应领域任务
- 更稳定输出某种格式

目标不同，数据形式和评估方式也会不同。

## 5.3 选择参数更新方式

常见选择包括：

- 全参数微调
- LoRA
- QLoRA
- Adapter
- Prefix-Tuning

这一选择本质上是在权衡：

- 效果上限
- 显存成本
- 训练速度
- 部署便利性

## 5.4 训练与评估

训练时通常会关注：

- 训练 loss
- 验证集效果
- 是否过拟合
- 输出风格是否偏移

除了自动指标，也常需要做人工抽样评估，因为很多生成任务很难只靠单个分数说明问题。

## 5.5 部署与使用

训练完成后，还要考虑：

- 模型如何部署
- LoRA 权重如何加载
- 是否需要和基座模型合并
- 是否支持多任务切换

所以微调不是“训完就结束”，后面还有工程落地问题。

# 六 什么时候该微调，什么时候不该微调

适合微调的场景：

- 任务模式比较稳定
- 输出格式要求明确
- 有足够高质量样本
- 希望长期复用某种能力

不一定要先微调的场景：

- 问题主要是知识缺失，但外部知识可检索
- 需求变化很快，样本不稳定
- 只需要通过 Prompt、RAG、工具调用就能解决

所以很多时候，问题不一定是“模型需要微调”，而可能是：

- 上下文没给够
- RAG 没做好
- 工具调用没接好

# 七 一句话总结

如果说：

- 预训练决定模型有没有基础能力
- 偏好对齐决定模型更偏向怎样行为

那么：

- 微调决定模型能不能把这些能力稳定地用在具体任务上

其中，SFT 解决“学什么输出”，LoRA/QLoRA/Adapter 解决“怎么以更低成本把它训出来”。
