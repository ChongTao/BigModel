# 一 Prompt Engineering 是什么

Prompt Engineering（提示词工程）指的是：**通过更合理地组织提示词，让大语言模型更稳定、更准确地完成任务。**

它关注的重点不是训练模型参数，而是优化**输入给模型的指令形式**。

可以先把它理解为：

- 模型能力决定“它大致能做什么”
- Prompt Engineering 决定“你怎么把这份能力更稳定地调出来”

所以 Prompt Engineering 的核心目标通常包括：

- 提高回答准确性
- 提高输出稳定性
- 增强可控性
- 降低歧义和幻觉

## 1.1 Prompt Engineering 解决什么问题

同一个模型，在不同提示词下，输出效果可能差别很大。

常见问题包括：

- 指令不清，模型理解偏了
- 输出格式不稳定
- 约束条件被忽略
- 任务太复杂，模型一步做不好

Prompt Engineering 要解决的，本质上是：**如何把任务说清楚、说完整、说得足够可执行。**

## 1.2 Prompt Engineering 不是什么

也要明确边界：

- 它不是模型训练本身
- 它不是 RAG
- 它不是工具调用协议
- 它也不等于完整 Agent 工作流

有些方法虽然和 Prompt 强相关，但已经不只是 Prompt 层的问题，例如：

- `RAG`：属于检索增强架构
- `Tool Use / ReAct`：属于推理与行动范式
- `Reflection / Reflexion`：属于反思与自检范式

这些内容可以通过提示词触发，但本质上需要额外的系统能力或执行机制支持。

# 二 核心原则

一个高质量 Prompt，通常至少要把下面几件事交代清楚：

## 2.1 角色

告诉模型“你现在扮演什么角色”。

例如：

- 你是一名技术文档编辑
- 你是一名代码审查助手
- 你是一名法律文本摘要助手

角色的作用，不是让模型“演戏”，而是帮助它快速进入合适的输出风格和任务视角。

## 2.2 指令

告诉模型“你到底要它做什么”。

指令应该尽量具体，避免模糊词。

例如：

- 不要只写“分析一下”
- 更好写法是“请总结 3 个主要风险，并按严重程度排序”

## 2.3 上下文

告诉模型“它完成任务需要知道哪些背景信息”。

上下文可能包括：

- 用户背景
- 业务目标
- 限制条件
- 外部资料
- 上一轮对话内容

很多时候，模型答不好，不是因为不会，而是因为上下文没给够。

## 2.4 输出要求

告诉模型“结果应该长什么样”。

例如：

- 使用 Markdown 输出
- 输出 JSON
- 不超过 3 点
- 每点不超过 100 字

输出要求越清楚，结果越稳定。

## 2.5 示例

当任务格式复杂时，直接给示例往往比抽象解释更有效。

示例可以帮助模型学习：

- 格式
- 风格
- 粒度
- 任务边界

这也是 few-shot prompting 的基础。

# 三 常见 Prompt 方法

## 3.1 Zero-Shot Prompting

零样本提示，是指不提供示例，只给任务说明，让模型直接完成任务。

适合：

- 模型已经很熟悉的常见任务
- 任务格式简单明确

优点：

- 成本低
- 写法简单

局限：

- 对格式复杂或约束较多的任务不够稳

## 3.2 Few-Shot Prompting

Few-shot prompting 是指在提示词中加入少量示例，让模型先“看懂格式和风格”，再处理新输入。

适合：

- 任务格式固定
- 输出风格要求明确
- 需要模型模仿某种模式

优点：

- 对格式控制更强
- 对风格一致性更好

局限：

- 会占用更多上下文
- 示例选不好，可能把模型带偏

## 3.3 Chain-of-Thought（CoT）

CoT 的核心是：让模型显式展示推理过程，而不是只给最终答案。

适合：

- 数学题
- 逻辑推理
- 多步分析任务

优点：

- 常能提升复杂推理正确率

局限：

- 会增加输出长度
- 不一定适合所有任务

## 3.4 Self-Consistency

Self-Consistency 是 CoT 的增强思路：

- 先让模型生成多个不同推理路径
- 再选择最一致的答案

它更像是在用“多次采样 + 多条思路投票”提升正确率。

适合：

- 单一正确答案问题
- 对推理正确率要求较高的任务

## 3.5 Generate Knowledge

Generate Knowledge 的思路是：先让模型生成相关知识，再基于这些知识回答问题。

它的目标是：

- 先把可用知识激活出来
- 再做最终回答

这类方法适合知识依赖较强、但又不一定需要外部检索的任务。

## 3.6 Prompt Chaining

Prompt Chaining 是把一个复杂任务拆成多个连续子任务，再让模型逐步完成。

例如：

1. 先抽取信息
2. 再归类
3. 最后生成总结

它本质上仍然属于 Prompt 设计思路，但已经开始接近简单工作流。

## 3.7 Active-Prompt

Active-Prompt 更强调：不是人工手写唯一提示词，而是通过自动探索、筛选和迭代，寻找更优 Prompt。

它偏研究和优化视角，不是日常最常用的入门方法，但有助于理解：

**Prompt 本身也可以成为被优化的对象。**

# 四 与其他能力的边界

下面这些内容和 Prompt Engineering 强相关，但不应直接等同于 Prompt 技术：

## 4.1 RAG

RAG 的重点是“引入外部知识”，不是单纯重写提示词。

即使最终会把检索结果拼进 Prompt，它本质上仍然是检索增强架构。

详见 [../../2 RAG/1 RAG概述.md](../../2%20RAG/1%20RAG%E6%A6%82%E8%BF%B0.md)。

## 4.2 Tool Use / ReAct

Tool Use、ReAct 的重点是“推理 + 行动 + 观察”的执行闭环，不只是提示词写法。

Prompt 可以描述这种范式，但真正落地通常还需要工具调用机制。

详见 [../3 推理模型/1 ReAct.md](../3%20%E6%8E%A8%E7%90%86%E6%A8%A1%E5%9E%8B/1%20ReAct.md)。

## 4.3 Reflection / Reflexion

反思模式强调“生成后评估并修正”，本质上是推理/工作流范式，而不只是 Prompt 文本技巧。

详见 [../3 推理模型/2 Reflexion.md](../3%20%E6%8E%A8%E7%90%86%E6%A8%A1%E5%9E%8B/2%20Reflexion.md)。

# 五 常见 Prompt 结构模板

这些模板的价值，不在于必须死记，而在于帮助你系统化组织提示词。

## 5.1 ICIO

- Instruction：任务指令
- Context：背景上下文
- Input Data：输入数据
- Output Indicator：输出要求

它适合绝大多数“任务明确、输入明确、输出格式明确”的场景。

## 5.2 APE

- Action：行动
- Purpose：目的
- Expectation：期望

它适合“需要同时说清楚动作、动机和结果”的沟通型任务。

## 5.3 CARE

- Context：语境
- Action：行动
- Result：结果
- Example：示例

它适合需要补充场景感和例子的说明型任务。

## 5.4 TRACE

- Task：任务
- Request：请求
- Action：操作
- Context：语境
- Example：示例

它适合任务拆得更细、执行要求更明确的场景。

## 5.5 一个实用的四段式写法

如果不想记太多框架，也可以先用一个更朴素的四段式结构来写 Prompt：

1. 任务说明
2. 上下文
3. 问题
4. 输出格式

可以这样理解：

- **任务说明**：告诉模型你要它做什么
- **上下文**：补足背景、限制条件和已有信息
- **问题**：明确这次真正要回答的核心点
- **输出格式**：约束结果应该长什么样

例如：

```text
任务说明：
你是一名资深产品经理，请从专业角度进行分析。

上下文：
该产品是一款面向大学生的时间管理 App，目前用户留存率较低。

问题：
请分析用户留存率低的 3 个主要原因。

输出格式：
请使用 Markdown 编号列表，每点不超过 100 字。
```

这个四段式并不是新的理论框架，而是把前面讲过的“指令、上下文、输出要求”用更直接的方式组织出来，特别适合日常工作场景。

# 六 实际使用建议

如果只是日常使用 Prompt，通常不需要一上来就套复杂框架。

更实用的做法是优先保证这四点：

1. 任务说清楚
2. 上下文给够
3. 输出格式明确
4. 必要时给示例

很多 Prompt 问题，并不是“缺高级技巧”，而是基础信息没说完整。

# 七 一句话总结

如果说：

- 模型能力决定“能不能做”
- 上下文决定“知道多少”

那么：

- Prompt Engineering 决定“你怎么把任务说清楚，让模型更稳定地做出来”

它优化的是**输入组织方式**，不是替代训练、RAG 或完整工作流。
