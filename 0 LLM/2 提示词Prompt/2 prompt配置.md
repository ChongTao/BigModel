# 一 常见生成参数

这一页讲的不是 Prompt 文本本身，而是**调用模型时常见的生成参数**。

它们通常用于控制：

- 输出长度
- 随机性
- 重复倾向
- 停止条件

如果你想看“这些参数背后的生成原理”，可继续阅读 [../1 大模型原理/4 采样与解码.md](../1%20%E5%A4%A7%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%90%86/4%20%E9%87%87%E6%A0%B7%E4%B8%8E%E8%A7%A3%E7%A0%81.md)。

## 1.1 先看一个边界

不同模型、不同 API 平台支持的参数并不完全一样。

例如有些平台支持：

- `temperature`
- `top_p`
- `max_tokens`
- `presence_penalty`
- `frequency_penalty`
- `stop`

但也有些平台会：

- 不支持其中一部分参数
- 改用别的参数名，例如 `max_completion_tokens`
- 对参数取值范围做不同限制

所以这篇适合当作**通用理解**，不是某一家 API 的绝对规范。

# 二 控制随机性的参数

## 2.1 temperature

`temperature` 用来调节输出分布的“尖锐程度”。

可以粗略理解为：

- 值更低：分布更尖，模型更保守、更稳定
- 值更高：分布更平，模型更发散、更多样

常见使用倾向：

- 代码生成、信息抽取、工具调用：通常偏低
- 日常问答、总结：通常中低
- 创意写作、文案生成：通常偏高

要注意：

- `temperature` 不是“知识正确率开关”
- 温度调低，不会凭空让模型知道它原本不知道的事实

## 2.2 top_p

`top_p` 也叫 nucleus sampling。

它的作用不是像 `temperature` 那样重塑整个分布，而是：

- 先按概率从高到低排序候选 token
- 再只保留累计概率达到 `p` 的最小候选集合
- 最后只在这个集合里采样

可以粗略理解为：

- 值更低：候选集合更小，输出更保守
- 值更高：候选集合更大，输出更多样

## 2.3 temperature 和 top_p 的区别

这两个参数都影响随机性，但控制方式不同：

- `temperature`：调节分布形状
- `top_p`：裁剪候选集合

所以它们不是同一个旋钮的两个名字。

实际使用时：

- 简单场景通常调一个就够
- 如果两个一起调，容易把行为搞得不直观

# 三 控制长度的参数

## 3.1 max_tokens

`max_tokens` 通常表示：**本次输出允许生成的最大 token 数。**

它控制的是“模型最多还能输出多长”，而不是整个上下文窗口本身。

要特别区分：

- `上下文窗口`：输入和输出加起来一共能处理多少 token
- `max_tokens`：这一次最多允许生成多少输出 token

不同平台里，这个参数也可能叫：

- `max_tokens`
- `max_completion_tokens`

如果设得太小，常见结果是：

- 输出被截断
- 回答没说完
- JSON 或代码结构不完整

# 四 控制重复和话题扩散的参数

## 4.1 presence_penalty

`presence_penalty` 的直觉作用是：

**如果某个 token 或话题已经出现过，就降低模型继续围绕它展开的倾向。**

它更偏向鼓励“去探索新内容”。

适合：

- 希望减少话题反复打转
- 希望内容更有展开性

## 4.2 frequency_penalty

`frequency_penalty` 的直觉作用是：

**如果某个 token 重复出现得太频繁，就提高继续重复它的代价。**

它更偏向抑制“字面重复”。

适合：

- 避免重复句
- 避免某些词反复出现

## 4.3 两者的区别

可以简单记：

- `presence_penalty`：更偏“不要总围着同一个点说”
- `frequency_penalty`：更偏“不要把同样的话反复说”

还要注意：并不是所有模型都支持这两个参数。

# 五 停止控制

## 5.1 stop

`stop` 用于指定：当模型生成到某些字符串或标记时，就停止输出。

它常用于：

- 控制固定格式输出
- 截断多余内容
- 多轮模板中的边界控制

例如你希望模型在生成到某个分隔符时停止，就可以配置对应 `stop` 字符串。

但要注意：

- `stop` 更像运行时截断条件
- 它不保证模型天然理解你的格式语义

# 六 实际使用建议

如果没有特别强的需求，可以先用下面这套朴素原则：

1. 先把 Prompt 写清楚，再调参数
2. 优先少调参数，不要一开始全开
3. 稳定任务先调低 `temperature`
4. 输出太短或经常截断时，再检查 `max_tokens`
5. 出现明显重复时，再考虑 `frequency_penalty` 或 `presence_penalty`

很多问题并不是参数没调对，而是：

- Prompt 不清楚
- 上下文不完整
- 任务本身不适合靠纯生成解决

# 七 一句话总结

如果说 Prompt 解决的是“任务怎么说清楚”，那么生成参数解决的是：

- 回答多长
- 回答多稳
- 回答会不会重复
- 什么时候停

它们能优化输出行为，但不能替代模型能力本身。
