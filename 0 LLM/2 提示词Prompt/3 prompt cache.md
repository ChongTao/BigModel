# 一 Prompt Cache 介绍

Prompt Cache（提示缓存）指的是：**当同一类请求在前缀部分重复出现时，模型服务端不必每次都重新处理整段输入，而是复用已经计算过的前缀结果，从而降低延迟和成本。**

它本质上不是“把最终答案缓存下来”，而是**把提示词中稳定、不变、可重复利用的部分缓存下来**。因此，Prompt Cache 更适合处理“前缀高度重复、只有后半段输入在变化”的场景。

从目录归类看，Prompt Cache 更适合放在“提示词 Prompt”里，因为开发者真正能直接优化的，通常不是底层缓存实现本身，而是：

- 提示前缀怎么组织
- system prompt 怎么保持稳定
- 工具定义和上下文模板怎么排列
- 哪些动态变量不应该破坏前缀

例如：

- 系统提示长期固定
- 大段工具定义长期固定
- RAG 中固定的背景说明长期固定
- 多轮任务中规则、角色、格式要求长期固定

这时，如果每次请求都从头重新计算，会浪费大量 token 和时间；而 Prompt Cache 的作用，就是把这些重复部分复用起来。

## 1.1 Prompt Cache 解决什么问题

在大模型应用中，很多请求并不是完全不同的，而是：

- 前面一大段相同
- 最后用户问题或局部上下文不同

常见例子：

- 同一个 Agent 每次都要带上固定 system prompt
- 同一个 Copilot 每次都要附带一套工具定义
- 同一个企业应用每次都要附带权限规则、风格规范、输出模板
- 同一个文档分析任务中，文档正文不变，问题不断变化

如果这些内容每次都重新送进模型并重新编码，成本很高，响应也会变慢。

Prompt Cache 主要优化三件事：

- **降低首 token 延迟**
- **降低重复输入成本**
- **提升高频、重复型工作流的吞吐**

## 1.2 Prompt Cache 的核心思想

可以把一次模型调用粗略拆成两部分：

- **稳定前缀**
- **动态后缀**

其中：

- 稳定前缀通常是 system prompt、工具定义、固定背景、长文档正文
- 动态后缀通常是当前用户问题、本轮新增状态、最新检索结果、当前任务变量

Prompt Cache 的基本原则就是：

**尽量让“稳定前缀”保持稳定，并尽量把变化内容放到后面。**

这样，当新请求到来时，模型服务端只需要处理变化部分，而不用重新处理整个上下文。

# 二 Prompt Cache 和普通缓存的区别

Prompt Cache 容易和传统缓存混淆，但它们并不是一回事。

## 2.1 不是结果缓存

传统缓存更像：

- 输入一样，直接返回历史结果

而 Prompt Cache 更像：

- 输入前半段一样，复用前半段的计算过程
- 后半段不同，仍然要继续推理并生成新答案

所以它缓存的不是最终输出，而是**提示前缀的计算状态**。

## 2.2 不是前端缓存

Prompt Cache 也不是浏览器或应用层的本地缓存，它通常发生在：

- 模型推理服务端
- 模型网关层
- Agent Runtime / 推理基础设施层

因此，它更偏提示前缀优化和推理基础设施优化，而不是页面优化。

## 2.3 通常是谁提供这项能力

对于大多数开发者来说，Prompt Cache 通常可以理解为：**模型或第三方 API 提供方在服务端实现的前缀缓存能力。**

也就是说，平时在业务代码里直接调用 OpenAI、Anthropic 或其他模型 API 时，Prompt Cache 往往不是你自己在本地实现的，而是：

- 由模型服务商在推理侧支持
- 由模型网关或推理基础设施层负责实现
- 由调用方通过固定前缀、稳定提示结构或平台提供的缓存控制方式来触发

所以从使用者视角看，它更像“API 平台提供的能力”；从平台实现视角看，它属于“模型推理基础设施的一部分”。

当然，这并不意味着只有第三方 API 才能做 Prompt Cache。如果团队自己部署模型，也可以在自建推理层实现类似的前缀缓存机制。

# 三 Prompt Cache 的典型适用场景

## 3.1 长 System Prompt

如果一个应用每次都要附带几百到几千 token 的系统提示，例如：

- 角色定义
- 安全规则
- 输出格式要求
- 企业知识边界

那么这些内容非常适合进入缓存前缀。

## 3.2 工具定义很多的 Agent

在 Agent 场景下，常见问题是工具 schema 很长，例如：

- 多个 MCP Tool
- 多个 Function Calling 定义
- 参数 schema 很复杂

这些内容如果每轮都完整发送，会占据大量 token。Prompt Cache 可以显著降低这类固定工具描述的重复开销。

## 3.3 长文档问答

例如一篇长论文、一份合同或一份 PRD：

- 文档正文不变
- 用户问题持续变化

这类场景非常适合做 Prompt Cache，因为大部分 token 都来自同一份文档本身。

## 3.4 多轮复杂任务

在长任务里，往往有很多长期不变的信息：

- 项目规则
- 环境说明
- 代码规范
- 执行约束

这些都可以成为可缓存前缀，而当前步骤、当前错误日志、最新输出则作为动态部分追加在后面。

# 四 Prompt Cache 的工作方式

虽然不同平台实现不同，但整体逻辑一般类似：

1. 服务端识别请求中的可复用前缀
2. 如果此前已经处理过同样的前缀，则命中缓存
3. 复用前缀对应的计算状态
4. 只对新增或变化部分继续计算
5. 返回新的完整响应

从使用者视角，可以把它理解成：

**第一次请求像“预热”，后续相同前缀的请求则进入“复用模式”。**

## 4.1 命中条件的核心

Prompt Cache 是否生效，最关键的不是“语义差不多”，而通常是：

- 前缀是否真正一致
- 顺序是否一致
- token 化后的结果是否一致
- 缓存是否还在有效期内

也就是说，哪怕你觉得两段提示“看起来差不多”，只要前缀文本、顺序或结构发生变化，就可能无法命中。

## 4.2 为什么放在前面的内容更重要

大多数 Prompt Cache 机制都优先复用**前缀**，而不是任意位置的重复内容。

所以在设计提示时，一个很重要的工程原则是：

- 把长期稳定内容放前面
- 把经常变化内容放后面

这也是 Prompt Cache 和 Prompt 结构设计直接相关的地方。

# 五 如何设计更容易命中的 Prompt Cache

## 5.1 固定 System Prompt

不要在 system prompt 中频繁插入动态变量，例如：

- 当前时间
- 随机 ID
- 每轮变化的状态字段

这些动态信息会破坏前缀稳定性。

更好的做法是：

- system prompt 保持固定
- 动态变量放到用户输入或后置上下文中

## 5.2 固定工具顺序和定义

如果工具定义顺序频繁变化，即使工具集合相同，也可能影响缓存命中。

建议：

- 保持工具列表顺序固定
- 保持 schema 字段顺序稳定
- 不要每轮无意义地重写描述文案

## 5.3 固定上下文包装模板

例如统一使用：

```text
<system_rules>
...
</system_rules>

<tool_definitions>
...
</tool_definitions>

<task_context>
...
</task_context>

<user_request>
...
</user_request>
```

这样做的好处是：

- 结构稳定
- 更利于缓存命中
- 更利于减少上下文混淆

## 5.4 把变化部分后移

例如不要这样设计：

```text
当前时间：2026-06-05。请遵循以下安全规则...
```

更适合改成：

```text
请遵循以下安全规则...

当前时间：2026-06-05
```

也就是把稳定规则放前面，把动态变量放后面。

## 5.5 避免无意义改写

下面这些操作都可能影响缓存命中：

- 频繁改标点
- 频繁改空行
- 同义改写 system prompt
- 每轮重排工具定义

从工程角度看，想要更高命中率，就要把“固定前缀”真正当成稳定资产来管理。

# 六 Prompt Cache 的价值与限制

## 6.1 价值

Prompt Cache 的价值主要体现在：

- **成本优化**：减少重复前缀的输入成本
- **性能优化**：降低重复上下文导致的首 token 延迟
- **系统优化**：提高高频任务和批量任务的吞吐

对于长上下文、长工具定义、长文档分析类应用，这种优化尤其明显。

## 6.2 限制

Prompt Cache 不是万能的，它也有明显边界：

- 前缀变化太多时，命中率会很低
- 缓存通常有生命周期，不会永久保留
- 不同模型和平台实现差异很大
- 缓存优化的是“重复前缀计算”，不是最终答案质量
- 如果提示设计本身混乱，缓存也救不了上下文质量问题

所以，Prompt Cache 是**性能优化手段**，不是提示设计的替代品。

# 七 Prompt Cache 与 Prompt Engineering、Context Engineering 的关系

Prompt Cache 虽然名字里有 Prompt，但它并不只是“怎么写一句提示词”的问题。更准确地说，它是**以 Prompt 结构设计为入口，以推理侧缓存命中为结果**的一类优化能力。

## 7.1 与 Prompt Engineering 的关系

Prompt Engineering 关注：

- 指令怎么写
- 角色怎么设定
- 输出格式怎么约束

在这个基础上，Prompt Cache 进一步要求你关注：

- 前缀是否稳定
- 模板是否模块化
- 动态变量是否后置
- 工具定义是否固定顺序

Prompt Cache 会反过来影响这些内容的写法，因为提示结构越稳定、越模块化，就越容易复用。

## 7.2 与 Context Engineering 的关系

Context Engineering 关注：

- 每轮给模型什么上下文
- 哪些上下文该保留
- 哪些信息该裁剪、压缩、重排

Prompt Cache 正是其中的重要优化手段之一。它要求你把上下文拆成：

- 长期稳定部分
- 短期变化部分

并据此设计上下文结构。

因此可以说：

**Prompt Cache 是“上下文复用”的工程化能力。**

# 八 不同平台中的 Prompt Cache

不同模型平台对 Prompt Cache 的支持方式并不相同，但大致可以分为两类：

## 8.1 自动缓存

有的平台会自动识别长前缀并进行缓存，开发者不需要显式声明，系统会在满足条件时自动命中。

这种方式的优点是：

- 使用简单
- 对业务改造小

缺点是：

- 可控性较弱
- 命中细节依赖平台实现

## 8.2 显式缓存控制

有的平台允许开发者显式标记哪些内容应该作为缓存边界或缓存块。

这种方式的优点是：

- 更可控
- 更适合复杂 Agent 或长文档场景

缺点是：

- 需要更理解上下文结构
- 对提示编排要求更高

从工程实践角度，核心并不是记住哪家平台怎么做，而是理解一个统一原则：

**稳定前缀越清晰，Prompt Cache 越容易发挥价值。**

# 九 使用 Prompt Cache 的最佳实践

- 让 system prompt 尽量长期稳定
- 把工具定义做成稳定模板
- 把动态变量放到后面
- 保持上下文结构统一，不要频繁改版
- 把长文档和长规则作为可复用前缀
- 不要把 Prompt Cache 当作提升回答质量的手段
- 关注命中率、延迟、输入 token 成本这三个指标

# 十 总结

Prompt Cache 本质上是在大模型调用过程中，**复用稳定提示前缀的计算结果**，从而降低成本和延迟。

它最适合的不是“一次性问答”，而是：

- 有长期固定规则的应用
- 有大量工具定义的 Agent
- 有长文档、长上下文反复提问的系统
- 有高频重复调用需求的业务

如果从工程角度理解，Prompt Cache 不是一个孤立技巧，而是大模型应用从“会用 Prompt”走向“会设计上下文结构和运行成本”的重要一步。
