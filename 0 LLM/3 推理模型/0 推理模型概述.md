# 一 推理模型概述

这一层关注的是：**当任务不是一步直答时，模型如何通过分步推理、行动、反思和修正来完成更复杂的问题。**

这里的“推理模型”不是指新的基座模型类别，而是指建立在 LLM 之上的推理模式和执行范式。

## 1.1 这一层解决什么问题

- 复杂问题如何拆成多个可执行步骤。
- 推理过程中如何引入外部行动和反馈。
- 出错后如何评估、反思和修正。
- 如何让模型从“直接回答”升级为“逐步解决问题”。

## 1.2 Reasoning 和 Acting 的区别

在这类任务里，通常可以把模型能力粗略分成两部分：

- **Reasoning**：模型在内部生成推理步骤，例如数学、逻辑、多步规划。
- **Acting**：模型与外部环境交互，例如检索、调用 API、运行工具。

很多现代推理范式的核心，就是如何把这两者组合起来。

## 1.3 常见推理模式

### 1.3.1 Chain-of-Thought（CoT）

CoT 的核心思想是：让模型**显式写出推理过程**，而不只是直接给最终答案。

它适合：

- 数学题
- 逻辑推理
- 需要分步展开的问题

它的优势是简单直接，但局限也很明显：只有“思考”，没有“行动”，因此在需要外部信息时容易产生幻觉。

### 1.3.2 Tree-of-Thoughts（ToT）

ToT 可以看作 CoT 的扩展版：不是只走一条思路，而是同时探索多个分支，再从中挑选更优路径。

它适合：

- 高复杂度推理
- 多路径规划
- 需要搜索最优解的问题

代价是计算成本明显更高。

### 1.3.3 Self-Ask

Self-Ask 的思路是：模型先把原问题拆成若干子问题，再逐步回答。

它适合：

- 多跳问答
- 信息整合
- 结构化拆解任务

相比普通 CoT，它更强调“先拆问题，再逐步求解”。

### 1.3.4 Tool-Use / MRKL

这类模式强调：模型不仅靠内部参数回答，还能通过工具获得外部信息。

典型特点是：

- 有明确的 `Action -> Observation`
- 工具返回真实数据
- 可以执行检索、计算、查询等操作

它解决的是“模型怎么接入外部能力”，但不一定天然提供完整推理闭环。

### 1.3.5 ReAct

ReAct（Reason + Act）把推理和行动交错组织起来：

- 先思考
- 再行动
- 再根据观察结果继续思考

它适合：

- 检索增强问答
- 工具调用任务
- 多步骤决策和 Agent 工作流

更具体内容见 [1 ReAct.md](./1%20ReAct.md)。

### 1.3.6 Reflection / Reflexion

这类模式强调：模型先给出答案，再对自己的答案进行评估、修正和迭代。

它适合：

- 代码生成
- 长文写作
- 需要自检正确性的复杂任务

更具体内容见 [2 Reflexion.md](./2%20Reflexion.md)。

### 1.3.7 Plan-and-Execute

Plan-and-Execute 会把任务拆成两个角色：

- Planner：先制定步骤
- Executor：再逐步执行

它更强调把“规划”和“执行”解耦，适合复杂工作流和多阶段任务。

## 1.4 这些模式之间的关系

可以把它们粗略理解成三类：

- **纯推理型**：CoT、ToT、Self-Ask
- **推理 + 行动型**：Tool-Use、ReAct、Plan-and-Execute
- **推理 + 反思型**：Reflection、Reflexion

这三类并不是彼此排斥的，很多系统会把它们组合使用。

例如：

- ReAct + Reflection
- Planner + Tool-Use
- CoT + Self-Consistency

## 1.5 与其他目录的关系

- 和 `提示词Prompt` 的区别：
  Prompt 更关注输入设计，推理模型更关注任务执行过程。
- 和 `工具与协议` 的关系：
  推理模型常常会调用工具，但工具层解决的是“怎么接入工具”，不是“怎么推理”。
- 和 `上下文工程` 的关系：
  推理模型依赖高质量上下文，但上下文工程本身不决定推理策略。
- 和 `RAG` 的关系：
  RAG 提供外部知识来源，推理模型决定如何把这些知识组织进任务求解过程。
