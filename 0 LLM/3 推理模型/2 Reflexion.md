# 一 Reflexion

Reflexion 可以理解为一种“带反思环路”的推理范式。它的核心思想不是让模型只生成一次答案就结束，而是让模型在尝试后对自己的结果进行评估、总结和修正。

相比传统强化学习依赖标量奖励，Reflexion 更强调：**用语言形式的反馈帮助模型从错误中学习。**

它主要解决的问题是：

- CoT 只负责展开推理，但不会系统地回头检查自己
- ReAct 能行动，但失败后不一定能沉淀经验
- 普通一次性生成很难在同一任务中持续改进

## 1.1 什么是“反思模式”

反思模式的核心不是简单把输出交给下一步，而是引入**反馈循环**：

1. 先执行任务或生成初始答案
2. 再对这个结果进行评估
3. 根据评估结果给出改进方向
4. 再生成更优版本或调整后续行动

因此，反思模式本质上是一种自我纠错机制。它的重点不在于“第一次就答对”，而在于“出错后能不能系统地变好”。

## 1.2 Reflexion 的三大组件

Reflexion 通常构建在 ReAct 或 CoT 之上，并加入评估和反思机制，形成闭环：

- **Actor：**负责生成行动轨迹或答案。
- **Evaluator：**负责判断当前结果是否成功、是否存在问题。
- **Self-Reflection：**根据评估结果生成语言反馈，指导下一轮尝试。

工作流程可以概括为：

`行动 -> 评估 -> 反思 -> 迭代`

## 1.3 Producer-Critic 结构

反思模式一个很高效的实现方式，是把流程拆成两个逻辑角色：

- **Producer（生产者）**：先完成初始生成或执行
- **Critic（评论者）**：专门分析结果并提出问题与建议

虽然单个模型也能“自我反思”，但把“生成”和“审查”拆成两个角色，通常能让反馈更稳定、更客观。

例如：

- 生产者负责写代码
- 评论者负责检查代码正确性、风格和完整性

这种结构在 Agent 系统、代码生成和复杂写作场景中都很常见。

## 1.4 记忆为什么重要

Reflexion 的效果通常和记忆能力强相关。

如果模型每轮都忘掉上一次的失败原因，那么“反思”就会退化成一次性的局部修补。

而当系统能够保留：

- 上一轮失败点
- 上一轮评估结论
- 用户的持续反馈

反思就会变成累积性的优化过程，而不是孤立事件。

这也是为什么 Reflexion 很适合和短期记忆、长期记忆、任务状态一起使用。

## 1.5 适用场景

Reflexion 特别适合需要试错和自检的任务，例如：

- 代码生成与修复
- 决策与规划
- 长文写作
- 解题正确性检查

它的核心价值在于：当第一次结果不够好时，系统仍有机会通过评估和语言反馈逐步收敛。

## 1.6 优点与局限

优点：

- 可解释性强
- 不一定需要重新微调整个模型
- 很适合和 ReAct 等范式结合使用

局限：

- 依赖评估环节质量
- 反思本身也可能产生幻觉
- 多轮迭代会带来更高 token 成本

一句话总结：Reflexion 让模型不只是“会做”，还开始具备“做完之后检查并改进”的能力。
