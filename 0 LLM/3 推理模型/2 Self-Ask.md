# 一 Self-Ask

Self-Ask 是一种“先把问题拆成子问题，再逐步求解”的推理方式。  
它的核心不是让模型直接给最终答案，而是先问自己：

- 这个问题是否需要分解
- 应该先回答哪些中间问题
- 这些中间答案如何组合成最终结论

从这个角度看，Self-Ask 可以理解为一种更结构化的多步问答范式。  
它特别适合那些看起来像一个问题、实际上包含多层隐含推理步骤的任务。

## 1.1 Self-Ask 解决什么问题

很多复杂问题之所以难，不一定是因为单步推理本身特别难，而是因为问题内部藏着若干没有被显式写出的中间问题。

例如：

- 一个事实问题依赖多个前提事实
- 一个比较问题需要先分别查清两个对象
- 一个时间关系问题需要先理清事件顺序
- 一个综合判断问题需要先把信息拆开再汇总

如果模型直接一步到位回答，容易出现：

- 漏掉中间前提
- 把多个子问题混在一起
- 只答到局部，没有真正完成整体求解

Self-Ask 解决的就是这类“需要先拆再答”的问题。

## 1.2 核心思想

Self-Ask 的核心流程通常可以概括为：

1. 判断当前问题是否需要拆解
2. 生成一个或多个子问题
3. 逐个回答这些子问题
4. 根据中间答案汇总出最终结论

它与普通 Chain-of-Thought 的区别在于，Self-Ask 更强调把推理过程显式组织成“子问题序列”，而不是只输出一段连续思维链。

一个典型形式可以写成：

```text
Question: ...
Are follow up questions needed? Yes.
Follow up: ...
Intermediate answer: ...
Follow up: ...
Intermediate answer: ...
So the final answer is ...
```

这种形式的价值在于，模型不是模糊地“想一想”，而是明确进入一种子问题驱动的求解方式。

## 1.3 Self-Ask 的工作机制

### 1.3.1 先识别是否需要拆解

不是所有问题都适合 Self-Ask。  
对于非常直接的问题，强行拆分反而会增加冗余。

因此 Self-Ask 的第一步通常是判断：

- 这个问题是否包含多个隐含前提
- 是否需要中间事实才能推出结论
- 是否有必要先把任务拆开

### 1.3.2 生成子问题

一旦判断需要拆解，模型会把大问题改写成若干更小、更可回答的问题。  
这些子问题通常满足两个条件：

- 单步上更容易回答
- 答案对最终结论有明确贡献

### 1.3.3 中间答案逐步累积

每个子问题回答后，都会形成一个中间结论。  
这些中间结论会成为下一步推理的上下文。

因此 Self-Ask 的本质是一种“分解式状态累积”，而不是一次性长推理。

### 1.3.4 汇总为最终答案

当关键子问题都得到回答后，模型再把它们组合起来，形成最终结果。  
这一步要求模型具备一定的归纳和整合能力，否则会出现“子问题都答对了，但结论没拼对”的情况。

## 1.4 适用场景

Self-Ask 特别适合下面几类任务。

### 1.4.1 多跳问答

例如一个问题需要先找到人物、地点、时间等中间事实，再推导最终答案。  
这类任务天然适合拆成多个 follow-up question。

### 1.4.2 复杂事实组合

当结论依赖多个独立事实时，Self-Ask 能帮助模型把信息逐条理清，再做汇总。

### 1.4.3 文档理解与信息整合

如果问题需要从长文本中抽取多个片段再拼接理解，Self-Ask 也很有帮助，因为它鼓励模型显式问出“我还缺哪部分信息”。

### 1.4.4 需要任务拆解的分析问题

当一个问题本身带有多个维度，例如：

- 先定义问题
- 再找证据
- 再做比较
- 最后给结论

Self-Ask 往往比直接回答更稳。

## 1.5 与 CoT、ReAct 的区别

### 1.5.1 与 CoT 的区别

CoT 的重点是“把思路写出来”；  
Self-Ask 的重点是“把问题拆成子问题再解”。

两者都属于多步推理，但 Self-Ask 的结构更偏问答式，而 CoT 更偏连续思维链。

可以简单理解为：

- CoT：逐步想
- Self-Ask：逐步问、逐步答

### 1.5.2 与 ReAct 的区别

ReAct 不只会拆问题或推理，还会调用外部动作，例如搜索、工具、环境交互。  
Self-Ask 本身主要还是一种内部推理框架。

所以：

- Self-Ask 更像“结构化自问自答”
- ReAct 更像“推理 + 行动 + 观察”的闭环

如果任务需要外部检索，Self-Ask 经常会与搜索工具结合使用；  
一旦加入外部动作，它就开始接近 ReAct 或检索增强推理。

## 1.6 优点

### 1.6.1 推理结构更清晰

由于子问题被显式写出来，整个推理链通常更容易理解和审查。

### 1.6.2 更适合多跳问题

对那些必须依赖中间事实的问题，Self-Ask 往往比直接长答案更稳定。

### 1.6.3 更容易发现缺失信息

当模型开始问子问题时，往往更容易暴露“自己其实还不知道什么”，这比直接给出一个看似完整但实际不可靠的答案更好。

### 1.6.4 更便于和检索结合

因为 Self-Ask 天然产出中间问题，所以这些问题很适合进一步拿去做检索、搜索或知识库查询。

## 1.7 局限

### 1.7.1 拆分本身可能出错

如果子问题拆错了，后面的推理再严谨也会建立在错误结构上。

### 1.7.2 不一定适合简单问题

对于简单直接的问题，Self-Ask 可能只是增加步骤和 token 开销，并不会显著提升质量。

### 1.7.3 仍然可能产生幻觉

如果 Self-Ask 只在模型内部进行，而不借助外部证据，那么中间答案依然可能是错的。  
它只是把错误显式拆开了，并不天然消除幻觉。

### 1.7.4 子问题过多时会变得冗长

拆得过细会导致流程拖沓，甚至让模型陷入不必要的中间步骤。

## 1.8 工程上的常见用法

在实际系统里，Self-Ask 常见有两种使用方式。

### 1.8.1 纯提示式

直接通过 prompt 要求模型先判断是否需要 follow-up questions，再给中间答案和最终答案。

这种方式实现简单，适合快速实验。

### 1.8.2 与检索或工具联动

先让模型生成子问题，再把这些子问题送去搜索、RAG 或工具调用，最后汇总结果。  
这种方式更接近真实系统，因为它把“问题拆解”变成了后续检索与行动的入口。

## 1.9 一个简化示例

例如问题是：

```text
加拿大现任总理出生时，英国君主是谁？
```

如果直接回答，模型容易混淆时间和人物关系。  
而 Self-Ask 会先拆成：

1. 加拿大现任总理是谁
2. 他出生于哪一年
3. 那一年英国君主是谁

最后再合成结论。  
这类问题就很适合 Self-Ask。

## 1.10 总结

Self-Ask 的核心价值，不是让模型“想得更长”，而是让模型先把复杂问题拆成可回答的中间问题，再逐步完成求解。  
它特别适合多跳问答、复杂事实组合和需要显式任务拆解的场景。

可以用一句话概括：

**Self-Ask 通过“先问清楚中间问题，再回答最终问题”的方式，把复杂推理变成更可管理的分步问答。**
