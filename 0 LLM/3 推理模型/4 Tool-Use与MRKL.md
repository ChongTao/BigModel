# 一 Tool-Use / MRKL

Tool-Use 指的是：模型在生成过程中，不只依赖内部参数，还能通过外部工具获得真实信息或执行操作。

这类模式的重点，不是“模型想得多复杂”，而是“模型能不能把外部能力接进求解过程”。

## 1.1 它解决什么问题

如果只靠模型参数，常见问题包括：

- 知识过期
- 事实不准
- 无法执行外部操作
- 无法访问数据库、搜索系统、计算器等工具

Tool-Use 的价值就在于：把外部能力接进来，弥补纯生成的不足。

## 1.2 核心机制

这类模式通常都有一个基本结构：

`Action -> Observation`

也就是说：

1. 模型决定要调用什么工具
2. 系统执行工具
3. 工具返回结果
4. 模型再基于这个结果继续回答或继续决策

## 1.3 常见适用场景

- API 工作流
- 搜索
- 数据库查询
- 计算器
- 文件读写

## 1.4 优点

- 可接入真实外部数据
- 可显著降低事实型幻觉
- 能从“会说”扩展到“能做”

## 1.5 局限

- 需要定义好工具协议
- 工具返回格式必须足够规范
- 单纯 Tool-Use 不一定天然提供完整推理闭环

这也是为什么 ReAct 往往被看作 Tool-Use 的更结构化形式：它不仅会调用工具，还会把“推理、行动、观察”串成一个循环。

## 1.6 MRKL 可以怎么理解

MRKL（Modular Reasoning, Knowledge and Language）可以理解为一种模块化的工具增强思路：

- 模型负责理解问题和做路由
- 不同工具模块负责执行各自擅长的能力

所以它更像是在强调：

**复杂系统不必把所有能力都塞进模型本身，而可以由模型负责调度多个外部模块。**

## 1.7 一句话总结

Tool-Use / MRKL 的核心价值是：**让模型不只依赖内部参数回答，而是能通过外部工具获取信息和执行动作。**
