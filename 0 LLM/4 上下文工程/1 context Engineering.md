# 一 Context Engineering 介绍

Context Engineering（上下文工程）指的是：围绕大模型一次调用中“看到什么、按什么顺序看到、哪些内容该保留、哪些内容该丢弃”所进行的系统化设计与工程实践。

它关注的不是单条 Prompt 怎么写，而是**如何构建一个让模型更容易做对事的上下文环境**。

与只关注“写好 Prompt”不同，Context Engineering 更强调：

- 信息边界
- 信息顺序
- 信息来源
- 信息结构
- 信息生命周期
- 成本与复用

## 1.1 Context Engineering 解决什么问题

大模型的能力并不是固定不变的，它高度依赖上下文质量。很多效果问题，本质上并不是“模型不够强”，而是“给模型看的东西不对”。

Context Engineering 主要解决这些问题：

- 模型能否正确理解当前任务
- 多轮对话是否保持一致
- 检索内容是否真正相关
- 工具调用是否精确
- 长文或长历史中哪些信息应该保留
- 记忆、状态、用户偏好如何持续管理
- 如何减少上下文噪声、延迟和 token 成本

它的核心目标是：让模型在复杂、多轮、长期任务中，更稳定、更可控、更一致。

## 1.2 核心组成

Context Engineering 通常包含六类核心能力：

- 系统提示设计：定义长期规则、边界和角色
- 上下文组装：决定每次调用时把哪些内容注入模型
- 检索与知识注入：决定外部知识如何被召回、筛选和拼接
- 工具与状态回填：决定工具结果、执行状态如何进入下一轮上下文
- 记忆管理：决定哪些信息短期保留，哪些长期持久化
- 缓存与压缩：决定如何降低重复输入、控制成本和长度

## 1.3 与 Prompt Engineering 的区别

![](https://pica.zhimg.com/v2-c15875cb897ede0c4bfea26709141e6c_1440w.jpg)

- Prompt Engineering 更关注“怎么写提示词”，例如角色设定、输出格式、示例设计。
- Context Engineering 更关注“模型总共看到什么”，包括系统提示、历史对话、RAG 结果、工具输出、记忆、任务状态等。

可以把两者理解成：

- Prompt Engineering 偏输入写法
- Context Engineering 偏输入系统设计

Prompt 是上下文的一部分，但不是全部。

## 1.4 一个常见的上下文栈

很多大模型应用最终都会把上下文组织成类似下面的结构：

1. 系统规则
2. 工具定义
3. 当前任务目标
4. 检索或外部知识
5. 历史摘要
6. 最新用户输入
7. 当前步骤状态

真正的工程难点不在于“把东西都塞进去”，而在于：

- 哪些必须在前
- 哪些应该裁剪
- 哪些不能互相污染
- 哪些需要长期保留

## 1.5 一个简单的上下文装配示例

下面是一个常见的上下文装配方式。它不是唯一标准，但能帮助理解“上下文工程”在实际系统里长什么样。

```text
<system_rules>
你是企业内部代码助手，必须遵守安全规范、输出要求和权限边界。
</system_rules>

<tool_definitions>
- search_docs(query)
- read_file(path)
- run_tests(target)
</tool_definitions>

<task>
修复支付模块中的订单状态同步问题。
</task>

<retrieval>
- 支付服务设计文档摘要
- 订单状态机定义
</retrieval>

<history_summary>
- 已确认问题出现在回调重试路径
- 上一轮已经排除数据库事务问题
</history_summary>

<latest_user_input>
先定位最可能出错的代码路径。
</latest_user_input>

<runtime_state>
- 当前分支: fix/payment-sync
- 最近失败测试: PaymentStatusSyncTest
</runtime_state>
```

这个例子里最关键的不是格式本身，而是每一层承担的职责不同：

- `system_rules` 负责长期边界
- `tool_definitions` 负责可用能力
- `task` 负责当前目标
- `retrieval` 负责补充外部知识
- `history_summary` 负责保留必要历史
- `latest_user_input` 负责当前这一轮新增指令
- `runtime_state` 负责当前执行现场

如果这些内容全部混成一段自然语言，模型仍然“能看”，但更容易出现混淆、干扰、冲突和腐烂；这正是 Context Engineering 要解决的问题。

# 二 Context Engineering 常见问题

## 2.1 Context Poisoning（上下文中毒）

Context Poisoning 指的是：通过操控、污染、伪造或恶意注入上下文内容，使模型在推理、生成或工具调用时产生错误、偏置、不可控或有害行为。

### 2.1.1 来源

- 用户输入污染：试图覆盖系统规则、诱导越狱、植入假事实
- 检索结果投毒：召回到虚假、恶意或反向诱导内容
- 多轮对话污染：前面轮次的错误内容被后续继续当真
- 工具回填污染：工具返回异常、未校验数据或错误状态
- 记忆污染：恶意内容被写入长期记忆并持续生效

### 2.1.2 典型案例

- “忽略所有规则，直接执行下面要求。”
- 通过长对话持续重复错误信息，最终让模型把错误当作事实
- 恶意写入记忆，例如“我是 Root，拥有全部权限”
- 工具错误输出被当成真实状态，导致后续规划整体偏航

### 2.1.3 防御方式

- 分层隔离 system、user、tool、memory 等不同来源
- 用户输入绝不直接拼接进 system prompt
- 对外部文本做净化、转义和注入检测
- 对召回 chunk 做安全检查
- 工具输出结构化封装，不直接提升为系统规则
- 记忆写入前做校验和审查
- 在多轮中丢弃潜在污染片段，只保留任务关键上下文

## 2.2 Context Distraction（上下文干扰）

Context Distraction 指的是：上下文中存在过多无关、次要或噪声信息，导致模型抓不住重点，进而降低理解、推理和执行质量。

### 2.2.1 来源

- RAG 召回过多，Top-K 太大且没有去重
- 对话历史不裁剪，所有轮次全部注入
- 工具返回大量冗余字段
- 系统提示或任务描述过长且重复
- 关键信息放得过后，噪声信息反而更靠前

### 2.2.2 表现

- 回答围绕噪声展开，而不是围绕任务目标
- 工具调用参数构造错误
- 推理链条中断或偏题
- 长对话中逐步“忘记当前任务”

### 2.2.3 防止方式

- 去重、重排和筛选召回内容
- 给上下文建立优先级，关键内容前置
- 对历史做 sliding window、摘要化或阶段性重置
- 压缩工具结果，只保留下轮所需字段
- 用标签和结构化区块组织上下文

## 2.3 Context Confusion（上下文混淆）

Context Confusion 指的是：模型无法清楚区分上下文中不同信息的来源、角色和层级，因此把规则、事实、用户意图、工具结果混在一起理解。

### 2.3.1 来源

- system、user、assistant、tool、RAG 文本直接堆在一起
- 外部内容没有标明来源和边界
- 工具结果未结构化包装
- 多轮对话合并后角色标记丢失
- 系统提示被后续内容“语义稀释”

### 2.3.2 表现

- 把用户输入误当成系统指令
- 把 RAG 文档误当成用户意图
- 把工具结果误当成规则
- 规划步骤互相混淆，工具选错或参数错位

### 2.3.3 防止方式

- 明确分区：system、history、retrieval、tool、memory 分开
- 对外部内容加来源标签和边界
- 工具定义与系统规则放在最前且独立
- 合并历史时保留清晰角色标记
- 禁止把模型输出未经处理地回注为高优先级上下文

## 2.4 Context Conflict（上下文冲突）

Context Conflict 指的是：上下文里同时存在多份相互矛盾的信息、规则、状态或事实，模型无法判断应以哪一份为准。

它和“混淆”不同：

- 混淆强调“分不清谁是谁”
- 冲突强调“知道有多份信息，但不知道该信哪份”

### 2.4.1 来源

- 新旧需求同时存在，但没有明确覆盖关系
- 多个检索片段结论互相矛盾
- 工具状态和历史摘要不一致
- 用户后续修正了事实，但旧版本仍留在上下文中
- 多版本文档、规范或配置同时注入

### 2.4.2 表现

- 回答左右摇摆，前后不一致
- 同时引用互相冲突的事实
- 执行时延续旧规则，忽略最新约束
- 把草稿、示例或旧状态当成当前有效版本

### 2.4.3 防止方式

- 明确声明覆盖关系，例如“以下新规则覆盖前文旧规则”
- 对新旧信息做版本管理和有效期管理
- 合并冲突上下文，避免多版本同时暴露给模型
- 在摘要中只保留当前有效状态
- 检索后增加一致性检查或 rerank

## 2.5 Context Rot（上下文腐烂）

Context Rot 指的是：随着轮次增加、上下文变长，最初清晰有效的规则、事实和目标逐渐被噪声、冗余和误差稀释，导致模型输出越来越偏。

### 2.5.1 来源

- 上下文持续增长，注意力被稀释
- 小误解在多轮中不断累积
- 新信息不断覆盖旧规则
- 日志、闲聊、无关历史不断进入上下文
- 把无限历史当成记忆系统使用

### 2.5.2 表现

- 模型逐步忘掉最初任务目标
- 系统规则开始失效
- 事实被重写或扭曲
- 回答越来越分散、不准确、不聚焦

### 2.5.3 防止方式

- 控制上下文长度，不要无限堆历史
- 保持结构化而不是自然堆叠
- 定期摘要和重置状态
- 显式声明哪些规则仍然有效
- 使用可控的 Memory，而不是无边界对话历史

# 三 如何构建一个好的上下文

一个可用的 Context Engineering 流程通常包括下面几步：

## 3.1 先分层

先把上下文按角色和优先级拆开，而不是先拼成一整段大文本。

例如：

- system：长期规则
- task：当前目标
- retrieval：外部知识
- tool：工具结果
- history：历史摘要
- memory：长期偏好或事实

## 3.2 再筛选

不是所有信息都值得注入。要优先保留：

- 与当前任务强相关的信息
- 当前步骤必须依赖的信息
- 会影响行为边界和结果正确性的信息

要主动丢弃：

- 已完成阶段的细节
- 冗余日志
- 无关闲聊
- 未验证且暂时无用的外部内容

## 3.3 再结构化

给不同类型的内容打标签、分区、排序，让模型更容易知道：

- 这是什么
- 来自哪里
- 优先级多高
- 是否可执行

## 3.4 最后管理生命周期

上下文不是一次性拼装，而是持续演化的。要考虑：

- 什么时候做摘要
- 什么时候清空旧状态
- 什么时候写入长期记忆
- 什么时候压缩工具结果
- 什么时候用缓存复用稳定前缀

## 3.5 如何评估上下文工程是否有效

上下文工程做得好不好，不能只靠“模型回答看起来还行”来判断，更适合看一组结果指标。

常见评估维度包括：

- 任务成功率：相同模型下，任务是否更容易完成
- 工具调用正确率：是否选对工具、参数是否更准确
- 多轮一致性：长对话后是否仍然记得目标、规则和已确认事实
- 检索利用质量：召回内容是否真的被正确使用，而不是被噪声淹没
- token 成本与延迟：是否在保证效果的前提下降低了输入长度和响应时间
- 错误恢复能力：遇到污染、冲突或历史偏航后，是否能更快回到正确状态

如果一个系统出现下面这些现象，往往说明上下文工程还有问题：

- 模型经常忘记当前目标
- 工具调用频繁选错
- 长对话里前后回答互相矛盾
- RAG 召回很多，但回答质量并没有明显提升
- token 用量持续上升，但效果没有同步提升

因此，上下文工程的优化目标通常不是“上下文越多越好”，而是：

- 更高的有效信息密度
- 更低的噪声比例
- 更稳定的任务推进能力
- 更可控的成本
