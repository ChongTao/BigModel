# 一 上下文压缩

**上下文压缩（Context Compression）** 指的是：在上下文窗口有限、对话持续变长或外部信息过多时，对上下文进行摘要化、抽取化、筛选化或结构化处理，以减少 token 消耗，同时尽量保留当前任务真正需要的信息。

它的目标不是“把所有内容都缩短”，而是：

- 保留关键任务信息
- 丢弃冗余和噪声
- 控制成本和延迟
- 降低 Context Distraction 与 Context Rot

## 1.1 为什么需要上下文压缩

模型的 context window 始终是有限资源。上下文过长时，常见问题包括：

- 计算成本升高
- 延迟增加
- 注意力被噪声稀释
- 关键内容被淹没
- 多轮历史逐渐腐烂、偏航

所以，上下文压缩本质上是一种**上下文预算管理**能力。

## 1.2 上下文压缩不等于“简单摘要”

很多人把上下文压缩理解成“把历史总结一下”，这只对了一部分。

真正的上下文压缩至少有四类思路：

- 抽象压缩：保留语义，不保留原句
- 抽取压缩：保留原始事实、字段、片段
- 选择性压缩：只保留与当前任务相关的部分
- 结构化压缩：把杂乱内容整理成稳定结构

不同任务需要的压缩方式并不一样。

# 二 常见压缩方式

## 2.1 抽象压缩（Abstractive Compression）

抽象压缩可以理解为“模型重新写一个更短的版本”，重点在于**保留语义和任务状态**，而不是保留原始表述。

例如：

输入：

> 我问你如何实现一个 gRPC 客户端，你给了我示例代码，并解释了连接池和超时设置。现在我准备接着处理重试机制。

压缩后：

> 用户正在实现 gRPC 客户端，已完成连接池和超时设置讨论，下一步要处理重试机制。

适用场景：

- 多轮长对话
- 项目推进类任务
- 继续推理比保留原句更重要的场景

优点：

- 压缩率高
- 适合延续任务状态

局限：

- 可能丢失精确细节
- 可能引入摘要偏差

提示模板示例：

```text
请将以下历史对话压缩为不超过 N 字的摘要，仅保留：
1. 当前任务目标
2. 已确认的关键结论
3. 未完成事项
4. 用户明确提出的约束

不要加入原文中没有的信息。
```

## 2.2 抽取压缩（Extractive Compression）

抽取压缩不是“重新概括”，而是**从原文中挑出必须保留的事实、字段、代码或约束**，尽量不改写。

例如要保留的内容可能包括：

- API schema
- 关键配置项
- 用户明确需求
- 代码片段
- 已确认的决策点

![](https://www.dailydoseofds.com/content/images/2025/05/summarizer.svg)

适用场景：

- 技术讨论
- 代码生成与调试
- 文档问答
- 需要高精度保真的场景

优点：

- 不容易改写原意
- 适合保留关键细节

局限：

- 压缩率通常低于抽象压缩
- 内容可能仍然偏长

提示模板示例：

```text
从以下历史内容中，仅抽取后续任务必需的信息：
- 事实
- 参数
- 配置
- 代码
- 约束

不要改写，不要总结，不要推理，只做抽取。
```

## 2.3 选择性压缩（Selective Compression）

选择性压缩的核心不是“怎么缩写”，而是“**哪些内容根本不应该继续保留**”。

典型做法是：

历史对话 -> 相关性判断 -> 只留下与当前任务强相关的部分

适用场景：

- 多任务混杂的长会话
- 有大量闲聊、噪声、旁支讨论
- Agent 在长流程中切换阶段

优点：

- 对控制噪声非常有效
- 能显著降低上下文干扰

局限：

- 相关性判断做错会误删关键信息

提示模板示例：

```text
只保留与“当前任务”直接相关的内容。
对于已经完成的阶段、无关闲聊、扩展讨论，全部删除。
不要总结无关内容，只保留与当前任务强相关的信息。
```

## 2.4 结构化压缩（Structural Compression）

结构化压缩指的是：把原本杂乱的对话、日志、工具回填结果，整理成稳定的结构化上下文。

例如整理成：

- Goals
- Constraints
- Known Facts
- Decisions
- Open Issues
- Todo

这种方式的关键优势在于：**不仅缩短了内容，还提升了可读性和可执行性。**

适用场景：

- 项目式对话
- Agent 工作流
- 长任务状态管理
- 工具密集型系统

优点：

- 稳定
- 可控
- 便于后续继续执行

局限：

- 需要预先设计结构
- 不适合所有开放式对话

提示模板示例：

```text
请将历史对话整理为以下结构：

# Goals
# Constraints
# Known Facts
# Decisions
# Todo

保持内容准确，不添加原文未提供的信息。
```

# 三 如何选择压缩策略

可以按任务类型做一个简单判断：

- 如果重点是“延续任务状态”，优先用抽象压缩
- 如果重点是“保留准确细节”，优先用抽取压缩
- 如果重点是“减少噪声”，优先用选择性压缩
- 如果重点是“维持长期状态机”，优先用结构化压缩

很多系统并不是只用一种，而是组合使用：

- 先选择性过滤
- 再抽取关键字段
- 最后结构化整理

## 3.1 什么时候触发上下文压缩

上下文压缩不是每一轮都必须做，它更像一种在特定条件下触发的运行策略。

常见触发时机包括：

- 接近上下文窗口上限时：输入 token 持续增长，已经开始挤压有效信息空间
- 多轮对话达到一定轮次时：历史越来越长，继续原样保留的收益已经下降
- 任务阶段切换时：上一阶段的细节不必原样带入下一阶段，只需要保留结果和状态
- 工具返回过长时：日志、搜索结果、接口返回体过大，不适合直接原样回填
- 检索结果进入上下文前：召回块太多、太长时，先压缩再注入通常更稳
- 准备写入记忆前：不是所有历史都值得持久化，常常需要先压缩成更稳定的摘要或结构

可以把它理解成一个简单原则：

- 上下文还短、信息密度高时，不急着压
- 上下文开始变长、变乱、变贵时，就该考虑压缩

很多系统里，上下文压缩的触发条件会直接做成规则，例如：

- 超过固定 token 阈值就触发
- 连续 N 轮后触发一次阶段性摘要
- 工具输出超过长度限制时只保留关键字段
- 检索召回超过 Top-K 或总长度阈值时先做筛选与压缩

所以，压缩策略不仅包括“怎么压”，也包括“何时压、压什么、压到什么程度”。

## 3.2 压缩对象通常有哪些

除了“用什么方法压缩”，另一个更工程化的问题是：“到底在压什么”。

常见压缩对象通常有下面几类：

### 3.2.1 对话历史压缩

这是最常见的一类。目标是把越来越长的历史对话，压缩成当前任务仍然需要的摘要、事实或状态。

常见保留内容：

- 当前目标
- 已确认结论
- 未完成事项
- 用户明确约束

### 3.2.2 检索结果压缩

RAG 场景下，召回结果经常过长、过多、重复。检索结果压缩的目标是把“可能相关”变成“足够回答问题”。

常见做法：

- 去重
- rerank 后只保留高相关片段
- 把长段落压成关键事实
- 提取引用片段而不是整段注入

### 3.2.3 工具输出压缩

很多工具返回的是原始结构化数据、长日志或完整接口响应，这些内容通常不适合直接原样进入下一轮上下文。

常见保留内容：

- 关键字段
- 错误摘要
- 对下一步决策有用的状态变化

例如：

- 测试结果只保留失败用例和错误摘要
- 搜索结果只保留高相关条目
- API 返回只保留业务字段，不保留完整元数据

### 3.2.4 运行状态压缩

在长任务或 Agent 工作流里，系统往往还会维护运行状态，例如：

- 当前阶段
- 已完成步骤
- 待执行步骤
- 已知阻塞点

这类内容更适合被整理成结构化状态，而不是散落在自然语言历史里。

### 3.2.5 记忆写入前压缩

并不是所有历史都值得进入长期记忆。很多时候要先把多轮内容压缩成更稳定、更抽象的事实或偏好，再决定是否写入 memory。

例如：

- 用户长期偏好
- 已验证的项目规则
- 稳定有效的任务经验

这一步的重点不是保留全部细节，而是保留长期有复用价值的内容。

# 四 常见风险

上下文压缩本身也可能制造问题，常见风险包括：

- 摘要遗漏关键约束
- 抽取不完整，漏掉关键参数
- 相关性判断错误，误删重要内容
- 压缩后失去来源信息
- 压缩结果反复回注，进一步放大早期错误

因此，一个好的压缩系统通常需要：

- 明确压缩目标
- 明确保留字段
- 保留来源边界
- 定期重建压缩结果，而不是无限迭代压缩

## 4.1 如何判断“压缩过头了”

上下文压缩的难点不在于“能不能压短”，而在于“压短之后任务还能不能做好”。

如果压缩过头，常见现象包括：

- 模型忘记关键约束
- 工具调用开始变错
- 回答看似流畅，但细节明显失真
- 长任务中阶段状态丢失
- 检索结果虽然变短了，但回答质量反而下降

所以，压缩质量不能只看 token 是否下降，还要看关键能力是否被保住。

## 4.2 常见评估指标

评估上下文压缩时，常见可以观察这些指标：

- 关键信息保留率：任务目标、约束、参数、状态是否仍然完整
- 任务成功率：压缩前后，同类任务完成率是否下降
- 工具调用正确率：函数、工具、参数是否更容易出错
- 回答质量：是否更容易偏题、遗漏、幻觉
- token 节省比例：压缩后到底节省了多少输入长度
- 延迟变化：压缩是否真的带来更快响应

如果压缩后只降低了 token，但任务成功率明显下滑，那么这次压缩通常就是失败的。

## 4.3 一个简单原则

一个好的压缩策略，通常应该同时满足三件事：

- 信息更短
- 任务不坏
- 成本更低

如果只能做到“更短”，但做不到“任务不坏”，那它就不是好的上下文压缩。
