# 一 推理工程概述

## 1.1 推理工程是什么

推理工程关注的不是“模型会不会生成”，而是“模型能不能在真实系统里稳定、低延迟、可扩展地生成”。  
当大模型从研究环境进入生产环境后，问题会迅速从“参数规模够不够大”转向“显存够不够、并发能不能扛、延迟能不能压住、成本能不能接受、系统能不能持续运行”。

因此，推理工程讨论的是模型部署之后的一整套执行问题，包括：

- 模型如何加载
- 请求如何进入与排队
- GPU 资源如何分配
- KV Cache 如何管理
- 多请求如何批处理
- 如何在吞吐、延迟、成本之间取平衡

如果说训练工程决定了模型“学到了什么”，那么推理工程更接近于决定模型“在真实世界里能以什么代价把能力交付出来”。

## 1.2 推理系统要解决的核心问题

从工程视角看，一个推理系统通常要同时解决五类问题。

第一类是**资源问题**。  
大模型推理高度依赖显存、带宽和算力，而这些资源都是有限且昂贵的。模型越大、上下文越长、并发越高，系统就越容易触碰资源瓶颈。

第二类是**调度问题**。  
真实请求不会整齐地一起到达，也不会同时结束。不同请求的 prompt 长度、生成长度、优先级、响应时间要求都不一样，因此系统必须动态调度，而不是像离线训练那样使用固定 batch。

第三类是**性能问题**。  
推理系统不仅要能跑，还要跑得“像服务”。用户通常关心首 token 延迟、完整响应时间和高峰期稳定性，而平台侧还会关心吞吐、GPU 利用率和单位请求成本。

第四类是**兼容与部署问题**。  
不同模型架构、量化方式、硬件平台、服务协议和部署环境之间存在大量差异。一个可用的推理系统需要在模型能力和工程落地之间做适配。

第五类是**可靠性问题**。  
一旦进入生产环境，推理系统就不只是“能生成文本”这么简单，它还必须具备监控、容错、限流、灰度、回滚和容量规划能力。

## 1.3 推理链路的关键组成

一个典型的推理链路，通常可以拆成下面几个部分：

1. **模型加载层**  
负责权重加载、格式转换、量化适配、设备放置和运行时初始化。

2. **请求接入层**  
负责接受外部请求、身份校验、限流、排队和协议兼容，例如 OpenAI 风格 API 或自定义 RPC。

3. **调度与批处理层**  
负责把不同请求映射到执行队列中，决定哪些请求一起进入 prefill，哪些请求在 decode 阶段被动态合批。

4. **执行引擎层**  
这是推理系统的核心执行部分，真正负责 attention、矩阵计算、KV Cache 读取与写入、采样与解码。

5. **缓存与状态管理层**  
包括 KV Cache、prefix cache、会话状态、请求上下文等。很多性能提升都来自这一层，而不是模型本身。

6. **输出与观测层**  
负责流式输出、日志、指标、追踪、错误上报和 benchmark 数据采集。

从这个角度看，推理框架并不只是“跑模型的库”，它更像是一个围绕模型执行构建的小型操作系统。

## 1.4 常见优化方向

推理工程的优化通常集中在几个固定方向。

### 1.4.1 内存优化

这类优化主要解决“模型能不能放得下、上下文能不能撑得住”的问题，常见手段包括：

- KV Cache 优化
- PagedAttention
- 权重量化
- 激活与缓存压缩
- 更高效的显存复用

### 1.4.2 调度优化

这类优化解决“请求怎么排得更合理”，典型手段包括：

- Continuous Batching
- Prefill / Decode 分离调度
- 多优先级队列
- 长短请求隔离
- 请求路由与副本分配

### 1.4.3 执行优化

这类优化更偏底层算子与执行路径，包括：

- Kernel 融合
- 更高效的 attention 实现
- CUDA / TensorRT / Triton 优化
- Speculative Decoding
- MoE 推理优化

### 1.4.4 系统优化

这类优化不直接改变单次计算速度，但会明显影响整体服务表现，例如：

- Prefix Cache
- 响应流式化
- 热点模型预热
- 自动扩缩容
- 监控与异常恢复

这些方向往往不是互斥的。一个成熟系统通常会同时做内存、调度、执行和系统层优化。

## 1.5 推理工程与训练、微调、框架选型的关系

推理工程和训练工程密切相关，但关注点不同。

- **训练工程** 关心模型如何学到能力
- **微调工程** 关心模型如何适配特定任务
- **推理工程** 关心模型如何把能力稳定交付出来

训练阶段的很多设计会直接影响推理难度。例如模型参数规模、上下文长度、架构类型、是否为 MoE、是否适合量化，都会决定推理系统后续的资源压力。

同样，推理框架选型也不是独立问题。  
选择 vLLM、TGI、TensorRT-LLM、llama.cpp 之类方案时，本质上不是在选“哪个工具更流行”，而是在选：

- 更偏吞吐还是更偏低延迟
- 更偏通用 GPU 还是更偏 NVIDIA 深度优化
- 更偏云端服务还是更偏本地部署
- 更偏标准接口还是更偏底层控制

因此，推理工程并不是部署阶段最后补的一层，而是贯穿模型能力、硬件条件和业务目标之间的桥梁。

## 1.6 常见指标与权衡

推理系统通常要同时看几类指标。

| 指标 | 关注点 |
| :--- | :--- |
| TTFT | 首 token 延迟，影响用户“等多久才看到响应” |
| TPOT / tokens per second | 生成速度，影响流式体验 |
| 吞吐量 | 单位时间内能处理多少请求 |
| GPU 利用率 | 资源是否真正被吃满 |
| P95 / P99 延迟 | 高峰期尾部请求是否失控 |
| 单请求成本 | 每次调用的资源消耗是否可接受 |

这些指标之间经常相互冲突。

- 提高 batch 往往有助于吞吐，但可能拉高首 token 延迟
- 更激进的缓存复用可以降低成本，但会增加系统复杂度
- 更重的量化可以节省显存，但可能损失质量或兼容性
- 更复杂的调度策略可以提高整体利用率，但也可能带来不可预测性

所以推理工程很少存在“唯一最优解”，更常见的是围绕具体场景做取舍：

- 面向聊天产品，更重视交互延迟
- 面向批量离线生成，更重视吞吐与成本
- 面向企业平台，更重视稳定性、兼容性和可观测性

## 1.7 总结

推理工程的核心，不是把模型“跑起来”，而是把模型变成一个真正可交付的服务能力。  
当模型进入真实环境后，瓶颈往往不再只是模型质量，而是资源、调度、缓存、延迟、吞吐和可靠性这些工程问题。

可以用一句话概括：

**训练决定模型能做到什么，推理工程决定这些能力能否以可接受的成本和体验被稳定使用。**
