# 二 KV Cache 与 PagedAttention

## 2.1 为什么推理离不开 KV Cache

在大语言模型推理里，用户每生成一个新 token，模型都要再做一次 attention 计算。  
如果每一步都把历史 token 的 Key 和 Value 重新算一遍，那么随着上下文变长，重复计算会迅速变多，延迟也会持续上升。

KV Cache 的核心作用，就是把历史 token 在每一层 attention 中产生的 `K` 和 `V` 保存下来，后续解码时直接复用，而不是重复计算。  
因此它本质上是一个**用显存换时间**的机制：

- 不用 KV Cache：每一步都要重新处理全部历史上下文
- 使用 KV Cache：只需要计算新 token 的 `Q/K/V`，再与历史缓存做 attention

对推理系统来说，这个优化不是“可选增强”，而是长上下文和高并发场景下的基础能力。没有 KV Cache，大模型服务很难同时满足延迟、吞吐和成本要求。

## 2.2 KV Cache 的基本工作方式

以 decoder-only 模型为例，一次请求大致分为两个阶段：

### 2.2.1 Prefill 阶段

用户输入 prompt 后，模型会先处理整段输入。  
这个阶段会一次性计算出各层的历史 token 对应的 `K` 和 `V`，并把它们写入 KV Cache。

这个阶段的特点是：

- 计算量大，因为要处理完整 prompt
- 并行度高，适合做大矩阵计算
- 通常决定首 token 延迟（TTFT）的上限

### 2.2.2 Decode 阶段

生成阶段每次只新增一个 token。  
此时历史 token 的 `K` 和 `V` 已经在缓存里，不需要重复计算，只需：

1. 为当前 token 计算新的 `Q/K/V`
2. 将新的 `K/V` 追加进缓存
3. 用当前 `Q` 和全部历史 `K/V` 做 attention
4. 采样出下一个 token

因此 decode 的瓶颈通常不再是“重复算历史”，而是：

- 从 KV Cache 里读历史块
- 在多请求之间做调度
- 控制显存占用和碎片

## 2.3 KV Cache 带来的收益与成本

### 2.3.1 直接收益

- 显著减少 decode 阶段的重复计算
- 上下文越长，收益通常越明显
- 使流式生成的 token 间延迟更稳定
- 为 continuous batching、prefix reuse 等机制提供基础

### 2.3.2 直接成本

KV Cache 节省了算力，但会消耗大量显存。  
随着以下因素增加，缓存开销会快速上升：

- 模型层数更多
- hidden size 更大
- head 数量更多
- 上下文长度更长
- batch / 并发请求更多
- 数据类型精度更高，例如 FP16 高于 FP8 / INT8

所以很多时候，线上系统的真正限制不是“算不动”，而是“KV Cache 放不下”。

## 2.4 如何理解 KV Cache 的内存占用

可以把 KV Cache 粗略理解为：

`显存开销 ~= 层数 × token 数 × 每个 token 的 K/V 大小`

更直观地说，单个 token 进入缓存后，几乎会在每一层都留下一份 Key 和一份 Value。  
因此上下文长度从 `4k` 增加到 `32k`，缓存并不是小幅增加，而是近似按 token 数线性增长。

工程上通常需要同时关注三类容量：

- **单请求容量**：一个长上下文请求最多占多少缓存
- **总池容量**：整张卡或整个实例能容纳多少 token 的缓存
- **碎片后可用容量**：理论容量之外，真正还能分配出去多少

这也是为什么很多服务“理论上显存够”，但高并发下仍然会 OOM 或频繁触发回收。

## 2.5 KV Cache 的生命周期

从系统角度看，KV Cache 不是一个静态数组，而是有完整生命周期的运行时资源。

### 2.5.1 分配

请求进入 prefill 后，需要为其分配缓存空间。  
如果系统使用连续内存布局，往往要一次申请较大连续区域。

### 2.5.2 写入

每处理一个 token，就向对应层的缓存写入新的 `K/V`。  
写入过程需要和调度器配合，保证不同请求不会互相覆盖。

### 2.5.3 复用

如果多个请求共享相同前缀，系统可能直接复用已有 prefix cache，而不是重新计算全部 prompt。

### 2.5.4 扩展

随着 decode 持续进行，请求会不断追加新 token。  
这要求缓存能动态增长，而不是只适合固定长度。

### 2.5.5 回收

请求结束、超时、取消或被抢占后，对应缓存必须及时释放。  
如果回收不及时，会出现：

- 显存占用长期偏高
- 新请求无法进入
- 系统误判容量不足

## 2.6 KV Cache 相关的核心瓶颈

### 2.6.1 显存容量瓶颈

这是最直接的问题。  
长上下文、多轮对话和高并发会同时推高缓存总量，导致服务实例能承接的请求数快速下降。

### 2.6.2 内存碎片问题

如果缓存按请求连续分配，那么不同请求长度不一、结束时间不同，很容易把显存切碎。  
碎片的结果往往不是“显存完全用满”，而是“还有剩余，但没有合适的大块可分配”。

### 2.6.3 带宽瓶颈

Decode 阶段经常受内存读写影响。  
即使算力还够，如果每一步都要从显存读取大量历史 `K/V`，系统也可能先撞上显存带宽上限。

### 2.6.4 调度耦合问题

KV Cache 不是独立存在的。  
它和 batching、抢占、优先级、prefix reuse、请求取消策略都强耦合。调度器设计不当时，缓存利用率会明显下降。

## 2.7 PagedAttention 在解决什么问题

PagedAttention 的核心目标，不是改变 attention 数学形式，而是改变 **KV Cache 的组织方式**。

传统做法常把一个请求的缓存视为一段连续内存。  
这种方式简单，但在动态请求场景下有两个明显问题：

- 请求长度事先很难精确预测
- 连续内存扩容和回收容易造成严重碎片

PagedAttention 借鉴了操作系统“页”的思想，把 KV Cache 拆成固定大小的 block / page：

- 物理上不要求一个请求占用连续显存
- 逻辑上仍然能把这些 block 视为连续 token 序列
- 请求增长时只需追加新的 block
- 回收时按 block 释放，碎片更可控

因此它解决的核心不是“attention 能不能算”，而是“在真实服务环境里，KV Cache 能不能高效分配、扩展和回收”。

## 2.8 PagedAttention 与传统连续内存管理的区别

| 维度 | 连续内存 KV Cache | PagedAttention |
| :--- | :--- | :--- |
| 分配方式 | 往往需要大块连续区域 | 按固定 block 分配 |
| 扩容方式 | 可能需要搬迁或预留过量空间 | 追加 block 即可 |
| 碎片表现 | 动态场景下容易恶化 | 通常更易控制 |
| 请求长度适配 | 对长度波动较敏感 | 更适合长度不确定场景 |
| 调度灵活性 | 较弱 | 更适合高并发动态调度 |
| 实现复杂度 | 相对简单 | 更复杂，需要映射表和 block 管理 |

PagedAttention 的代价也很明确：

- 需要维护逻辑块到物理块的映射
- 内核实现更复杂
- block 大小选择不合理时会带来内部浪费或访存效率问题

所以它不是“零成本优化”，而是为了服务化场景做出的工程权衡。

## 2.9 KV Cache 相关设计点

### 2.9.1 Block 大小如何选

Block 太小：

- 映射表更大
- 管理开销更高
- kernel 处理可能更复杂

Block 太大：

- 最后一个 block 容易留出较多空洞
- 小请求浪费更明显

因此 block 大小通常要在管理成本、访存效率和内部碎片之间折中。

### 2.9.2 数据类型如何选

KV Cache 不一定必须使用和权重相同的精度。  
降低缓存精度可以减少显存压力，但需要权衡：

- 精度损失是否会影响效果
- 是否有额外转换开销
- 硬件和 kernel 是否支持

### 2.9.3 是否支持 Prefix Cache

如果大量请求共享系统 prompt、模板前缀或公共上下文，那么 prefix cache 的价值很高。  
这能显著降低 prefill 计算和重复缓存写入。

### 2.9.4 是否支持 Swap / Offload

当显存不足时，有些系统会考虑把部分缓存换到 CPU 内存甚至更低层存储。  
这能提高可承接容量，但通常会明显增加延迟，因此更适合作为兜底机制，而不是常态路径。

### 2.9.5 是否支持抢占与恢复

在高优先级请求插队或资源紧张时，系统可能需要中断低优先级请求并回收其缓存。  
这要求缓存和调度设计天然支持恢复、迁移或重算。

## 2.10 常见问题与误区

### 2.10.1 “有了 KV Cache，attention 复杂度就完全不是问题了”

不对。  
KV Cache 主要减少了历史 `K/V` 的重复计算，但 decode 阶段依然要读取历史缓存并做 attention。  
上下文越长，访存和带宽压力仍然会增加。

### 2.10.2 “KV Cache 只是一个实现细节”

不对。  
在真实推理系统里，KV Cache 直接决定：

- 最大上下文能力
- 可承接并发数
- TTFT 和 TPOT 表现
- OOM 风险和服务稳定性

它属于推理系统的一等公民。

### 2.10.3 “PagedAttention 等于 prefix cache”

不对。  
PagedAttention 主要解决的是缓存组织和内存管理问题；  
prefix cache 主要解决的是公共前缀复用问题。  
两者经常一起出现，但关注点不同。

### 2.10.4 “只要显存够大，就不需要关心缓存管理”

不对。  
即使显存总量很大，碎片、带宽、调度冲突、回收不及时等问题仍然会限制系统表现。

## 2.11 实践里通常关注哪些指标

评估 KV Cache 设计时，通常不只看单点延迟，还会一起看：

- 最大可服务上下文长度
- 单实例最大并发请求数
- 每张卡可容纳的总 cached tokens
- TTFT、TPOT、P95 / P99 延迟
- 显存利用率与碎片率
- cache hit ratio（若使用 prefix cache）
- 抢占、回收、超时后的稳定性

如果一个方案在 benchmark 下速度很快，但高并发时缓存池频繁抖动、OOM 或尾延迟恶化，那么它在生产里通常并不算真正可用。

## 2.12 总结

KV Cache 解决的是“大模型生成时不要反复计算历史上下文”的问题，是推理系统最关键的基础设施之一。  
但它带来的代价同样明显：显存占用、动态扩容、碎片控制、带宽压力和调度耦合。

PagedAttention 的价值，就在于把 KV Cache 从“静态连续数组”升级成“适合动态服务场景的分页式资源”。  
它并没有改变模型本身的能力，而是显著改善了推理系统对长上下文和高并发请求的承载方式。

可以用一句话概括：

**KV Cache 决定大模型能否高效复用历史，PagedAttention 决定这种复用能否在真实服务系统里规模化落地。**
