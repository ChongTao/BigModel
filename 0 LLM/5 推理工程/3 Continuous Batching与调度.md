# 三 Continuous Batching 与调度

## 3.1 为什么静态 Batch 不适合在线推理

在训练场景里，固定 batch 是很自然的做法。  
样本通常可以预先切分、对齐和堆叠，系统只需要尽可能把 GPU 跑满。

但在线推理不是这样。真实请求有几个明显特点：

- 到达时间不一致，不会整齐地“一批一批”进入
- prompt 长度不同，prefill 开销差异很大
- 输出长度不可提前精确预测
- 用户对延迟敏感，不会愿意为了凑 batch 长时间排队
- 不同请求还可能有不同优先级、超时时间和服务等级

如果仍然采用静态 batch，通常会出现两类问题：

- **等人凑批**：为了把 batch 填满，新请求要先排队，TTFT 变差
- **尾部拖累**：batch 中某些请求已经快结束，某些请求还很长，整体资源利用变低

所以在线推理的核心目标，不是构造“完美固定批次”，而是让请求在动态进入、动态退出的情况下，持续维持高利用率。

## 3.2 Continuous Batching 的核心思想

Continuous Batching 可以理解为一种**滚动批处理**：

- batch 不是一开始就固定死
- 每一步调度后，都可以有请求离开 batch
- 也可以有新请求补进 batch
- 系统持续重组执行队列，而不是等整批完成再处理下一批

这和传统静态 batch 的关键区别在于，batch 成了一个动态集合，而不是一次性确定的封闭集合。

对 decoder-only 模型来说，最典型的场景是：

1. 某些请求已经完成生成，退出 batch
2. 某些请求继续 decode，下一个 token 还要算
3. 新到达的请求进入 prefill 或等待插入
4. 调度器根据资源情况重新拼装下一轮执行集合

这样做的直接好处是：

- GPU 空闲槽位能更快被新请求填补
- 长短请求不会像静态 batch 那样强绑定到一起
- 系统能在更高并发下保持稳定吞吐

## 3.3 在线推理中的调度问题

Continuous Batching 不是单纯“把请求拼起来”这么简单，它本质上是一个运行时调度问题。调度器通常至少要回答以下问题：

- 哪些请求先进入 prefill
- 哪些请求先参与下一轮 decode
- 新请求什么时候插入，是否打断当前节奏
- 显存和 KV Cache 不够时，哪些请求等待、抢占或回收
- 长请求和短请求如何共存，避免互相拖累
- 高优先级请求如何插队，又不至于把普通请求饿死

因此调度器决定的，不只是顺序问题，而是整套系统的：

- 吞吐
- TTFT
- TPOT
- P95 / P99 延迟
- 显存利用率
- 请求公平性

很多推理框架表面上都支持 continuous batching，但最终效果差距很大，原因通常就出在调度器实现细节上。

## 3.4 Prefill 与 Decode 的不同压力

调度设计里最重要的一点，是不要把 prefill 和 decode 看成完全相同的工作负载。

### 3.4.1 Prefill 更像大块计算

Prefill 阶段要一次处理整段 prompt。  
它的特点通常是：

- 算力消耗更集中
- 矩阵计算更大，更容易利用 GPU 并行
- 会一次性写入大量 KV Cache
- 直接影响首 token 延迟

如果 prefill 排队太久，用户会明显感觉“模型迟迟不开始说话”。

### 3.4.2 Decode 更像高频小步执行

Decode 阶段每次只生成一个 token。  
它的特点通常是：

- 单步计算较小，但执行频率高
- 依赖 KV Cache 读取
- 更容易受调度和访存影响
- 直接影响流式输出是否顺畅

如果 decode 被频繁打断，用户会感觉模型“卡顿”或输出抖动。

### 3.4.3 两者混调为什么困难

Prefill 和 decode 共用同一批 GPU 资源时，经常会互相挤压：

- prefill 太激进，会拉高 decode 延迟，影响正在输出的请求
- decode 优先级太高，新请求又可能迟迟进不了系统，TTFT 恶化

所以很多成熟系统不会简单地把两者混在一个队列里，而是显式区分：

- 分别排队
- 分别限流
- 分别设预算
- 在不同时间片间切换

## 3.5 常见调度策略

### 3.5.1 FIFO

最简单的策略是先进先出。  
优点是实现简单、行为直观；缺点是遇到超长请求时，后面的短请求可能一起受拖累。

### 3.5.2 Shortest Job First 倾向

有些系统会倾向优先处理短请求或短输出请求。  
这样通常能改善平均延迟和整体周转时间，但代价是长请求可能更容易饥饿。

### 3.5.3 优先级队列

企业场景常见做法是按租户、接口类型或 SLA 设置优先级。  
高优先级请求可以更快进入执行队列，但需要额外机制避免低优先级请求长期得不到服务。

### 3.5.4 Prefill / Decode 分离调度

这是在线推理里很常见的一类设计。  
系统会分别管理 prefill 和 decode 队列，再按预算或时间片决定谁先使用资源。

这类方案通常比统一队列更稳定，因为它承认两类负载的本质差异。

### 3.5.5 Chunked Prefill

当 prompt 很长时，可以不一次吃完整段 prefill，而是拆成多个 chunk。  
这样做的目标是避免一个超长 prompt 长时间独占 GPU，给 decode 留出插空机会。

### 3.5.6 抢占与恢复

在资源紧张或高优先级请求插队时，系统可能中断一部分低优先级请求，稍后恢复。  
这要求调度器与 KV Cache 管理天然配合，否则抢占的成本会太高。

## 3.6 吞吐、延迟与公平性的权衡

调度从来不是只优化一个指标。  
大多数时候，系统是在三件事之间找平衡：

- **吞吐**：单位时间处理更多 token 或更多请求
- **延迟**：让请求更快开始、更快结束
- **公平性**：不同请求不要长期被挤压或饿死

典型冲突包括：

- batch 更大，吞吐通常更好，但 TTFT 可能更差
- 更偏向短请求，平均延迟更好，但长请求体验更差
- 高频插入新请求，系统响应更灵活，但 decode 稳定性可能下降
- 强优先级策略对关键请求友好，但会破坏整体公平性

因此工程上通常不会追求单一“最优调度器”，而是根据业务目标设策略：

- 面向聊天产品，更重视 TTFT 和流式稳定性
- 面向离线批量生成，更重视吞吐和成本
- 面向多租户平台，更重视隔离、公平和尾延迟

## 3.7 Continuous Batching 与 KV Cache 的关系

Continuous Batching 能成立，一个前提是请求可以在动态进出 batch 的同时保留自己的历史状态。  
这个历史状态，核心就是 KV Cache。

如果没有 KV Cache：

- 请求每次重新进入执行集合时，都要重算历史上下文
- 动态调度的收益会大幅下降
- 多请求滚动调度的成本会变得难以接受

所以从系统视角看：

- `KV Cache` 负责保存每个请求的历史
- `Continuous Batching` 负责让这些请求动态共享执行机会
- `PagedAttention` 等机制负责让缓存分配和回收更可持续

这三者通常是配套出现的，而不是孤立优化点。

## 3.8 常见工程问题

### 3.8.1 Decode 被 Prefill 压住

如果新请求持续涌入，而系统又过度偏向 prefill，那么已经开始输出的请求可能被频繁延后，造成 token 流不稳定。

### 3.8.2 长请求拖垮队列

当系统没有做长度感知调度时，少数超长请求可能长期占用缓存和执行槽位，导致整体吞吐下降。

### 3.8.3 只看平均值，忽略尾延迟

有些调度策略 benchmark 看起来很快，但 P95 / P99 很差。  
这通常意味着系统把一部分请求“牺牲掉了”，只是平均值掩盖了问题。

### 3.8.4 批次很满，但 GPU 利用率并不高

batch 大不等于效率高。  
如果 batch 内请求状态差异太大、访存受限、调度切换频繁，GPU 仍然可能吃不满。

### 3.8.5 调度器和缓存池脱节

如果调度器只按“谁该先跑”决策，而不感知 KV Cache 容量、碎片和回收状态，就容易出现队列能排、资源却落不下的情况。

## 3.9 系统架构中的位置

在一个完整推理系统里，调度器通常位于请求接入层和执行引擎之间，扮演“资源编排器”的角色。

它通常要同时连接几类模块：

- 请求入口：接收新请求、取消、超时信号
- 队列管理：维护等待队列、运行队列、优先级队列
- KV Cache 管理器：查询容量、分配 block、回收状态
- 执行引擎：提交 prefill / decode 执行批次
- 观测系统：记录 TTFT、TPOT、队列等待时间、抢占次数等指标

因此调度器不是一个小附加模块，而是推理系统的中枢之一。  
很多线上稳定性问题，最终都能追溯到调度策略和资源管理之间的耦合不当。

## 3.10 实践里通常关注哪些指标

评估 continuous batching 和调度策略时，常见指标包括：

- TTFT
- TPOT 或 tokens/s
- 每秒完成请求数
- 运行中 batch 大小分布
- prefill 队列等待时间
- decode 队列等待时间
- P95 / P99 延迟
- 抢占次数与恢复成本
- GPU 利用率
- KV Cache 利用率

如果一个方案只在低并发下看起来顺滑，而一上并发就出现排队暴涨、尾延迟恶化或 decode 抖动，那么通常是调度策略没有真正匹配线上负载。

## 3.11 总结

Continuous Batching 解决的核心问题，不是“如何把请求硬凑成一批”，而是“如何让动态到达、动态完成的请求持续共享 GPU 资源”。  
它把在线推理从静态批处理问题，转成了一个持续运行的调度问题。

真正决定系统表现的，往往不是是否支持 continuous batching，而是：

- 是否区分 prefill 和 decode
- 是否理解 KV Cache 的容量约束
- 是否能在吞吐、延迟和公平性之间做稳定权衡

可以用一句话概括：

**KV Cache 让请求记住历史，Continuous Batching 让请求动态拼批，调度器则决定谁能在什么时刻以什么代价获得算力。**
