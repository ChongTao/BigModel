# 九 推理性能指标与 Benchmark

## 9.1 为什么推理评测不能只看 tokens/s

很多人在看推理性能时，第一反应是比较 `tokens/s`。  
这个指标当然重要，但如果只看它，往往会得出错误结论。

原因很简单：在线推理不是单一吞吐问题，而是多目标系统问题。  
同一个方案可能：

- `tokens/s` 很高，但首 token 很慢
- 平均速度很快，但尾延迟很差
- 单卡 benchmark 很漂亮，但多并发时抖动严重
- 离线批量生成表现很好，但聊天场景体验很差

所以评测推理系统时，不能只问“每秒生成多少 token”，还要问：

- 用户多久能看到第一个 token
- 生成过程是否稳定
- 高并发下会不会排队
- 长上下文下是否还能维持性能
- 成本是否可接受

换句话说，`tokens/s` 更像一个局部指标，而不是完整答案。

## 9.2 常见性能指标

### 9.2.1 TTFT

`TTFT`，即 Time To First Token，首 token 延迟。  
它衡量的是用户发起请求后，多久能看到第一个输出。

这个指标通常受以下因素影响最大：

- 请求排队时间
- prompt 长度
- prefill 计算开销
- 调度器是否优先照顾 decode 中请求

对聊天产品来说，TTFT 往往比总生成速度更敏感，因为它直接影响“模型是否立刻开始响应”。

### 9.2.2 TPOT

`TPOT`，即 Time Per Output Token。  
它衡量生成阶段每个 token 的平均耗时，也可反过来看成 decode 速度。

这个指标主要反映：

- decode 调度是否顺畅
- KV Cache 读取是否高效
- batch 动态重组是否稳定
- 流式输出是否连续

如果 TTFT 不错，但 TPOT 很差，用户通常会感觉模型“开口快，但说话卡”。

### 9.2.3 Throughput

吞吐通常表示单位时间内系统能处理多少 token 或多少请求。  
常见口径包括：

- output tokens/s
- total tokens/s
- requests/s
- completed requests/min

不同口径不能混着比较。  
例如只统计输出 token 和同时统计输入加输出 token，结论可能差很多。

### 9.2.4 延迟分位数

除了平均延迟，还需要看：

- P50
- P90
- P95
- P99

因为线上问题往往不是平均值变差，而是尾部请求明显恶化。  
如果只看平均值，很容易把小部分非常差的请求掩盖掉。

### 9.2.5 GPU 利用率

GPU 利用率衡量资源是否真的被用起来。  
但它也不能单独解释一切：

- 利用率低，不一定说明调度差，可能是负载不足
- 利用率高，也不一定说明体验好，可能是排队严重换来的

所以 GPU 利用率更适合和 TTFT、TPOT、吞吐一起看。

### 9.2.6 KV Cache 相关指标

在大模型推理里，缓存相关指标非常关键，例如：

- cache pool usage
- cached tokens
- prefix hit ratio
- block fragmentation
- OOM / eviction / preemption 次数

很多系统性能退化，本质并不是算力不足，而是 KV Cache 管理失效。

## 9.3 离线压测与在线观测的区别

很多 benchmark 成绩只说明“在实验环境里能跑多快”，并不等于“在线服务一定好用”。

### 9.3.1 离线压测关注上限

离线压测通常更容易控制变量：

- 固定模型
- 固定输入长度
- 固定输出长度
- 固定并发数
- 固定硬件环境

它适合回答的问题是：

- 单卡理论吞吐大致有多高
- 某个优化是否确实带来提速
- 两个框架在相同条件下差异多大

### 9.3.2 在线观测关注稳定性

线上系统面对的是动态流量：

- 请求到达不均匀
- prompt 长度分布变化大
- 输出长度不可预测
- 多租户和优先级共存
- 超时、取消、重试频繁发生

所以在线观测更关注：

- 排队是否恶化
- 尾延迟是否抖动
- 高峰期是否触发抢占或 OOM
- 指标是否随着流量变化而失稳

离线 benchmark 更像实验室成绩，在线观测才更接近真实战斗力。

## 9.4 Benchmark 怎么设计才靠谱

一个可靠 benchmark 的关键，不是把数字做大，而是把条件说明白。

### 9.4.1 明确测试口径

至少要写清楚：

- 模型名称和参数规模
- 量化方式
- 硬件型号与卡数
- 输入长度分布
- 输出长度分布
- 并发数或请求到达方式
- 是否流式输出
- 统计的是 output tokens 还是 total tokens

如果这些条件不透明，结果几乎没有可比性。

### 9.4.2 控制变量

Benchmark 里最常见的问题是一次改了太多东西。  
例如换了框架、换了量化、换了 batch 策略、换了提示词长度，最后很难知道性能变化到底来自哪里。

更稳妥的做法是一次只改一个关键变量，然后重复测试。

### 9.4.3 不只测单一长度

如果只测一种 prompt 长度，结果很容易失真。  
更合理的方式是覆盖几类典型场景：

- 短输入短输出
- 短输入长输出
- 长输入短输出
- 长输入长输出

因为 prefill 和 decode 的压力分布，在不同组合下完全不同。

### 9.4.4 区分暖机前后

首次请求和稳定运行后的表现可能差很多。  
例如模型加载、图编译、缓存预热都会影响前几次结果。

所以 benchmark 应明确：

- 是否已经 warmup
- warmup 做了几轮
- 统计结果是否排除首轮异常值

### 9.4.5 看趋势，不只看单次最好成绩

单次峰值成绩不可靠。  
更有价值的是看：

- 多轮结果是否稳定
- 方差是否过大
- 并发上升后曲线怎么变化
- 某个阈值后系统是否突然退化

很多系统不是慢，而是“不稳定”。

## 9.5 不同场景下关注的核心指标

### 9.5.1 聊天助手

聊天类产品通常更重视：

- TTFT
- TPOT
- P95 / P99 延迟
- 流式输出稳定性

这类场景里，用户感知往往优先于理论吞吐。

### 9.5.2 离线批量生成

离线场景通常更重视：

- 总吞吐
- 单位 token 成本
- GPU 利用率
- 长时间运行稳定性

因为这类任务不强调即时交互，更关心规模化产出。

### 9.5.3 企业多租户平台

平台场景通常同时关注：

- 吞吐
- 尾延迟
- 公平性
- 租户隔离
- 限流和抢占效果

这类系统的难点不只是快，而是“在复杂流量结构下还能稳”。

## 9.6 常见误判

### 9.6.1 “平均 tokens/s 高，所以用户体验一定好”

不对。  
如果 TTFT 很差，或者 TPOT 抖动明显，用户体验依然会很差。

### 9.6.2 “单卡快，多卡一定更快”

不对。  
多卡会引入通信、同步、调度和分布式缓存管理开销。扩展后未必线性提升。

### 9.6.3 “离线压测结果可以直接代表线上效果”

不对。  
线上流量的长度分布、到达模式和优先级冲突，往往足以改写 benchmark 结论。

### 9.6.4 “GPU 利用率越高越好”

不对。  
如果利用率是靠更长排队、更高尾延迟换来的，那只是把用户体验变差了。

### 9.6.5 “同样是 100 tokens/s，两个结果就可直接比较”

不对。  
必须先确认：

- 是 output 还是 total tokens
- prompt 长度是否一致
- 并发是否一致
- 测试时间窗口是否一致
- 是否包含 warmup 或排队时间

没有统一口径的数字，比较意义很弱。

## 9.7 一套简单评测框架

如果只是做工程决策，而不是写论文，可以先用一套简化框架评估：

1. 固定模型、硬件和量化方式
2. 准备 3 到 4 组典型输入输出长度组合
3. 分别测试低并发、中并发、高并发
4. 记录 TTFT、TPOT、吞吐、P95 / P99
5. 同时记录 GPU 利用率和 KV Cache 使用情况
6. 重复多轮，观察是否稳定

如果要进一步贴近线上，可以再补：

- 混合长度请求
- 突发流量
- 请求取消
- 高优先级插队
- 长时间 soak test

这样得到的结果，通常比单一 `tokens/s` 更能支持实际选型。

## 9.8 总结

推理评测的核心，不是找一个“最大吞吐数字”，而是理解系统在不同负载下的真实行为。  
一个可用的 benchmark，应该同时回答性能、稳定性、成本和体验问题。

因此在看推理系统时，至少要同时关注四件事：

- 首 token 快不快
- 生成过程稳不稳
- 高并发下会不会退化
- 这些结果的成本高不高

可以用一句话概括：

**Benchmark 的价值，不是证明系统在某个点上最快，而是暴露系统在真实负载下会怎样变快、变慢或失稳。**
