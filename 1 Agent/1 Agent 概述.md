# 一 Agent介绍

Agent 是一种能够围绕目标自主感知环境、做出决策、执行动作并根据反馈调整策略的智能体。

和普通的“问一句答一句”的 LLM 应用相比，Agent 更强调三点：

- **目标驱动**：围绕目标持续推进，而不只是单轮生成文本
- **可执行**：能够调用工具、访问外部系统、操作环境
- **可迭代**：会根据中间结果、失败信息、环境反馈继续调整下一步动作

## 1.1 Agent 定义的演进

Agent 的定义在不同阶段有不同重心，可以粗略理解为三个阶段：

### 1.1.1 传统 AI 阶段

在传统 AI / 强化学习语境里，Agent 通常指：

- 感知环境（Perception）
- 基于状态做决策（Decision Making）
- 执行动作（Action）
- 接收环境反馈（Feedback）

这个阶段的重点是“**自主体**”和“**环境交互**”。

### 1.1.2 LLM Agent 阶段

大模型兴起后，Agent 的定义开始更多落到 LLM 应用工程中。此时常见表述是：

`Agent = LLM + Planning + Memory + Tools`

核心意思是：让 LLM 不仅负责生成文本，还负责拆解任务、调用工具、维护上下文，并在多轮交互中完成任务。

这一阶段的重点是“**让 LLM 从聊天模型变成可执行系统**”。

### 1.1.3 工程化视角：LLM + Harness

近两年的工程实践里，一个更偏工程化的理解越来越常见：

`Agent ≈ LLM（或 Model） + Harness`

这里的 **Harness** 不是单一模块，而是围绕模型运行的一整套执行系统，通常包括：

- 上下文组装与状态管理
- 工具注册、选择、调用与结果回填
- 记忆读写
- 规划与执行循环
- 权限控制、失败重试、终止条件
- 观测、评测、日志与安全护栏

也就是说，**单独的 LLM 并不能直接构成 Agent，真正让它“能干活”的，是外层的运行时与控制系统**。  
从这个角度看，前面提到的 Planning、Memory、Tools，本质上都可以视为 Harness 的组成部分。

因此，`LLM + Planning + Memory + Tools` 更像是对 Agent 能力构成的描述，`LLM + Harness` 更像是对 Agent 工程实现方式的总结。

## 1.2 Agent 的最小运行闭环

理解 Agent，最重要的不是单个模块，而是它的运行闭环。一个最小可用的 Agent，通常至少包括以下流程：

1. **接收目标**
   用户给出任务、约束和期望结果。
2. **读取状态**
   获取当前上下文、历史信息、环境状态或已有中间结果。
3. **生成下一步决策**
   判断是直接回答、拆分子任务，还是调用某个工具。
4. **执行动作**
   调用检索、代码、浏览器、数据库、API 等外部能力。
5. **观察结果**
   读取工具输出、错误信息、环境变化。
6. **更新记忆或状态**
   保存关键中间结果，避免重复劳动，也为后续步骤提供上下文。
7. **继续循环或终止**
   如果目标达成则停止，否则继续下一轮决策。

可以把它简化为：

`目标 -> 计划 -> 行动 -> 观察 -> 更新状态 -> 再决策`

这也是 Agent 和普通 LLM App 的关键差异：  
普通 LLM App 往往是“一次输入，一次输出”；而 Agent 是“围绕目标持续执行直到结束”。

## 1.3 Agent 的核心组成

- **LLM / Model**：负责理解目标、推理、生成决策
- **规划（Planning）**：将复杂目标拆成可执行步骤
- **记忆（Memory）**：保存上下文、经验和历史状态
- **工具（Tools）**：扩展模型能力边界，让它能读取外部信息和执行动作
- **行动（Action）**：真正对外部世界产生影响
- **反馈（Observation / Feedback）**：从环境中读取结果，并决定下一步
- **Harness / Runtime**：把上述能力串起来，控制整个执行循环

### 1.3.1 LLM / Model

LLM 是 Agent 的推理核心，负责理解用户目标、生成计划、判断是否调用工具，以及根据反馈决定下一步动作。  
如果没有外层 Harness，LLM 更像一个会对话的模型；接入 Harness 后，才会变成一个能持续执行任务的系统。

### 1.3.2 规划

规划是指让智能体根据目标自动制定行动计划，并按步骤调用工具、检索信息或生成内容以完成任务。

需要注意的是，**规划**、**推理**、**反馈**不是一回事：

- **规划**：做任务拆解，决定先做什么、后做什么
- **推理**：判断当前最合理的下一步是什么
- **反馈**：根据执行结果修正后续动作

![](https://miro.medium.com/v2/resize:fit:2000/format:webp/1*A8lNI_zfrb3yUppdxW56LA.png)

常见规划方式可以粗分为两类：

- **无反馈规划**：一次性生成完整计划并执行，适合流程固定、工具确定性高的任务
- **有反馈规划**：执行过程中持续根据结果调整策略，典型方式如 ReAct，适合环境不确定、多工具协作的任务

### 1.3.3 记忆

记忆用于记录历史交互、任务状态或外部信息，使智能体能够在多轮交互或复杂任务中保持上下文连续性和决策一致性。

最粗粒度上，记忆通常分为两类：

- **短期记忆**：与当前任务直接相关的即时上下文，通常受上下文窗口限制
- **长期记忆**：跨任务保留的历史经验、事实知识、偏好信息

在更细的划分里，还可以继续细分为工作记忆、情节记忆、语义记忆、工具化记忆等。详细内容见：[1 Agent/1 记忆/1 记忆概述.md](./1%20记忆/1%20记忆概述.md)

### 1.3.4 工具

工具是智能体调用的外部能力模块或 API，使其能够执行特定操作、获取数据或增强任务能力。

常见工具类型包括：

- **检索类工具**：向量检索、全文搜索、知识库查询、网络搜索
- **执行类工具**：代码执行、Shell、SQL、浏览器自动化
- **系统类工具**：文件读写、消息发送、任务调度
- **外部服务类工具**：地图、支付、工单、业务 API

一个完整的工具调用过程通常包括：选择合适工具、构造参数、执行工具、读取结果，并判断是否需要重试或切换工具。

### 1.3.5 行动与反馈

行动是 Agent 真正对外部世界产生影响的步骤，例如发起检索、执行代码、修改文件、调用业务 API。  
反馈则是 Agent 从环境中读取回来的结果，例如工具输出、错误信息、页面变化或系统状态更新。

很多 Agent 系统是否有效，关键不在于“会不会生成计划”，而在于是否能基于反馈持续修正行动。

### 1.3.6 Harness / Runtime

Harness / Runtime 是把模型、规划、记忆、工具、行动和反馈串起来的外层执行系统。  
它通常负责上下文管理、状态维护、工具注册与调度、权限控制、失败重试、终止条件、日志与观测等能力。

## 1.4 什么时候适合用 Agent

适合使用 Agent 的典型场景：

- 任务目标明确，但完成路径不固定
- 需要多步推理和多工具协作
- 需要根据环境反馈不断调整策略
- 任务执行周期较长，需要保存状态和中间结果

不太适合直接使用 Agent 的场景：

- 流程固定且规则明确
- 输入输出关系稳定，几乎不需要试错
- 对成本、延迟、稳定性要求极高
- 更适合用传统工作流直接编排

这类场景中，直接使用 Workflow 往往更简单、更稳定、更可控。

## 1.5 Agent 和 Workflow 的区别

- **Workflow** 更强调预先定义好的固定流程
- **Agent** 更强调围绕目标进行自主决策

![](https://mmbiz.qpic.cn/sz_mmbiz_png/j3gficicyOvauNZyziaHia56WajsqXbwHetmLJB9ibxhH3sv1v13JicVPwldlCbJEfBdHsIA5NhH9iauZsgeJmGFzJzPQ/640?wx_fmt=png&from=appmsg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=1)

但在真实工程中，两者往往不是替代关系，而是组合关系：

- 主流程用 Workflow 固定下来
- 某些不确定节点交给 Agent 自主决策

所以很多实际系统并不是“纯 Agent”或“纯 Workflow”，而是 **Workflow + Agent** 的混合架构。

## 1.6 常见 Agent 架构模式

常见模式包括：

1. **单 Agent**
   一个 Agent 负责理解任务、调用工具并完成执行。适合较简单任务。
2. **Planner-Executor**
   一个模块负责规划，一个模块负责执行。适合复杂任务分解。
3. **Router Agent**
   负责把请求路由到不同子 Agent 或工具链。
4. **Multi-Agent**
   多个 Agent 分工协作，例如研究、编码、评审、执行分别由不同 Agent 完成。

## 1.7 常见问题

Agent 的能力强，但工程难度通常也更高。常见问题包括：

- **长链路漂移**：执行轮数变多后逐渐偏离原目标
- **工具误调用**：选错工具、参数错误、误解工具返回
- **记忆污染**：把错误中间结果写入记忆，影响后续决策
- **终止条件不清**：任务已经完成却继续调用工具，造成成本浪费
- **成本和延迟失控**：多轮思考、多次调用工具导致整体耗时和费用上升
- **可观测性差**：出了问题却不知道是计划错、工具错，还是上下文错

因此，Agent 的核心挑战通常不在“让模型会说话”，而在“让系统稳定地做事”。
