# 一 为什么需要“记忆”

大模型本身是一个**条件概率模型**，它不具备持续状态，每次推理都是“**从零开始**”。如果没有记忆，通常会出现以下问题：

- 上下文丢失，对话断裂
- 无法个性化，用户体验差
- 无法持续执行长任务，需要重复说明背景
- 不能积累经验，上一轮做对或做错都很难传递到下一轮

记忆就是让大模型能够跨越单次输入上下文，持续记住和利用信息，而不仅仅依赖有限的上下文窗口。记忆是智能体实现长期规划、持续学习和个性化交互的基础能力。

> CoT / ReAct 解决了 Agent 的“行动闭环”，而 Memory 解决了 Agent 的“经验积累”和“状态延续”。

## 1.1 记忆在 Agent 架构中的位置

在 Agent 系统里，记忆不是孤立模块，而是 **Harness / Runtime** 的一部分。  
它通常与上下文管理、工具调用、规划执行、状态维护一起工作。

记忆主要服务于四类目标：

- **状态延续**：让 Agent 知道“当前任务做到哪一步了”
- **个性化**：让 Agent 记住用户偏好、角色设定、长期习惯
- **经验积累**：让 Agent 能利用过去做过的事情优化当前决策
- **长期任务支持**：让 Agent 可以跨会话、跨阶段完成复杂任务

从运行链路上看，记忆通常发生在两个位置：

1. **读记忆**
   在生成下一步决策前，先检索与当前任务相关的历史信息。
2. **写记忆**
   在任务执行后，把有价值的新信息写回外部记忆系统。

因此，记忆不是单纯“存历史聊天记录”，而是 Agent 执行闭环的一部分。

# 二 记忆的常见分类

记忆通常有两种常见划分方式：

- **按时间尺度划分**：短期记忆、长期记忆
- **按内容类型划分**：工作记忆、情节记忆、语义记忆、程序性记忆

这两套分类不是互斥关系，而是从不同角度描述同一套记忆系统。

## 2.1 短期记忆（Short-term Memory）

- **定义**：与当前会话、当前任务直接相关的即时上下文
- **典型内容**：最近几轮对话、当前任务状态、最近一次工具结果、临时约束条件
- **实现方式**：
  - 会话上下文（conversation history）
  - scratchpad / reasoning buffer
  - 当前任务状态缓存
- **特点**：读取快、与当前任务强相关，但生命周期短
- **缺点**：随着对话变长，旧信息会被截断或压缩，不适合保存长期知识

![](https://www.dailydoseofds.com/content/images/2025/04/image-17.png)

> 长上下文、RoPE 插值、ALiBi、滑动窗口注意力等技术，可以增强短期上下文承载能力，但它们更偏向模型和上下文工程能力，不等于完整的 Agent Memory 方案。

## 2.2 长期记忆（Long-term Memory）

- **定义**：跨会话、跨任务保留的持久记忆
- **典型内容**：
  - 用户偏好
  - 历史任务结果
  - 长期事实知识
  - 结构化 profile / summary / state
- **存储方式**：通常依赖外部存储，如数据库、向量库、KV、知识图谱
- **检索方式**：根据 query、任务状态、用户身份、时间范围或结构化字段进行召回

长期记忆不等于“只有向量检索”。常见实现包括：

- 基于 embedding 的语义检索
- 基于关键词 / 标签 / 时间 / 用户 ID 的过滤检索
- 结构化 profile / summary 存储
- 知识图谱或实体关系存储

![](https://www.dailydoseofds.com/content/images/2025/04/image-19.png)

## 2.3 工作记忆（Working Memory）

- **定义**：任务执行过程中的临时工作区，介于短期上下文与长期记忆之间
- **作用**：保存中间步骤结果，供后续推理和工具调用使用
- **常见内容**：
  - 当前 plan
  - 子任务状态
  - 工具返回的中间结果
  - 待确认的信息
- **实现方式**：
  - 会话缓存
  - scratchpad
  - 临时状态对象
  - 临时图节点 / 临时表

工作记忆更强调“**正在做的事**”，而不是“过去发生过什么”。

## 2.4 情节记忆（Episodic Memory）

- **定义**：按事件或会话组织的交互历史
- **作用**：帮助 Agent 回忆“之前发生了什么”
- **典型内容**：
  - 某次会话中的关键事件
  - 某次任务执行过程和结果
  - 用户与 Agent 的历史交互轨迹
- **实现方式**：事件日志 + 时间戳 + 向量表示 + 检索索引

情节记忆更像“经历过的事情”。

## 2.5 语义记忆（Semantic Memory）

- **定义**：把历史信息抽象总结为可复用的知识
- **作用**：让 Agent 不只是记住“某次对话”，而是记住“稳定事实”
- **例子**：
  - 用户更喜欢用 Go 写后端
  - 该项目默认使用 MySQL
  - 某客户的时区是爱尔兰
- **实现方式**：
  - 对话总结
  - profile 抽取
  - 事实槽位更新
  - 知识库写入

语义记忆更像“总结后的知识”。

## 2.6 程序性记忆（Procedural Memory）

- **定义**：关于“怎么做事”的记忆，也可以理解为技能、策略、操作套路
- **典型内容**：
  - 某类任务的标准处理流程
  - 特定用户偏好的工作方式
  - 一套稳定可复用的操作规则
- **例子**：
  - 处理代码评审时先找 bug，再给总结
  - 修改文档时不要丢失已有图片引用
  - 发布前必须先跑敏感信息扫描

程序性记忆更接近“行为规则”和“工作习惯”，不只是事实存储。

## 2.7 外部记忆工具 / Memory Store

外部记忆工具不是一种“记忆类型”，而是记忆的**承载介质**或**访问接口**。

常见形式包括：

- 向量数据库
- 关系型数据库 / KV 存储
- 知识图谱
- 个人笔记系统（Notion、Obsidian）
- CRM、工单、日历、待办事项系统

这些系统本身不是记忆的定义，但常被 Agent 作为记忆存储和读取的后端。

# 三 记忆生命周期

一个完整的 Agent Memory 系统，通常不只是“存一下、查一下”，而是包含完整的生命周期：

## 3.1 写入什么

不是所有信息都值得记忆。通常更适合写入的内容包括：

- 稳定的用户偏好
- 可复用的任务结果
- 长期有效的事实知识
- 后续步骤仍需要的任务状态
- 可以提升未来决策质量的经验信息

不适合直接写入的内容包括：

- 纯噪声对话
- 临时且马上失效的信息
- 未经确认、可能错误的推断

## 3.2 存到哪里

不同类型的信息适合不同存储：

- 对话片段、语义相似内容：向量库
- 用户 profile、事实槽位：结构化数据库
- 任务状态：KV / session store
- 实体关系：知识图谱

很多系统会混合使用多种存储，而不是只靠一个向量库。

## 3.3 什么时候检索

常见触发方式包括：

- 用户发起新请求时
- Agent 进入下一步规划前
- 调用工具前需要补充背景时
- 多轮任务恢复执行时

检索条件也不只是语义相似度，还可能结合：

- 用户 ID
- 会话 ID
- 时间范围
- 任务类型
- 标签 / 实体 / 关键词

## 3.4 如何更新

新的记忆和旧的记忆可能冲突，因此需要更新策略，例如：

- 覆盖旧值
- 合并摘要
- 保留版本
- 标记置信度

例如用户之前说“我主要写 Java”，后来持续多次表达“现在主要写 Go”，系统就应该更新而不是盲目累积。

## 3.5 如何遗忘

记忆越多并不一定越好。长期不清理会导致噪声增加、检索污染、成本上升。

常见做法包括：

- 定期摘要压缩
- 低价值信息淘汰
- 按时间归档
- 基于访问频次衰减
- 敏感信息按策略删除

# 四 Agent Memory 与 RAG 的区别

Agent Memory 和 RAG 经常一起出现，但它们解决的问题并不完全一样：

- **RAG** 主要解决“从外部知识源中召回与当前问题相关的信息”
- **Memory** 主要解决“让 Agent 记住用户、任务、状态和历史经验”

两者可能共享同样的技术手段，例如：

- embedding
- 向量检索
- rerank
- 外部数据库

但目标不同：

- RAG 更偏“外部知识补充”
- Memory 更偏“历史状态延续与个性化”

可以简单理解为：

- FAQ、知识库、文档检索，更像 RAG
- 用户偏好、任务历史、个人 profile，更像 Memory

# 五 常见问题

## 5.1 记忆污染

如果把错误信息、推测性结论、低质量摘要写入记忆，后续检索时就会不断放大错误。

## 5.2 错误记忆固化

一旦错误信息被系统当成“长期事实”保存，就可能在多个任务里持续影响决策。

## 5.3 检索噪声过大

长期记忆越多，检索出来的内容不一定越有用，反而可能把无关信息带进 prompt。

## 5.4 用户隐私与权限问题

用户画像、偏好、行为记录本身就可能包含敏感信息，需要权限边界、保留策略和删除能力。

## 5.5 记忆与工具边界混乱

有些系统把所有外部数据源都当作“记忆”，会导致架构边界模糊。更合理的做法是区分：

- 什么是长期保留的 Agent Memory
- 什么只是运行时查询的外部系统
