# 一 为什么要单独比较

单看每个记忆方案的介绍，很容易觉得它们都在做“长期记忆”。  
但真正落到工程实现时，它们的差异并不在“能不能存”，而在：

- **围绕什么对象组织记忆**
- **记忆最终以什么形态存储**
- **检索出来的是原始历史，还是整理后的记忆对象**
- **更新逻辑是追加、CRUD，还是上下文调度**

所以更合理的方式是：

- 单篇文档只讲该方案本身
- 横向比较集中在一篇文档里统一看

# 二 一张表看差异

| 方案 | 核心问题 | 记忆主单位 | 存储组织方式 | 检索/使用方式 | 更适合什么 |
| --- | --- | --- | --- | --- | --- |
| **Memobase** | 如何持续认识某个用户 | `user`、`profile`、`event` | 以用户为中心组织；原始输入经 `blob -> buffer -> profile/event` 沉淀 | 按用户历史和当前聊天生成 `context`，返回用户画像和相关事件 | 个性化助手、用户运营、CRM、陪伴类产品 |
| **Mem0** | 如何把交互变成可维护的通用记忆层 | `memory item` | 以记忆对象为中心；强调新增和显式治理；常配合混合存储 | 根据 query / user / context 检索相关 memory item 回填 prompt | 通用 Agent memory layer、多类型长期记忆管理 |
| **MemGPT / Letta** | 在有限上下文下如何持续工作 | `memory block`、`archival memory`、`agent state` | 以上下文层级和 agent state 组织；核心是主上下文与外部记忆的切换 | 运行时决定何时检索、迁出、压缩、换入 | 长任务、长上下文、多阶段复杂执行 |
| **Graphiti** | 如何在动态环境里维护带时间语义的图记忆 | `episode`、`entity`、`relation` | 以时序上下文图组织，增量更新节点、边和时间变化 | 结合语义、关键词、图搜索与 rerank 取回图上下文 | 动态世界模型、关系网络、时序事实追踪 |
| **LangMem** | 如何给现有 Agent 轻量接入长期记忆能力 | `memory tool call`、`store entry` | 以内存工具和 background manager 组织，底层依赖 LangGraph store 或自定义 store | 通过 manage/search tools 和后台 memory manager 读写记忆 | LangGraph 生态、轻量长期记忆接入、可组合 memory workflows |
| **Neo4j Agent Memory** | 如何把对话、知识图谱和推理痕迹统一成图记忆 | `conversation`、`graph entity`、`reasoning trace` | 以 Neo4j 图模型组织 conversation、knowledge graph、reasoning memory | 基于图原生查询和图上下文回填给 agent | 强关系推理、企业图谱、图原生长期记忆服务 |
| **通用向量库 / RAG** | 如何召回和当前问题相关的信息 | `document chunk` | 以文档切片或片段向量存储为主 | 相似度检索 + rerank 后回填上下文 | FAQ、知识库问答、文档检索 |

先给一个最短结论：

- **Memobase** 更像“用户档案系统”
- **Mem0** 更像“通用记忆管理层”
- **MemGPT / Letta** 更像“记忆感知的运行时”
- **Graphiti** 更像“时序图记忆引擎”
- **LangMem** 更像“记忆工具箱”
- **Neo4j Agent Memory** 更像“图原生 memory service”
- **向量库 / RAG** 更像“知识召回后端”

# 三 差异到底在哪里

## 3.1 围绕什么对象组织记忆

这是几种方案最本质的分水岭。

### Memobase：围绕用户组织

Memobase 的主语不是“这段对话”或“这条 memory”，而是“**这个用户是谁**”。  
它会持续维护这个用户的长期档案，包括：

- profile
- 事件历史
- 日程
- 社交关系

所以它天然适合“一个用户会持续回来，系统要越来越了解他”的场景。

### Mem0：围绕记忆对象组织

Mem0 的主语更像“**哪些内容值得变成长期 memory**”。  
它关心的是把交互提炼成一个个可维护的 memory item，然后做：

- 新增
- 更新
- 删除
- 合并
- 检索

所以它更像一个通用 memory orchestration layer，而不是专门的用户画像系统。

### MemGPT / Letta：围绕上下文窗口组织

MemGPT / Letta 的主语不是用户，也不是单条 memory，而是“**当前上下文里该保留什么**”。  
它重点解决的是：

- 哪些内容应该留在主上下文
- 哪些内容应该迁出
- 什么时候再检索回来

因此它本质上更偏 **runtime / context scheduler**。

### Graphiti：围绕时序图组织

Graphiti 的主语更像“**世界里有哪些实体、关系和时间变化**”。  
它强调的不是 memory item CRUD，而是：

- episode 写入
- 图增量更新
- 时间语义
- 图检索与上下文组装

因此它更偏 **temporal graph memory**。

### LangMem：围绕 memory workflow 组织

LangMem 的主语更像“**如何给现有 agent 配 memory 工具和后台整理流程**”。  
它不是强绑定某一种 memory schema，而是提供：

- hot-path memory tools
- background memory manager
- 与 store 的集成能力

因此它更偏 **memory primitives / workflow layer**。

### Neo4j Agent Memory：围绕图原生记忆服务组织

Neo4j Agent Memory 的主语更像“**如何把对话、图谱、推理痕迹都放进图模型**”。  
它强调：

- conversation memory
- knowledge graph memory
- reasoning memory

因此它更偏 **graph-native memory service**。

### 通用向量库 / RAG：围绕文档片段组织

RAG 类方案一般并不主动维护“用户状态”或“持续演化的长期记忆”，而是把外部知识切成片段后索引起来。  
它最擅长的是回答：

- 当前问题相关的外部资料是什么

而不是：

- 这个用户长期喜欢什么
- 这项偏好什么时候发生变化

## 3.2 记忆最终怎么存

### Memobase：结构化档案 + 事件流

Memobase 的长期记忆不是简单的聊天记录堆叠。  
它更像三层结构：

1. 原始输入先以 `blob` 形式进入系统
2. 经过 `buffer` 聚合与抽取
3. 沉淀成 `profile` 和 `event`

所以它的长期存储更接近：

- 一组可更新的 profile 槽位
- 一组带时间戳的 event 记录

这是“**结构化用户档案**”思路。

### Mem0：记忆对象 + 混合存储

Mem0 更强调“记忆对象”本身的生命周期。  
不同类型的 memory 可以落到不同介质里，例如：

- 向量存储
- 图数据库
- 键值/文档存储

所以它不是一种固定存储结构，而是“**按记忆类型选择存储后端**”。

### MemGPT / Letta：主上下文 + 外部记忆层级

MemGPT / Letta 的关键不只是外部有没有存，而是：

- 主上下文里有什么
- memory blocks 里有什么
- archival memory 里有什么

它更像“**内存层级**”而不是普通数据库建模。

### Graphiti：时序图 + 图数据库后端

Graphiti 的存储核心是：

- episode
- entity / relation
- temporal graph

它的重点不是存片段，而是把事实变化持续写进图。

### LangMem：store + memory workflows

LangMem 自身不是数据库。  
它更像：

- 一层 memory API
- 一组 memory tools
- 一个 background manager

真正持久化通常落在 LangGraph store 或自定义 store 上。

### Neo4j Agent Memory：图原生 conversation / KG / reasoning

Neo4j Agent Memory 的长期存储更接近：

- conversation memory
- knowledge graph memory
- reasoning traces

而且三者都围绕 Neo4j 图模型展开。

### 通用向量库 / RAG：切片 + embedding

RAG 最典型的存储形态就是：

- 文档切片
- embedding
- 检索索引

它在“存知识片段”这件事上很直接，但在“维护稳定长期状态”这件事上天然不如前三者。

## 3.3 检索出来的到底是什么

### Memobase

通常取回的是：

- 用户 profile
- 与当前问题相关的历史 event
- 由这些内容组装后的 context

它不是简单返回“最相似的聊天片段”，而是返回“**这个用户现在值得带回来的画像和事件**”。

### Mem0

通常取回的是：

- 已经提取好的 memory item
- 和当前 query / user / task 最相关的长期记忆

它更强调“**经过治理后的 memory**”，而不是原始历史回放。

### MemGPT / Letta

取回动作本身是 runtime 的一部分。  
重点不是“查出来什么”，而是：

- 什么时候查
- 查完是否装入主上下文
- 哪些旧内容要迁出

它更强调“**上下文调度**”。

### Graphiti

通常取回的是：

- 和 query 相关的图节点与边
- 带时间语义的关系上下文
- 经过 rerank 的图检索结果

它更强调“**图上下文组装**”。

### LangMem

通常取回的是：

- store 中的 memory entries
- 通过 search tool 找到的相关记忆
- 可能由后台整理过的长期知识

它更强调“**记忆工具调用和工作流**”。

### Neo4j Agent Memory

通常取回的是：

- 对话记忆
- 图谱中的实体关系
- reasoning traces

它更强调“**图原生的多类记忆统一检索**”。

### RAG

通常取回的是：

- 文档切片
- 资料片段
- 排序后的外部知识

它更偏“**知识补充**”。

## 3.4 更新机制有什么不同

### Memobase

更新的重点是：

- 用户画像槽位怎么改
- 哪些新交互应该形成事件
- 某次事件给 profile 带来了什么增量

所以它更像“档案持续更新”。

### Mem0

更新的重点是：

- 一条 memory 应新增还是更新
- 是否应该删除
- 是否要和旧记忆合并

所以它更像“memory CRUD + 演化管理”。

### MemGPT / Letta

更新的重点是：

- 当前上下文是否需要压缩
- 哪些信息需要迁入或迁出
- 何时把信息沉淀到外部记忆

所以它更像“上下文状态管理”。

### Graphiti

更新的重点是：

- 新 episode 如何写入图
- 图中的节点和关系如何增量更新
- 时间变化如何被保留下来

所以它更像“时序图演化管理”。

### LangMem

更新的重点是：

- tool 何时写入记忆
- background manager 如何抽取和整合知识
- store 中的长期记忆如何被持续整理

所以它更像“memory workflow 管理”。

### Neo4j Agent Memory

更新的重点是：

- 对话如何变成 conversation memory
- 实体关系如何进入图谱
- reasoning traces 如何进入长期记忆

所以它更像“图原生多层记忆更新”。

### RAG

更新通常只是：

- 新文档入库
- 重新切片
- 重建或增量更新索引

所以它对“长期状态演化”的表达最弱。

# 四 怎么选

## 4.1 如果你最关心“持续认识用户”

优先看 **Memobase**。

典型特征是：

- 每个用户都有长期档案
- 要记偏好、事件、关系、时间安排
- 系统重心在个性化和用户连续性

## 4.2 如果你最关心“给 Agent 提供统一记忆层”

优先看 **Mem0**。

典型特征是：

- 想统一管理多类型 memory
- 希望支持新增、更新、删除、合并
- 不只关心用户画像，还关心任务经验、事实知识、长期状态

## 4.3 如果你最关心“长任务下上下文怎么调度”

优先看 **MemGPT / Letta**。

典型特征是：

- 输入远大于上下文窗口
- 任务持续时间长
- 必须处理内外存切换和上下文压缩

## 4.4 如果你最关心“关系和时间变化”

优先看 **Graphiti**。

典型特征是：

- 事实和关系会变化
- 需要图结构和时间语义
- 希望通过图检索组装上下文

## 4.5 如果你最关心“给现有 Agent 轻量加记忆能力”

优先看 **LangMem**。

典型特征是：

- 已经在 LangGraph / LangChain 生态里
- 想要 memory tools 和 background manager
- 想保留对存储层的控制

## 4.6 如果你最关心“图原生的 conversation + KG + reasoning memory”

优先看 **Neo4j Agent Memory**。

典型特征是：

- 强关系建模
- 希望 conversation 和记忆图谱统一
- 需要 reasoning traces 进入长期记忆

## 4.7 如果你最关心“把资料查出来给模型看”

优先看 **通用向量库 / RAG**。

典型特征是：

- 外部知识检索是主需求
- 不太需要长期用户状态
- 不需要复杂记忆演化

# 五 一句话区分

- **Memobase**：把“用户”当作长期记忆的中心。
- **Mem0**：把“memory 对象”当作长期记忆的中心。
- **MemGPT / Letta**：把“当前上下文窗口”当作记忆调度的中心。
- **Graphiti**：把“实体、关系、时间变化”当作长期记忆的中心。
- **LangMem**：把“memory tools 与 workflow”当作长期记忆接入的中心。
- **Neo4j Agent Memory**：把“图原生 conversation / KG / reasoning memory”当作长期记忆的中心。
- **RAG / 向量库**：把“文档片段召回”当作主要目标。
