# 一 Graphiti 介绍

[Graphiti](https://github.com/getzep/graphiti) 是 Zep 团队开源的 **temporal context graph / 时序上下文图** 框架，目标是给 AI Agent 提供更适合动态环境的图记忆能力。

如果从 Agent Memory 的视角看，Graphiti 更像：

- **时序知识图谱记忆**
- **图原生 memory engine**
- **围绕实体、关系、事件和时间变化组织的长期记忆层**

它不是把历史对话简单切片做向量检索，而是试图把交互、结构化数据和外部信息持续整合进一张**可查询、可追溯、带时间语义的图**。

## 1.1 核心价值

Graphiti 的核心价值在于：**不仅记住“有什么关系”，还记住“这些关系在什么时间成立、如何变化”**。

这类设计主要解决几个问题：

1. **事实会变化**
   用户状态、业务状态、世界状态并不是静态的。
2. **关系很重要**
   很多任务不只是查某个片段，而是要理解实体之间如何连接。
3. **历史可追溯**
   Agent 不只要知道当前事实，还常常要知道“之前发生过什么变化”。

## 1.2 主要特点

- **时序图结构**：不仅记录实体和关系，还记录时间变化
- **增量更新**：支持持续写入新 episode，而不是每次全量重建图
- **混合检索**：结合语义、关键词、图搜索和 rerank
- **适合动态环境**：适合用户状态、企业数据、外部世界不断变化的场景
- **MCP 友好**：官方提供 Graphiti MCP Server，方便给 MCP 客户端提供图记忆

# 二 核心机制

## 2.1 Episode 驱动写入

Graphiti 的一个关键思路是：**不是直接存“memory item”，而是把新信息作为 episode 写入，再从中构建和更新图结构**。

典型过程可以概括为：

1. Agent 获得新的交互、文档或外部观察
2. 这些内容以 episode 形式进入系统
3. 系统抽取实体、关系和时序变化
4. 图中对应节点、边和时间语义被增量更新
5. 后续 query 到来时，再从图中检索最相关的上下文

这和“先抽一条稳定事实，再存成 profile”不同。  
Graphiti 更强调：**把现实世界的变化过程映射进图**。

## 2.2 时序图记忆

Graphiti 最值得补充的点，是它不是普通知识图谱，而是**temporally-aware knowledge graph**。

这意味着它更适合表达这类问题：

- 某个用户现在住在哪里？
- 这个客户之前在哪个时间段有不同偏好？
- 某个实体关系在什么时候建立、又在什么时候失效？

所以它不只是“图检索”，而是“带时间维度的图记忆”。

## 2.3 检索机制

Graphiti 的检索不是单一模式，通常会结合：

- **语义检索**
- **关键词匹配**
- **图遍历 / 图查询**
- **rerank**

官方 quickstart 里也明确展示了：

- semantic and keyword matching
- graph-based search
- reranking
- source node UUID 相关检索

所以它更像一套**面向图记忆的混合检索引擎**。

## 2.4 存储与后端

从工程实现上看，Graphiti 并不把自己限制在单一后端上。  
官方示例明确支持：

- **Neo4j**
- **FalkorDB**
- **Amazon Neptune**

这说明 Graphiti 的重点不是“发明一个新的图数据库”，而是给图数据库之上增加一层更贴近 Agent Memory 的写入、更新和检索逻辑。

# 三 适合场景

## 3.1 动态用户状态与业务状态

如果系统里的事实经常变化，例如：

- 用户偏好变化
- 账户状态变化
- 企业实体关系变化

那么 Graphiti 会比静态向量检索更自然。

## 3.2 强关系推理场景

例如：

- 客户 / 公司 / 地点 / 产品关系网络
- 复杂实体关系问答
- 带来源追踪的上下文拼装

这类任务需要的不是“找最像的片段”，而是“顺着关系把上下文拼起来”。

## 3.3 长期动态 Agent

对于长期运行、持续接收新观察的 Agent，Graphiti 很适合做：

- 世界模型
- 用户关系图
- 带时间语义的上下文基础设施

# 四 与其他记忆方案的关系

Graphiti 的横向定位可以概括为：

- 它不是简单的向量检索后端
- 它更偏“时序图记忆引擎”
- 它强调实体、关系、事件和时间变化

如果要看它和 Memobase、Mem0、MemGPT、LangMem、Neo4j Agent Memory 的系统对比，统一放在：[10 记忆方案对比.md](./10%20记忆方案对比.md)

# 五 局限与注意点

## 5.1 图建模复杂度更高

Graphiti 的表达能力很强，但图建模、实体抽取、关系治理也更复杂。

## 5.2 对动态数据场景更有优势

如果场景只是 FAQ、静态知识检索或简单 profile memory，Graphiti 往往会显得偏重。

## 5.3 需要图数据库基础设施

虽然 Graphiti 屏蔽了一部分复杂度，但底层仍然依赖图数据库能力，运维和查询思维都比普通向量库更重。

# 六 总结

可以把 Graphiti 理解为：

> 一个围绕“实体、关系、时间变化”组织长期记忆的时序图记忆引擎。

它最重要的价值不在“多存一点上下文”，而在于：

- 让记忆具备关系结构
- 让关系具备时间语义
- 让 Agent 能在动态环境里基于图检索和组装上下文
