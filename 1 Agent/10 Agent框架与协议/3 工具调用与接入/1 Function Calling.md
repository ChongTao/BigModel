# 一 Function Calling 介绍

Function Calling（函数调用）指的是：模型不直接输出自然语言答案，而是先以**结构化参数**的形式表达“应该调用哪个函数、传什么参数”，再由外部程序执行函数，并把结果返回给模型或用户。

它解决的核心问题不是“模型如何自己执行代码”，而是：

- 如何让模型用结构化方式表达调用意图
- 如何让外部程序安全接管执行
- 如何把工具调用从自然语言猜测变成可验证的接口行为

在 Agent 语境里，Function Calling 可以理解为“工具调用入口格式”。它通常位于 Agent 的工具选择与执行链路中，负责把模型的动作意图转换成可执行请求。

## 1.1 为什么需要 Function Calling

如果没有 Function Calling，模型调用工具通常只能靠自然语言描述，例如：

- “请调用天气接口查询上海温度”
- “去执行这个函数”

这样的问题是：

- 参数不稳定
- 难以直接执行
- 容易解析错误
- 缺乏 schema 约束

Function Calling 的价值就在于：让模型输出从“描述动作”变成“提交调用请求”。

## 1.2 它解决什么问题

- 工具调用结构化
- 参数提取标准化
- 运行时更容易校验和兜底
- 多工具场景下更容易做选择与编排

# 二 基本工作流程

一个典型的 Function Calling 流程通常是：

1. 开发者定义可用函数及其参数 schema
2. 用户提出请求
3. 模型判断是否需要调用函数
4. 模型返回结构化调用请求
5. 业务程序执行函数
6. 执行结果返回给模型或直接返回给用户

关键点在于：

- 模型负责“决定调什么、传什么”
- 运行时负责“是否真的执行、怎么执行、结果怎么处理”

## 2.1 一个简单示例

假设定义了一个 `get_weather` 函数：

```json
{
  "name": "get_weather",
  "description": "获取指定城市的实时天气",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名称"
      }
    },
    "required": ["city"]
  }
}
```

当用户说“帮我查看上海天气”时，模型可能返回：

```json
{
  "name": "get_weather",
  "arguments": {
    "city": "上海"
  }
}
```

随后由外部程序真正执行 `get_weather("上海")`，把结果交回系统。

# 三 Function Calling 的关键组成

## 3.1 工具定义

工具定义通常至少包括：

- 名称
- 描述
- 参数 schema
- 必填字段

描述越清楚、schema 越严格，模型越容易稳定地产生正确调用。

## 3.2 调用决策

模型需要判断：

- 当前问题需不需要工具
- 应该调用哪一个工具
- 参数应该如何填充

这一步本质上仍然依赖模型理解能力，但输出形式已经被结构化约束。

## 3.3 运行时执行

模型不会真的自己执行函数，真正执行的是外部程序或 Agent Runtime。

所以 Function Calling 始终是“两段式”的：

- 模型产出调用请求
- 系统执行调用请求

## 3.4 结果回填

函数执行后，结果可以有两种常见处理方式：

- 直接返回给用户
- 再交给模型做整合、解释或下一步推理

后一种方式在多步工具调用或 Agent 工作流里更常见。

# 四 它和 Tool Use / ReAct 的区别

Function Calling 很容易和 Tool Use、ReAct 混在一起，但它们不是同一层。

- Function Calling：强调**调用接口格式**
- Tool Use：强调**模型具备使用工具完成任务的能力**
- ReAct：强调**推理-行动-观察的闭环流程**

可以把三者理解为：

- Function Calling 是“怎么表达调用”
- Tool Use 是“什么时候需要用工具”
- ReAct 是“如何把推理和工具调用串成多步闭环”

所以 Function Calling 更偏接口层，而不是完整推理范式。

# 五 常见问题与稳定性来源

Function Calling 不稳定时，常见原因包括：

- schema 定义含糊，字段语义不清
- 多个工具描述过于相似
- 参数允许范围不明确
- 提示词没有明确何时该调工具、何时不该调
- 运行时缺少参数校验、错误处理和重试机制

## 5.1 常见错误

- 选错函数
- 参数缺失
- 参数类型错误
- 明明不该调用却强行调用
- 应该多步调用却只调用一步

## 5.2 提升稳定性的做法

- 工具名称和描述要避免歧义
- schema 尽量严格，不要过度宽松
- 工具数量不要无节制增长
- 给出明确的调用边界和失败兜底逻辑
- 在运行时做参数校验、权限校验和重试

# 六 与 MCP、Harness 的关系

- 与 MCP 的关系：Function Calling 解决“模型怎么发起结构化调用”，MCP 更关注“工具如何被标准化暴露出来”。
- 与 Harness 的关系：Harness 会把 Function Calling 接入完整的执行、校验、状态管理和安全控制流程。

因此，Function Calling 通常不是终点，而是大模型接入外部能力的**最基础接口层**。

# 七 一句话总结

Function Calling 的本质，是让模型把“我要调用工具”这件事，从自然语言描述，升级为可执行、可校验的结构化接口请求。
