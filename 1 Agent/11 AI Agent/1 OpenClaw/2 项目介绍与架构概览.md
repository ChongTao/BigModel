# OpenClaw 项目介绍与架构概览

## 1 项目定位与整体架构

OpenClaw 可以理解为一个面向多渠道接入的 Agent 运行时：上层连接 Telegram、Slack、微信等消息渠道，中间通过 Gateway 和 Agent 编排推理与执行，下层通过 Skills、Tools、Memory 和插件体系扩展能力。

OpenClaw 的重点不只是“对话”，而是把渠道接入、任务执行、记忆管理和插件扩展整合到同一套 Agent 系统中，适合用于构建长期运行、可接入外部平台、可持续扩展能力的 AI Agent。

## 2 核心概念

| 模块                | 别名            | 职责                                                         |
| :------------------ | :-------------- | :----------------------------------------------------------- |
| **Gateway（网关）** | 心脏 / 神经中枢 | 连接所有渠道，接收指令，负责消息路由、认证授权、会话管理和安全边界 |
| **Agent（智能体）** | 大脑            | 理解意图，规划任务，调度技能                                 |
| **Skills（技能）**  | 双手            | 执行具体操作（搜索、文件、浏览器、脚本等）                   |
| **Memory（记忆）**  | 笔记本          | 记住用户偏好、历史和长期上下文                               |

### 2.1 Gateway（网关）

Gateway 是 OpenClaw 的**中央控制平面**，本质上是一个 HTTP/WebSocket 服务器，所有消息的进出都经过它。

**职责拆解：**

| 子模块                   | 职责                                                    |
| :----------------------- | :------------------------------------------------------ |
| `inbound`                | 接收各渠道的入站消息，封装为标准 Envelope，绑定会话     |
| `auth`                   | 多模式认证（OAuth / API Key / 多账号轮换）+ 速率限制    |
| `call`                   | 通过 ACP 协议触发 Agent 进程，传递消息上下文            |
| `session`                | 维护多轮对话的会话生命周期（绑定、续期、清理）          |
| `outbound`               | 将 Agent 的响应通过渠道插件回传给用户                   |
| `channel-health-monitor` | 实时监控各渠道连接的健康状态                            |
| `boot`                   | 启动时读取 `BOOT.md`，以 Agent 身份执行一次性初始化检查 |

**ACP（Agent 控制协议）：** Gateway 与 Agent 之间的通信协议，负责 Agent 进程的 spawn / stop / query 生命周期管理。每条用户消息最终通过 ACP 触发一个 Agent 实例处理。

这里的 ACP 在本文中主要用于说明 OpenClaw 内部 Gateway 与 Agent 的控制与调度关系，可以先把它理解为系统内部的控制面协议，再结合具体实现判断是否与外部公开协议完全等价。

**核心 API：**

```typescript
startGatewayServer()    // 启动完整 Gateway 服务
runBootOnce()           // 启动时执行 BOOT.md 检查（只执行一次）
```

### 2.2 Channel（渠道）

Channel 是 OpenClaw 与外部消息平台之间的**适配层**，以插件形式实现，每个渠道一个插件包（位于 `extensions/channels/`）。

**渠道插件的能力契约（`ChannelCapabilities`）：**

| 适配器类型  | 说明                                    |
| :---------- | :-------------------------------------- |
| `Auth`      | 账号认证与绑定（OAuth、Token、扫码等）  |
| `Config`    | 渠道级配置 Schema 及 UI 提示            |
| `Messaging` | 接收入站消息，解析消息格式              |
| `Outbound`  | 发送出站消息（文本、图片、文件等）      |
| `Threading` | 消息线程/话题管理（如 Slack 线程回复）  |
| `Groups`    | 群组/频道管理                           |
| `Directory` | 用户目录查询                            |
| `Security`  | 渠道级安全策略（DM 白名单、群组权限等） |
| `Status`    | 渠道状态上报                            |
| `Streaming` | 流式输出支持                            |

**账号状态流转：**

```
not linked → configured → linked → enabled
```

**已支持渠道（29 个）：**
WhatsApp · Telegram · Slack · Discord · 飞书 · iMessage · 微信 · QQ · Signal · LINE · Matrix · Microsoft Teams · Mattermost · Google Chat · IRC · Nextcloud Talk · Nostr · Synology Chat · Tlon · Twitch · Zalo · BlueBubbles · WebChat ...

### 2.3 Skills（技能）

Skills 是 OpenClaw 能力扩展的**基本功能单元**，类似手机 App，封装一类具体的业务能力。

**技能 vs 工具的区别：**

|          | Skills（技能）                   | Tools（工具）          |
| :------- | :------------------------------- | :--------------------- |
| 粒度     | 业务级（如"管理 GitHub Issues"） | 原子级（如"读取文件"） |
| 来源     | 独立包，可单独安装/卸载          | 内置于 Agent 核心      |
| 发现方式 | `AGENTS.md` 注册                 | `TOOLS.md` 注册        |
| 分发     | ClawHub 技能市场                 | 随 OpenClaw 核心发布   |

**安装后的自动行为：**

- 更新 `AGENTS.md`（行为手册，让 Agent 知道新增了哪些能力）
- 更新 `TOOLS.md`（工具清单，程序性记忆）

**内置技能分类（50 个）：**

| 类别     | 代表技能                                                    |
| :------- | :---------------------------------------------------------- |
| 通讯     | `discord`、`slack`、`imsg`（iMessage）                      |
| 生产力   | `apple-notes`、`notion`、`obsidian`、`bear-notes`、`trello` |
| 开发     | `coding-agent`、`github`、`gh-issues`、`oracle`             |
| 媒体     | `spotify-player`、`canvas`、`gifgrep`、`openai-image-gen`   |
| 语音     | `openai-whisper`、`sherpa-onnx-tts`、`voice-call`           |
| 系统监控 | `healthcheck`、`model-usage`、`session-logs`                |
| 实用     | `summarize`、`1password`、`things-mac`、`tmux`              |

```sh
# 从 ClawHub 安装技能
openclaw skills install <技能名>
```

### 2.4 Memory（记忆）

Memory 是 OpenClaw 的**持久化认知系统**，让 Agent 在多次对话之间保持连续性。它不是简单的对话历史，而是结构化的分层记忆。

**技术实现：**

| 能力       | 实现                                                |
| :--------- | :-------------------------------------------------- |
| 语义检索   | 向量嵌入（Embedding）+ 相似度搜索                   |
| 关键词检索 | BM25 全文搜索                                       |
| 混合搜索   | Vector + BM25 融合排序                              |
| 存储后端   | SQLite（默认）/ LanceDB（可选）                     |
| 嵌入提供商 | OpenAI · Gemini · Voyage · Mistral · Ollama · Local |

**核心接口：**

```typescript
interface MemorySearchManager {
  search(query, opts?): Promise<MemorySearchResult[]>  // 语义/混合搜索
  readFile(params): Promise<{ text, path }>             // 读取记忆文件
  sync?(params?): Promise<void>                         // 同步/重建索引
  probeEmbeddingAvailability(): Promise<...>            // 检查嵌入可用性
}

type MemorySearchResult = {
  path: string       // 记忆文件路径
  startLine: number  // 命中段落起始行
  endLine: number    // 命中段落结束行
  score: number      // 相关性评分
  snippet: string    // 内容片段
  source: "memory" | "sessions"
}
```

**记忆文件分层（存储于 `~/.openclaw/workspace/`）：**

| 文件          | 层级       | 说明                           |
| :------------ | :--------- | :----------------------------- |
| `SOUL.md`     | 价值观层   | AI 性格与行为准则，几乎不变    |
| `USER.md`     | 用户模型层 | 对你的认知模型，交互中持续更新 |
| `MEMORY.md`   | 工作记忆层 | 高频读写的当前上下文摘要       |
| `memory/*.md` | 情景记忆层 | 历史事件记录，只追加不修改     |

> Memory 是一个**独占插槽**：同一时刻只能激活一个记忆插件（memory-core 或 memory-lancedb），防止数据冲突。

### 2.5 Tools（工具）

Tools 是 Agent 在执行任务时可直接调用的**原子能力集合**，内置于 `src/tools/` 和 `src/agents/tools/`，无需单独安装。

**工具分类：**

| 工具              | 功能                                     |
| :---------------- | :--------------------------------------- |
| `message-tool`    | 向渠道发送消息（文本/富文本）            |
| `memory-tool`     | 搜索和读取记忆文件                       |
| `browser-tool`    | 浏览器自动化（打开页面、截图、点击）     |
| `canvas-tool`     | 在 Canvas 界面绘制 Agent 驱动的 UI       |
| `cron-tool`       | 创建/管理定时任务（对应 HEARTBEAT.md）   |
| `gateway-tool`    | 调用 Gateway 内部方法                    |
| `discord-actions` | Discord 特定操作（表情回应、频道管理等） |

**工具错误类型：**

```typescript
class ToolInputError extends Error { status: 400 }       // 参数错误
class ToolAuthorizationError extends ToolInputError { status: 403 } // 权限不足
```

**工具 vs 技能：** 工具是 Agent 的"基础指令集"（底层），技能是封装了多个工具调用的"业务流程"（上层）。

### 2.6 Plugin（插件）

Plugin 是 OpenClaw 的**能力扩展机制**，将渠道、模型提供商、工具、钩子等可选能力以插件形式解耦出核心。

**插件分类：**

| 类型                | 说明                   | 示例                                      |
| :------------------ | :--------------------- | :---------------------------------------- |
| **Channel Plugin**  | 接入新的消息渠道       | `@openclaw/discord`、`@openclaw/feishu`   |
| **Provider Plugin** | 接入新的 AI 模型提供商 | `@openclaw/anthropic`、`@openclaw/openai` |
| **Memory Plugin**   | 替换记忆存储后端       | `@openclaw/memory-lancedb`                |
| **Tool Plugin**     | 注册新的 Agent 工具    | 自定义工具包                              |
| **Hook Plugin**     | 注入生命周期钩子       | 消息预处理、审计日志                      |

**插件执行模型：**

| 类型               | 运行方式                                 | 信任级别         |
| :----------------- | :--------------------------------------- | :--------------- |
| 原生插件（Native） | 进程内（in-process），通过 jiti 动态加载 | 完全信任，无沙箱 |
| Bundle 插件        | 仅元数据/内容包，安全默认                | 受限             |

**插件生命周期：**

```
package.json 声明 openclaw.extensions 字段
   ↓
Gateway 启动时自动发现插件包
   ↓
读取 openclaw.plugin.json 清单，验证配置 Schema
   ↓
检查启用/禁用/互斥槽（如 memory 插槽唯一性）
   ↓
通过 jiti 加载 index.ts，注入 PluginRuntime
   ↓
注册到对应表面（渠道/提供商/工具/路由/命令）
```

**PluginRuntime（插件运行时上下文）：**

```typescript
interface PluginRuntime {
  version: string           // OpenClaw 版本
  config: RuntimeConfig     // 配置访问
  subagent: SubagentRuntime // 子 Agent 调度
  system: SystemRuntime     // 系统操作
  media: MediaRuntime       // 媒体处理
  tts: { textToSpeechTelephony }  // 语音合成
  tools: ToolsRuntime       // 工具执行
  channel: ChannelRuntime   // 渠道操作
  events: EventsRuntime     // 事件总线
  logging: LoggingRuntime   // 日志（自动脱敏）
  modelAuth: { getApiKeyForModel, resolveApiKeyForProvider }
}
```

**开发插件（最小示例）：**

```typescript
// index.ts
import { definePlugin } from '@openclaw/plugin-sdk'

export default definePlugin({
  id: 'my-plugin',
  setup(runtime: PluginRuntime) {
    runtime.tools.register({
      name: 'my_tool',
      description: '做某件事',
      run: async (input) => { /* ... */ }
    })
  }
})
```

> 新插件优先发布到 npm，通过本地 extension 加载进行开发调试。社区插件列表：https://docs.openclaw.ai/plugins/community

### 2.7 典型请求处理链路

从架构职责上看，一次典型请求通常按下面链路流转：

1. 用户消息从 Telegram、Slack、微信等渠道进入 Gateway。
2. Gateway 完成入站封装、认证校验、会话绑定，并通过 ACP 将请求分发给目标 Agent。
3. Agent 结合 `SOUL.md`、`USER.md`、`MEMORY.md` 等上下文做意图理解与任务规划。
4. Agent 按需调用内置 Tools 或已安装 Skills，执行搜索、文件处理、浏览器操作、外部 API 调用等动作。
5. 执行结果会回写到工作记忆或历史记忆，再由 Gateway 通过对应渠道回传给用户。

这条链路对应的是 OpenClaw 最核心的闭环：`渠道接入 -> Agent 推理 -> Skills/Tools 执行 -> Memory 更新 -> 渠道回传`。



## 3 配置文件系统

OpenClaw 的配置分为两大类：

### 3.1 系统配置（System Configuration）

**存储路径：`~/.openclaw/`**

| 文件/目录 | 说明 | 权限 |
| :--- | :--- | :--- |
| `config.yml` | 主配置文件（YAML + Zod Schema 验证）<br/>包含：Gateway 监听地址、模型提供商、渠道启用状态 | 通常 0600 |
| `sessions.json` | 会话存储库（Session Store）<br/>保存各渠道的登录凭证和账户绑定信息 | 加密存储 |
| `auth/` | 加密凭证目录<br/>保存 OAuth 令牌、API Key 等敏感信息 | 0700（目录只读） |
| `secrets/` | 密钥库（Credentials Vault）<br/>支持 pass / 1Password / AWS Secrets Manager 等后端 | 加密 |
| `workspace/` | 认知文件系统（Cognitive File System）<br/>存放 Agent 的持久化记忆（SOUL.md / USER.md / MEMORY.md 等） | 0755 |
| `extensions/` | 本地插件目录<br/>用于开发/加载本地 Channel / Tool 插件 | 0755 |
| `logs/` | 日志目录<br/>包含 Gateway / Agent / 渠道的运行日志 | 0755 |

**config.yml 示例结构：**

```yaml
# Gateway 配置
gateway:
  host: "0.0.0.0"
  port: 3000
  baseUrl: "http://localhost:3000"

# 启用的渠道
channels:
  telegram:
    enabled: true
    config:
      botToken: "${TELEGRAM_BOT_TOKEN}"  # 从环境变量读取
  slack:
    enabled: true
    config:
      botToken: "${SLACK_BOT_TOKEN}"
  wechat:
    enabled: false

# AI 模型提供商
providers:
  anthropic:
    apiKey: "${ANTHROPIC_API_KEY}"
    model: "claude-opus-4-6"
  deepseek:
    apiKey: "${DEEPSEEK_API_KEY}"
    model: "deepseek-chat"
  openai:
    apiKey: "${OPENAI_API_KEY}"
    model: "gpt-4"

# 记忆存储后端
memory:
  backend: "sqlite"  # 可选：sqlite / lancedb
  embeddingProvider: "openai"  # 向量嵌入提供商
  vectorDimension: 1536

# 日志级别
logging:
  level: "info"  # debug / info / warn / error
```

### 3.2 认知文件系统（Cognitive File System）

OpenClaw 将 Agent 的认知状态映射到 `~/.openclaw/workspace/` 下的 Markdown 文件，形成**结构化的分层记忆**。

| 文件 | 认知对应 | 特点 | 更新频率 |
| :--- | :--- | :--- | :--- |
| **SOUL.md** | 超我 / 价值观 | AI 的性格、底线和行为准则；定义"这个 AI 是谁" | 静态，极少变更 |
| **IDENTITY.md** | 自我概念 | AI 的名字、昵称、头像 URL 等外在形象 | Bootstrap 时生成，后续稳定 |
| **USER.md** | 心智理论 | 对你（用户）的认知模型：工作、兴趣、偏好、禁忌 | Agent 动态更新，越用越懂你 |
| **MEMORY.md** | 语义记忆 | 高频读写的当前工作上下文和摘要 | 日常读写，压缩和过期机制 |
| **memory/*.md** | 情景记忆 | 历史事件记录，记录每次对话和任务执行 | 仅追加（Append-Only），永久保存 |
| **AGENTS.md** | 行为手册 | Agent 可用的所有技能（Skills）清单 | 安装新技能时自动更新 |
| **TOOLS.md** | 程序性记忆 | Agent 可用的所有原子工具（Tools）清单 | Agent 按需更新 |
| **HEARTBEAT.md** | 定时任务清单 | 周期性自动执行的任务（每日检查、提醒等） | 周期性读取 |
| **BOOTSTRAP.md** | 出生仪式 | 首次启动时执行的一次性初始化检查 | 完成后自动删除 |

**SOUL.md 示例：**

```markdown
# 我的灵魂

## 性格
- 友好、耐心、专业
- 遇到不确定的事会坦诚说出
- 遵守用户隐私

## 核心价值观
- 帮助用户高效完成工作
- 保护用户数据安全
- 持续学习和改进

## 底线（禁忌）
- 不生成恶意代码
- 不参与非法活动
- 不泄露用户隐私
```

**USER.md 示例：**

```markdown
# 关于你

## 基本信息
- 名字：张三
- 职位：产品经理
- 工作时区：GMT+8

## 工作偏好
- 喜欢简洁的摘要和要点列表
- 不喜欢超过 500 字的回复
```

### 3.3 记忆分层模型

记忆系统遵循**金字塔结构**，从稳定的价值观到实时的工作记忆：

| 层级 | 文件 | 容量 | 更新策略 | 用途 |
| :--- | :--- | :--- | :--- | :--- |
| **价值层**（L4 - 超我） | `SOUL.md` | ~2KB | 静态或手动更新 | 定义 Agent 性格和底线 |
| **身份层**（L3 - 自我） | `IDENTITY.md` | ~1KB | Bootstrap 固定 | 外在形象（名字、头像） |
| **用户模型层**（L2） | `USER.md` | ~10KB | Agent 动态学习 | 对用户的认知（工作、偏好、禁忌） |
| **工作记忆层**（L1） | `MEMORY.md` | ~50KB | 高频读写+过期清理 | 当前任务上下文、最近交互摘要 |
| **历史档案层**（L0 - 底层） | `memory/*.md` | 无限制 | 仅追加（Append-Only） | 完整的历史事件记录，用于长期学习 |

**更新机制示意图：**

```
User Input
    ↓
┌─────────────────────────────────────────┐
│ Agent 推理                              │
│ 1. 检索 SOUL.md（底线检查）             │
│ 2. 检索 USER.md（用户模型）             │
│ 3. 搜索 MEMORY.md + memory/*.md（历史）  │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Agent 任务执行                          │
│ 调用 Skills & Tools                     │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ 写入记忆（Append）                      │
│ 1. 追加到 memory/[timestamp].md（历史）  │
│ 2. 更新 USER.md（学习用户偏好）         │
│ 3. 更新 MEMORY.md（压缩工作摘要）       │
└─────────────────────────────────────────┘
```

### 3.4 配置初始化流程

**初始化命令：**

```sh
# 交互式引导（推荐）
openclaw onboard

# 快速模式（跳过详细步骤）
openclaw onboard --flow quickstart
```

**初始化步骤：**

1. **环境检查** ✓
   - Node.js 版本（需 v22+）
   - 磁盘空间
   - 网络连接

2. **系统配置** ⚙️
   - 选择 Gateway 监听端口（默认 3000）
   - 选择主 AI 模型提供商（Anthropic / DeepSeek / OpenAI 等）
   - 输入模型 API Key

3. **渠道启用** 📱
   - 选择要集成的消息渠道（Telegram / Slack / WeChat 等）
   - 配置各渠道的认证凭证（BotToken、OAuth 等）

4. **认知初始化** 🧠
   - 自动生成 IDENTITY.md（AI 名字和形象）
   - 生成 SOUL.md（AI 性格，可手动编辑）
   - 生成 BOOTSTRAP.md（首次启动检查）

5. **启动并验证** ✅
   - 启动 Gateway 服务
   - 执行一次测试对话
   - 删除 BOOTSTRAP.md

### 3.5 环境变量支持

配置文件支持环境变量注入，推荐的做法：

**`.env` 文件（不要提交到 git）：**

```bash
ANTHROPIC_API_KEY=sk-xxxx
DEEPSEEK_API_KEY=sk-xxxx
TELEGRAM_BOT_TOKEN=123456:ABC-DEF1234ghIkl-zyx57W2v1u123ew11
SLACK_BOT_TOKEN=xoxb-xxxx
```

**config.yml 中引用：**

```yaml
providers:
  anthropic:
    apiKey: "${ANTHROPIC_API_KEY}"
  deepseek:
    apiKey: "${DEEPSEEK_API_KEY}"

channels:
  telegram:
    config:
      botToken: "${TELEGRAM_BOT_TOKEN}"
```

> **安全提示**：敏感信息（API Key、Token）应通过环境变量或密钥库传入，不要硬编码到 config.yml。
