# 一 Skill 介绍

Skill 可以理解为：**把某一类任务的处理经验封装成一个可复用的能力包，供 Agent 在合适的时候动态加载和执行。**

通俗一点说，Skill 就是在给 AI 写一份“任务说明书”：

- 这类任务什么时候触发
- 遇到这类任务时应该怎么做
- 可以用哪些工具
- 需要遵守哪些约束
- 有哪些参考资料和脚本可以直接复用

从这个角度看，Skill 不是某一家产品独有的概念，而是一种更通用的 **Agent 能力封装模式**。  
Anthropic 的 Claude Skills、OpenSkills、一些 Agent 框架里的 task pack / capability pack，本质上都属于这个方向。

## 1.1 Skill 在 Agent 架构中的位置

如果放到 Agent 系统里看，Skill 通常位于：

- **Model / LLM** 之上：模型负责理解目标和做决策
- **Tool** 之上：工具提供原子能力，例如搜索、执行代码、调用 API
- **Harness / Runtime** 之内：运行时负责判断何时触发 Skill、加载哪些文件、怎么执行

可以简单理解为：

- **Tool**：解决“能做什么”
- **Skill**：解决“这类任务通常应该怎么做”
- **Workflow**：解决“固定流程如何编排”

因此，Skill 更像是“**任务级经验封装**”，而不是单一工具，也不是完全写死的流程。

## 1.2 为什么需要 Skill

如果没有 Skill，Agent 遇到每一类任务都要从零开始推理：

- 这类任务一般怎么拆解
- 应该先用什么工具
- 哪些步骤容易出错
- 有哪些参考资料需要先读

这样虽然通用，但不稳定，也浪费上下文和推理成本。

Skill 的作用就是把这些高频任务经验提前沉淀下来，让 Agent 在遇到相似任务时：

- 不用重新摸索
- 更稳定地调用工具
- 更快进入可执行状态
- 更容易形成可复用的专业能力

# 二 Skill 的核心特征

一个成熟的 Skill 通常具备以下特征：

## 2.1 可复用

Skill 不是为一次任务而写，而是为一类任务服务。  
例如“PDF 表单处理”“代码评审”“需求澄清”“SQL 排障”都可以沉淀为 Skill。

## 2.2 可组合

多个 Skill 可以在同一任务中组合使用。  
例如先用“需求澄清 Skill”补全边界，再用“代码生成 Skill”实现，再用“代码评审 Skill”检查结果。

## 2.3 可执行

Skill 不只是文字说明，它通常还会带：

- 脚本
- 模板
- 示例
- 参考文档

因此它不仅告诉 Agent “要做什么”，还告诉 Agent “可以怎么做”。

## 2.4 可迁移

只要 Skill 的组织方式清晰，它就可以在不同项目、不同 Agent、不同运行时之间迁移和复用。

# 三 Skill 的结构

不同生态对 Skill 的目录结构定义不完全一样，但一个典型 Skill 往往包含以下几类内容：

| 文件或目录 | 作用 | 常见内容 |
| --- | --- | --- |
| 核心说明文件 | 定义 Skill 的用途、触发条件、执行步骤 | `SKILL.md`、`README.md`、任务说明 |
| 辅助文档 | 补充规则、边界条件、参考知识 | `REFERENCE.md`、约束说明、FAQ |
| scripts 目录 | 存放可执行脚本 | Python、Shell、Node.js 脚本 |
| templates / examples | 提供模板和任务示例 | prompt 模板、配置模板、输入输出样例 |
| 其他附属文件 | 补充使用信息 | 许可证、依赖说明、测试数据 |

Claude 官方 Skills 常见的核心文件是 `SKILL.md`，其中通常会包含：

- Skill 名称
- 描述
- 适用任务
- 触发条件
- 执行指令
- 依赖工具或脚本

一个简化示例如下：

```yaml
name: pdf-form-processing
description: 处理 PDF 表单识别、字段检查和自动填充
trigger_conditions:
  - 用户提到“PDF 表单”
  - 用户上传 PDF 并要求填写字段
dependencies:
  - PyPDF2
  - pdfrw
```

然后正文部分再补充：

- 遇到这类任务时的执行步骤
- 工具调用约束
- 异常处理方式
- 输出格式要求

# 四 Skill 如何工作

Skill 的工作流程通常可以概括为：

1. **用户提出任务**
   Agent 收到请求，例如“帮我分析这个 PDF 表单”。
2. **运行时判断是否命中某个 Skill**
   系统根据任务描述、上下文和触发条件，判断某个 Skill 是否相关。
3. **加载最少必要信息**
   通常只加载 Skill 元数据、核心指令，以及本次任务真正需要的辅助文件和脚本。
4. **按 Skill 指令执行**
   Agent 根据 Skill 中定义的流程去选择工具、调用脚本、读取参考资料。
5. **返回结果并继续主任务**
   Skill 不是终点，它通常是 Agent 整体任务链中的一个可复用步骤。

它的关键价值在于：  
**让 Agent 遇到某类任务时，不必从零开始思考“怎么做”，而是可以直接调用一套成熟的处理套路。**

# 五 Claude Skills 作为一个具体实现

Anthropic 在 Claude 生态中把 Skill 做成了较明确的产品形态：  
将完成特定任务所需的说明、脚本、资源等打包成目录，供 Claude 在任务执行时动态发现和加载。

这一实现有几个特点：

- 元数据与正文分离
- 按需加载，不一次性加载所有内容
- 可以带脚本和附属文件
- 适合沉淀高频、专业化任务能力

因此，Claude Skills 是理解 Skill 机制的一个很好的具体例子，但不应该把 Skill 仅仅理解成“Claude 专属功能”。

# 六 Skill 与 Tool、Workflow、MCP 的区别

## 6.1 Tool 和 Skill

- **Tool**：原子能力，例如搜索引擎、Shell、数据库查询、浏览器操作
- **Skill**：一类任务的经验封装，定义“遇到这种任务通常如何组合使用工具”

可以理解为：

- Tool 是“有什么能力”
- Skill 是“怎么把这些能力用在一类任务上”

## 6.2 Skill 和 Workflow

- **Workflow**：预先定义好的固定流程，强调可控和稳定
- **Skill**：针对某类任务提供处理策略，但执行时仍保留一定灵活性

因此：

- 流程完全固定时，更适合 Workflow
- 任务模式重复出现，但细节每次不同，更适合 Skill

## 6.3 Skill 和 MCP

MCP 负责把外部世界的能力接进来，例如：

- 文件系统
- 浏览器
- 搜索
- 企业内部系统

Skill 则负责定义：

- 什么时候用这些能力
- 先用哪个，再用哪个
- 调用时要注意什么

可以理解为：

- **MCP / Tool** 负责连接能力
- **Skill** 负责组织任务经验

# 七 适合场景

Skill 特别适合以下场景：

- 某类任务反复出现，但每次细节不同
- 需要多步处理，但还不至于写死成完整 workflow
- 某类任务需要专业知识、约束和参考资料
- 某类任务经常要配合多种工具一起完成

例如：

- PDF 处理
- 代码评审
- 需求澄清
- 日报生成
- 故障排查
- 数据清洗

# 八 局限与注意点

## 8.1 Skill 不是越多越好

如果 Skill 过多、边界重叠严重，就会带来：

- 触发冲突
- 维护成本上升
- Agent 选择困难

## 8.2 Skill 需要持续维护

Skill 里包含的流程、脚本、依赖、参考文档都可能过时。  
如果不维护，Skill 会从“经验资产”变成“错误知识源”。

## 8.3 Skill 不能替代基础能力

Skill 再强，也仍然依赖底层：

- 模型能力
- 工具能力
- Runtime 调度能力

所以 Skill 更像“能力放大器”，而不是独立系统。

# 九 示例

以 Claude 官方 PDF 操作 Skill 为例，它通常会包含：

- 核心说明文件：定义 PDF 相关任务如何处理
- 辅助文档：补充 PDF 表单、字段、格式要求
- scripts：执行字段检测、表单填充、结果导出

这个例子说明的不是“PDF 很重要”，而是：

> 当某类任务反复出现，并且已经有稳定做法时，就很适合沉淀成 Skill。

另一个常见例子是“设计审问 / 需求澄清”类 Skill，例如在真正开始编码前，先通过一套固定提问框架把目标、边界条件和隐含假设挖出来。
