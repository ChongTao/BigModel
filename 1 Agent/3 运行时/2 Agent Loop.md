# 二 Agent Loop

Agent Loop 是 Agent Runtime 中最核心的执行闭环。  
如果说 Runtime 决定了 Agent 运行所依赖的整体系统，那么 Loop 决定的是：

**Agent 每一轮到底怎么思考、怎么行动、怎么读取反馈，以及什么时候停下来。**

很多看起来“很聪明”的 Agent，真正决定效果的并不是模型参数，而是 Loop 设计得是否稳定。

## 2.1 Agent Loop 是什么

Agent Loop 可以理解为 Agent 的基本工作循环。  
它会不断重复下面这类过程：

1. 理解当前目标和状态
2. 决定下一步动作
3. 调用工具或执行操作
4. 观察返回结果
5. 更新上下文、状态和记忆
6. 判断继续、暂停还是结束

最经典的表达方式是：

```python
while not done:
    think()
    act()
    observe()
```

这也是很多 Agent 系统采用 ReAct 风格的原因。  
Agent 并不是“一次生成完整答案”，而是通过多轮决策与反馈，逐步逼近目标。

从这个角度看，Agent 和普通对话式 LLM 应用的差别就在于：

- 普通 LLM App：一次输入，一次输出
- Agent：围绕目标持续循环执行，直到完成或被终止

## 2.2 一个最小 Agent Loop

一个最小可用的 Agent Loop，通常只需要四个核心环节：

`Goal -> Thought -> Action -> Observation -> Next Step`

可以写成更完整一点的伪代码：

```python
state = init_state(task)

while not state.done:
    context = build_context(state)
    decision = llm(context)

    if decision.type == "finish":
        state.done = True
        state.output = decision.answer
        break

    result = invoke_tool(decision.tool, decision.args)
    state = update_state(state, decision, result)
```

这个最小闭环已经包含了 Agent 的几个关键特征：

- 有任务目标
- 有状态
- 能做动作
- 能读取反馈
- 能基于反馈继续决策

如果没有这个闭环，系统通常只能算“带工具的 LLM”，而不是完整 Agent。

## 2.3 Loop 中每一步在做什么

虽然不同框架会把 Loop 拆成不同阶段，但核心步骤大体一致。

### 2.3.1 Receive Goal

Loop 首先要接收目标。这个目标不只是用户的一句话，还应包含：

- 任务约束
- 可用工具
- 输出要求
- 权限边界
- 当前环境信息

目标越模糊，Loop 越容易在后面出现漂移。

### 2.3.2 Think / Decide

这一阶段的核心是让模型判断：

- 当前是不是已经有足够信息
- 是否需要调用工具
- 下一步应该做什么
- 是否需要分解子任务
- 是否应该请求人工介入

它不一定总是“深度规划”，很多时候只是决定当前回合的最优动作。

### 2.3.3 Act

Act 阶段是 Agent 真正对外部世界产生影响的时候，例如：

- 搜索资料
- 读取文件
- 执行 Shell
- 查询数据库
- 调用业务 API
- 修改代码
- 发送消息

如果一个系统只有 Think，没有 Act，那么它只能提供建议，而不能真正推进任务。

### 2.3.4 Observe

Observe 是读取外部反馈的阶段。常见反馈包括：

- 工具输出
- 错误信息
- 文件变化
- 页面变化
- 测试结果
- 审批结果

这一阶段的价值在于让 Agent 不只是“按计划执行”，而是能基于真实结果调整后续动作。

### 2.3.5 Update

执行和观察之后，Loop 需要更新内部状态，例如：

- 当前是否完成某一步
- 是否需要重试
- 哪些变量已经确定
- 哪些工件已经生成
- 是否要写入短期或长期记忆

如果没有 Update，Agent 就很容易重复劳动，或者忘记刚刚发生了什么。

### 2.3.6 Continue / Stop

每轮结束时，Loop 都要做一个判断：

- 继续下一轮
- 暂停等待外部输入
- 进入人工审批
- 终止任务

终止条件设计不清，是很多 Agent 成本失控和行为漂移的直接原因。

## 2.4 Loop 如何和工具、记忆、状态配合

Agent Loop 从来不是单独存在的，它总是和 Runtime 里的其他模块耦合在一起。

### 2.4.1 和工具配合

Loop 决定“要不要调用工具”，Tool Runtime 决定“怎么安全、稳定地调用工具”。

典型链路是：

`Loop 决策 -> 选择 Tool -> 参数构造 -> 执行 -> 返回结果 -> 回到 Loop`

这里最关键的不是“能不能调工具”，而是：

- 工具选择是否正确
- 参数是否完整
- 权限是否匹配
- 错误是否可恢复
- 返回结果是否能被后续回合正确理解

### 2.4.2 和记忆配合

Loop 不可能每轮都只依赖当前输入，它必须和记忆系统协作。

常见做法是：

- 把当前任务的中间结果放入短期记忆
- 把用户偏好、历史经验写入长期记忆
- 把当前工作区状态写入 workspace memory

Loop 的稳定性很大程度上取决于记忆写入和读取是否合理。  
记得太少会遗忘，记得太多会污染上下文。

### 2.4.3 和状态配合

状态系统让 Loop 具备“持续任务能力”。

例如状态里可能记录：

```text
step=search_repo
retry=1
tool=git
artifact=patch.diff
status=running
```

这样下一轮 Loop 就不需要重新猜测自己处于什么阶段，而是可以直接接着推进。

### 2.4.4 和上下文管理配合

Loop 每次调用模型前，都依赖 Context Manager 组装上下文。  
也就是说，Loop 决定“现在要思考什么”，Context Manager 决定“模型能看到什么”。

很多 Agent 的失败并不是思维能力不够，而是上下文给错了、给多了，或者给得太乱。

## 2.5 什么时候继续，什么时候终止

Loop 设计里最容易被低估的一点，就是终止条件。

一个好的 Agent Loop，必须明确知道什么时候应该继续，什么时候应该停。

### 2.5.1 应该继续的典型情况

- 目标还没有完成
- 当前信息不足，需要继续检索或调用工具
- 上一步失败但可以恢复
- 存在清晰的下一步动作
- 正在等待某个中间结果

### 2.5.2 应该终止的典型情况

- 目标已经完成
- 已经得到足够可信的最终结果
- 遇到不可恢复错误
- 触发风险边界或权限限制
- 达到最大步数、最大成本或最长时间
- 明确需要人工接管

### 2.5.3 常见终止策略

工程里常见的终止策略通常包括：

- `max_steps`
- `max_cost`
- `max_tool_calls`
- `timeout`
- `confidence / verifier`
- `human_approval_required`

终止条件越明确，Loop 越不容易失控。

## 2.6 常见失控方式

Agent Loop 的问题，很多不是“第一步就错”，而是在循环中逐渐失控。

### 2.6.1 无限循环

Agent 一直在重复“思考 -> 调工具 -> 再思考”，但没有真正推进任务。  
常见原因包括：

- 没有明确终止条件
- 工具结果无法让模型判断是否完成
- 状态没有被正确更新

### 2.6.2 工具误调用

模型选错工具、构造错参数、误解工具返回，都会让 Loop 不断偏离目标。

这种问题在多工具场景尤其常见，因为模型未必真正理解每个工具的边界。

### 2.6.3 上下文漂移

Loop 越跑越长后，最初目标被后续噪音淹没，Agent 开始关注不重要的信息。

这通常和上下文膨胀、摘要失真、错误中间结果反复注入有关。

### 2.6.4 记忆污染

如果把错误的观察结果、不可靠推断或无关内容写入记忆，后续每一轮都可能被污染。

一旦记忆污染发生，Loop 往往不是单次出错，而是持续出错。

### 2.6.5 提前结束

模型经常会“自以为完成了”。  
如果没有校验器、测试或观察环节，Loop 可能过早返回一个并不可靠的结果。

### 2.6.6 成本失控

即便 Loop 没有明显逻辑错误，也可能因为：

- 思考轮数过多
- 工具调用过多
- 上下文过长
- 反复重试

而导致整体成本和延迟不可接受。

## 2.7 让 Agent Loop 更稳定的做法

稳定的 Loop 通常不是靠“更聪明的模型”获得，而是靠更严格的工程约束获得。

### 2.7.1 明确状态机

不要让 Agent 每一轮都从零开始猜自己在哪里。  
用显式状态表示执行阶段，通常会比完全自由循环稳定得多。

### 2.7.2 限制每轮目标

一轮只解决一个清晰问题，例如：

- 先决定是否需要搜索
- 再决定调用哪个工具
- 再处理结果

每轮目标越清晰，Loop 越不容易混乱。

### 2.7.3 强化观察和校验

不要只依赖模型自我判断“已完成”。  
尽量引入：

- 测试
- verifier
- 结构化检查
- 人工确认

让“完成”建立在可验证反馈上。

### 2.7.4 控制上下文窗口

对长任务要主动做：

- 摘要压缩
- 关键状态提取
- 工具结果裁剪
- 无关信息清理

否则上下文膨胀会让 Loop 越跑越不稳定。

### 2.7.5 让工具接口更清晰

Tool 描述越模糊，Loop 越容易误用。  
稳定的 Runtime 通常会给工具提供：

- 明确用途
- 严格 schema
- 参数约束
- 返回结构
- 错误类型

### 2.7.6 设置硬边界

很多时候真正让 Loop 稳定的，不是更自由，而是更严格：

- 限制最大步骤数
- 限制最大写入范围
- 限制高风险工具
- 对关键动作强制审批

这也是企业 Agent 和实验 Agent 的重要区别。

### 2.7.7 引入暂停与恢复

复杂 Loop 不应该只有“继续”或“失败”两种结果。  
更合理的设计是允许：

- 暂停
- checkpoint
- 人工修正
- 从指定状态恢复

这样长任务才有真正的可运营性。

## 2.8 Loop 与 Workflow 的边界

Agent Loop 和 Workflow 经常一起出现，但两者并不等价。

- **Loop** 更强调动态决策和反馈闭环
- **Workflow** 更强调固定步骤和可预测执行

一个简单理解方式是：

- 如果任务路径基本固定，更适合 Workflow
- 如果每一步都要根据观察结果动态决定，更适合 Agent Loop

真实系统里更常见的是混合模式：

- 外层由 Workflow 决定阶段
- 内层某个节点由 Agent Loop 自主运行

例如：

```text
Workflow:
Plan -> Search -> Code -> Test -> Review

其中 Search 或 Code 节点内部再运行一个 Agent Loop
```

这类混合设计通常比“全自由 Loop”更稳定，也比“全固定 Workflow”更灵活。

## 2.9 总结

Agent Loop 是 Agent 最小执行闭环，可以概括成：

`理解目标 -> 决定动作 -> 执行动作 -> 观察反馈 -> 更新状态 -> 再次决策`

它决定了 Agent 是否真的具备持续推进任务的能力。  
一个好的 Loop，不只是能跑起来，更要满足几个条件：

- 知道当前在做什么
- 知道为什么继续
- 知道什么时候停止
- 能利用反馈修正自己
- 不会因为长链路而逐渐失控

很多 Runtime 的核心竞争力，本质上都体现在 Loop 的设计质量上。  
Loop 稳不稳定，往往直接决定 Agent 是实验 Demo，还是可进入真实业务的系统。
