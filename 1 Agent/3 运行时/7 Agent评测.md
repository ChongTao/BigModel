# 七 Agent 评测

Agent 评测比普通模型评测更复杂。  
原因不是 Agent “更高级”，而是 Agent 的输出不再只是一个答案，而是一段完整执行过程：

- 它会多轮推理
- 会调用工具
- 会读写状态
- 会依赖上下文和记忆
- 会在环境中产生真实动作

因此，评测 Agent 时，不能只问“答案对不对”，还要问：

- 它是不是可靠完成了任务
- 它有没有走对过程
- 它是否安全、稳定、可恢复
- 它的成本、时延和成功率是否可接受

## 7.1 为什么 Agent 很难评测

普通 LLM 评测，很多时候是：

`输入 -> 输出 -> 对比标准答案`

但 Agent 往往是：

`任务 -> 多轮决策 -> 工具调用 -> 环境反馈 -> 状态更新 -> 最终结果`

这会带来几个评测难点。

### 7.1.1 路径不唯一

同一个任务，Agent 可能走出不同路径：

- 先搜索再调用工具
- 先读文件再制定计划
- 先调用工具 A，失败后切换到工具 B

最终结果可能一样，但执行轨迹完全不同。  
这意味着 Agent 很难像传统 QA 那样只有一个标准过程。

### 7.1.2 环境有不确定性

Agent 的结果常常受环境影响：

- 外部 API 变动
- 网络延迟或失败
- 工具返回不稳定
- 数据库状态变化
- 文件系统内容变化

这会导致同一个输入，在不同时间或不同环境里得到不同结果。

### 7.1.3 失败不只是一种

Agent 失败不只是“答错了”，还可能是：

- 选错工具
- 参数构造错误
- 陷入循环
- 提前结束
- 成本过高
- 越权执行
- 正确完成了结果，但过程不可接受

因此，Agent Eval 必须能区分不同类型的失败，而不是只统计一个总分。

### 7.1.4 结果正确不代表过程正确

有些 Agent 虽然最终结果看起来对，但可能：

- 依赖了错误中间推理
- 调用了不该调用的工具
- 泄露了过多上下文
- 进行了高风险操作
- 使用了不可复现的偶然路径

这类问题如果只看终局结果，很容易被掩盖。

## 7.2 Agent 评测和普通模型评测的区别

Agent 评测和普通模型评测最大的区别，在于评测对象不再只是“文本输出质量”，而是“任务执行系统质量”。

可以粗略对比如下：

| 对比项 | 普通模型评测 | Agent 评测 |
| --- | --- | --- |
| 评测对象 | 单轮回答 | 多步执行过程 |
| 输入输出 | Prompt -> Answer | Task -> Trace -> Result |
| 环境依赖 | 相对较弱 | 很强 |
| 工具调用 | 通常没有 | 通常是核心能力 |
| 状态影响 | 弱 | 强 |
| 失败类型 | 答错、幻觉 | 跑偏、循环、误调用、越权、成本失控 |
| 评测重点 | 答案质量 | 结果、轨迹、稳定性、安全性、效率 |

因此，Agent Eval 更接近系统评测，而不只是模型打分。

## 7.3 常见评测维度

一个可落地的 Agent Eval，通常至少要覆盖下面几类维度。

### 7.3.1 任务完成率

这是最基础的指标，用来回答：

**任务最终是否完成。**

例如：

- Jira 工单是否被正确查询
- 代码是否真的修改成功
- MR 是否真的创建
- 最终回复是否满足指定格式

任务完成率重要，但绝对不够。

### 7.3.2 正确性

正确性不只是“有结果”，而是“结果对不对”。

例如：

- 返回的信息是否真实
- 修改的文件是否正确
- 生成的 SQL 是否执行出正确结果
- 总结是否与原始内容一致

很多 Agent 的假阳性问题就出在这里：  
它“完成了动作”，但结果其实不正确。

### 7.3.3 稳定性

稳定性回答的是：

**同类任务多次运行，表现是否一致。**

常见关注点包括：

- 成功率波动大不大
- 是否对 prompt 微小变化过于敏感
- 是否经常某次成功、某次失败
- 长链路任务是否容易后期漂移

没有稳定性，Agent 很难进入生产环境。

### 7.3.4 工具使用质量

Tool Use 是 Agent 的核心能力之一，所以应该单独评测：

- 是否选对工具
- 是否传对参数
- 是否理解返回结果
- 是否进行了不必要的调用
- 出错后是否正确恢复

一个常见误区是只看“最终完成率”，忽略工具调用效率和质量。

### 7.3.5 成本与时延

很多 Agent 不是不能完成任务，而是代价过高。  
因此评测时通常也要关注：

- 总 token 消耗
- 总推理轮数
- 工具调用次数
- 总耗时
- 首次有效响应时间

如果一个 Agent 成功率略高，但成本高 5 倍，未必是更好的方案。

### 7.3.6 安全与治理

企业 Agent 不能只评“能不能做成”，还要评“有没有越界”。

例如：

- 是否访问了不该访问的文件
- 是否触发了高风险工具
- 是否绕过审批
- 是否输出了敏感信息
- 是否在被拒绝后仍尝试危险动作

这一维度往往决定 Agent 能否上线。

### 7.3.7 可恢复性

对于长任务和 Runtime 型 Agent，还应评估：

- 失败后能否重试
- 是否支持 checkpoint 恢复
- 人工介入后能否继续
- 中断后是否会丢失状态

这是普通模型评测里很少单独考虑，但在 Agent 中非常关键的一项。

## 7.4 任务完成率之外还要看什么

很多团队一开始只看“任务完成率”，但这很容易误导。

一个更完整的视角通常包括下面几类指标。

### 7.4.1 成功率

任务是否最终成功完成。  
这是最直观的顶层指标。

### 7.4.2 过程质量

Agent 是否走了合理路径，例如：

- 步数是否过长
- 是否绕远路
- 是否重复调用同一工具
- 是否频繁在无效步骤里打转

### 7.4.3 恢复能力

遇到错误时，是直接失败，还是能够：

- 重试
- 换工具
- 改参数
- 请求人工帮助

恢复能力决定 Agent 的实际鲁棒性。

### 7.4.4 资源效率

即使成功，也要看它用了多少资源：

- token
- 时间
- API 次数
- 工具次数
- 并发占用

工程上，效率通常和质量同样重要。

### 7.4.5 风险暴露

评测不能只看“完成”，还要看“有没有产生不该产生的行为”。  
例如：

- 是否越权
- 是否写错路径
- 是否暴露敏感数据
- 是否误触发外部动作

## 7.5 轨迹评测与结果评测

Agent Eval 通常可以分成两大类：结果评测和轨迹评测。

### 7.5.1 结果评测

结果评测关心最终产出是否达标，例如：

- 最终答案是否正确
- 文件是否修改正确
- 工单是否成功创建
- 测试是否通过

它的优点是直观、容易自动化；缺点是会忽略过程问题。

### 7.5.2 轨迹评测

轨迹评测关心 Agent 是怎么走到结果的，例如：

- 是否选对工具
- 是否在正确时机调用工具
- 是否重复无效步骤
- 是否发生回环
- 是否正确处理异常

轨迹评测对诊断问题尤其有价值，因为它能直接告诉你：

**Agent 到底是在哪一步坏掉的。**

### 7.5.3 为什么两者都要有

只看结果，容易放过高风险和高成本路径。  
只看轨迹，又可能过度惩罚那些“路径不优但结果可接受”的情况。

更合理的做法通常是：

- 结果评测负责判断“是否完成”
- 轨迹评测负责判断“是否以合理方式完成”

## 7.6 离线评测、在线评测与人工评审

成熟的 Agent Eval 通常不是一种方法，而是多层组合。

### 7.6.1 离线评测

离线评测通常使用固定任务集和固定环境回放，优点是：

- 可复现
- 成本可控
- 便于版本对比
- 适合回归测试

典型适用场景：

- prompt 更新后回归
- tool schema 调整后检查退化
- Runtime 改版前后对比

### 7.6.2 在线评测

在线评测发生在真实流量或仿真流量中，更接近生产表现。  
它更适合观察：

- 真实用户任务成功率
- 长尾场景
- 环境扰动下的稳定性
- 真实成本与时延

但在线评测的问题也很明显：

- 噪音更大
- 很难完全复现
- 风险更高

### 7.6.3 人工评审

很多 Agent 问题最终还是需要人工判断，尤其是：

- 路径是否合理
- 输出是否真正可用
- 风险是否可接受
- 工具调用是否体现了正确策略

人工评审不一定要覆盖全部样本，但通常应该覆盖：

- 高价值任务
- 高风险任务
- 自动评分不可靠的任务
- 线上失败案例

### 7.6.4 三者的关系

一套实用方案通常是：

- 离线评测做日常回归
- 在线评测看真实效果
- 人工评审做高价值抽检与失败复盘

## 7.7 一套简单可落地的 Agent Eval 方法

如果团队刚开始做 Agent Eval，不需要先追求复杂框架，先把一套基础流程跑起来更重要。

### 7.7.1 定义任务集

先挑选一批具有代表性的任务，覆盖：

- 简单任务
- 中等复杂任务
- 长链路任务
- 容易失败的边界任务
- 高风险任务

任务集不需要一开始很大，但必须有代表性。

### 7.7.2 定义成功标准

每个任务都应该提前定义：

- 什么叫成功
- 什么叫失败
- 什么叫部分成功
- 哪些行为必须判负

如果标准不清，评测结果会变成主观讨论。

### 7.7.3 记录完整轨迹

不要只存最终输出，要记录：

- 输入任务
- 模型每轮决策
- 工具调用
- 工具返回
- 状态变化
- 最终结果
- token 和时延

没有轨迹，基本无法做有效诊断。

### 7.7.4 建立多维评分

一套简单评分表可以包括：

- `success`
- `correctness`
- `tool_use`
- `efficiency`
- `safety`
- `recovery`

哪怕先用人工打标签，也比只有一个“成功/失败”强得多。

### 7.7.5 做版本对比

每次改动后，不只看绝对分数，还要看：

- 哪类任务提升了
- 哪类任务退化了
- 成本是升还是降
- 失败类型有没有变化

这比只看单次总成功率更有决策价值。

### 7.7.6 建立失败样本库

把典型失败案例沉淀下来，按类型归档，例如：

- 无限循环
- 误调用工具
- 错误恢复失败
- 结果正确但过程违规
- 安全护栏缺失

一个成熟团队的 Agent 优化，往往不是靠“拍脑袋改 Prompt”，而是靠失败样本驱动。

## 7.8 一套更适合企业场景的指标框架

如果是企业环境，可以把指标分成四层：

### 7.8.1 Outcome

看任务结果：

- 成功率
- 正确率
- 用户满意度

### 7.8.2 Process

看执行过程：

- 平均步数
- 工具调用正确率
- 回环率
- 恢复率

### 7.8.3 Efficiency

看效率：

- token 消耗
- 平均耗时
- 单任务成本

### 7.8.4 Governance

看治理：

- 越权率
- 审批命中率
- 风险动作拦截率
- 敏感信息暴露率

这样做的好处是，不会让“高成功率”掩盖“高风险、高成本”的问题。

## 7.9 总结

Agent 评测的本质，不是只给模型打一个分，而是评估一个执行系统是否可靠。

和普通模型评测相比，Agent Eval 更关注：

- 最终结果是否完成
- 执行过程是否合理
- 工具使用是否正确
- 成本和时延是否可接受
- 安全和治理是否合规
- 出错后能否恢复

可以用一句话概括：

**普通模型评测更像“判答案”，Agent 评测更像“判一个任务执行系统是否值得上线”。**

真正有用的 Agent Eval，通常不是单个 benchmark，而是一套持续运行的评测体系：

- 用离线评测做回归
- 用在线评测看真实表现
- 用人工评审做高价值判断
- 用失败样本库持续驱动优化
