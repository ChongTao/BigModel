# 一 RAG 是什么

RAG（Retrieval-Augmented Generation，检索增强生成）是一种把 **外部知识检索** 和 **大模型生成** 组合起来的方案。它的目标不是让模型只依赖参数记忆回答问题，而是在回答前先从知识库、文档库、API 或数据库中找到相关信息，再把这些信息作为上下文交给模型生成答案。

RAG 通常包含四个核心环节：

1. 查询理解：理解用户问题、任务目标和约束条件。
2. 检索：从外部知识源中找出与问题最相关的候选内容。
3. 增强：将候选内容筛选、压缩、组织后拼接到上下文中。
4. 生成：大模型结合检索结果生成最终回答。

> 除了以上四步，RAG 还依赖一个基础能力：Embedding 编码。它负责把文本转成向量，供向量检索、相似度计算和语义匹配使用。

典型价值：

- 实时性：可以接入最新文档和外部数据，不受训练截止时间限制。
- 准确性：让回答建立在可检索证据上，降低纯生成带来的幻觉。
- 可解释性：可以返回引用片段、来源文档或检索证据。
- 轻量化：相比重新训练或微调全量模型，更新知识库的成本通常更低。

RAG 的核心指标通常分成两类：

- 检索侧：召回率、Precision@K、MRR、nDCG。
- 生成侧：答案准确性、相关性、忠实度（faithfulness）、引用有效性。

> 在大语言模型应用中，RAG 主要用于缓解知识滞后、事实性不足和专业领域知识覆盖不够的问题，但它并不能自动解决所有推理问题，系统质量仍然取决于数据、检索和提示设计。

# 二 RAG 基本链路

从工程实现看，一个最常见的 RAG 链路如下：

1. 文档预处理：清洗文本、提取元数据、切分 chunk。
2. 建索引：为 chunk 建立倒排索引、向量索引或混合索引。
3. 查询预处理：对 query 做改写、扩展、过滤或路由。
4. 初召回：使用关键词检索、向量检索或混合检索取回 Top-K。
5. 重排与压缩：对候选结果做 rerank、去重和上下文压缩。
6. 构造 Prompt：把问题、约束、检索结果和格式要求拼进提示词。
7. 生成回答：调用模型输出结果，并可附带引用或证据。

如果你需要看不同 RAG 范式的分类，可以继续阅读 [2 RAG架构.md](./2%20RAG架构.md)。

# 三 文档切分 Chunking

文档切分的目标有两个：

1. 控制上下文窗口和索引粒度。
2. 在“检索精度”和“上下文完整性”之间做平衡。

文本分块示意：

![](https://www.dailydoseofds.com/content/images/2024/11/chunking-rag.gif)

## 3.1 固定大小分块

按照固定字符数或 token 数切分文本，例如 `chunk_size=300`。

- 优点：实现简单、处理快、适合结构弱的文本。
- 缺点：容易切断语义边界，导致上下文丢失。
- 适用：FAQ、短文档、格式不稳定的原始文本。

## 3.2 基于句子的分块

先做句子切分，再把一个或多个句子组合成 chunk。

- 优点：语义边界通常比固定长度更自然。
- 缺点：句子长度不均匀；中英文混排、表格、代码等内容切分效果不稳定。
- 适用：新闻、说明文、结构较清晰的自然语言文本。

## 3.3 递归字符分块

递归字符分块是 LangChain 中常见的实现方式。其核心思想是：优先按段落、句子等高层级分隔符切分，仍然过长时再递归使用更细粒度的分隔符。

- 优点：兼顾实现成本与语义完整性，通用性强。
- 缺点：效果依赖分隔符优先级和参数设置。
- 适用：大多数通用 RAG 系统的默认策略。

## 3.4 基于文档结构的分块

按照标题、章节、列表、表格、代码块等结构边界切分文档。

- 优点：更符合原始文档的逻辑层次，利于保留上下文。
- 缺点：依赖格式解析能力，预处理复杂。
- 适用：Markdown、HTML、PDF 转结构化文本后的文档。

## 3.5 混合分块

先按文档结构粗切，再对大块内容用递归字符或句子级方式细切。

- 优点：同时兼顾结构信息和块大小控制。
- 缺点：策略设计复杂，参数更多。
- 适用：企业知识库、产品文档、手册类场景。

## 3.6 语义分块

根据句向量相似度、主题变化或语义边界进行切分，使每个 chunk 尽量表示一个完整语义单元。

- 优点：chunk 内部语义聚合度高。
- 缺点：计算成本高，效果依赖模型质量。
- 适用：高精度问答、复杂专业文档。

## 3.7 分层分块

同时构建大块、中块、小块多个层级，并在不同层级上建立索引。

- 优点：可以在不同粒度上灵活检索和补充上下文。
- 缺点：索引和存储成本上升，实现更复杂。
- 适用：书籍、论文、长篇报告等层次清晰的长文档。

## 3.8 Parent Document / Small-to-Big

先用小块提高召回精度，再映射回父文档或更大块内容供生成阶段使用。

- 优点：兼顾“小块易召回”和“大块上下文完整”。
- 缺点：需要维护子块和父块映射关系。
- 适用：长文档问答、手册检索、章节级阅读理解。

## 3.9 命题分块

命题分块（Propositional Chunking）把文本切分为“完整事实”或“独立陈述”，例如一个主谓宾结构的事实单元。

- 优点：适合精确事实检索和知识抽取。
- 缺点：高度依赖 LLM 或 NLP 抽取质量。
- 适用：知识图谱构建、事实问答、结构化知识沉淀。

## 3.10 Agentic / LLM-based Chunking

使用大模型根据文档内容动态判断切分边界，而不是固定依赖长度或规则。

- 优点：理论上最贴近语义和任务目标。
- 缺点：成本高、稳定性和可重复性较差，实现复杂。
- 适用：高价值、低吞吐的精细化知识处理流程。

> Chunking 没有绝对最优解。设计重点通常不是“块越大越好”或“越小越好”，而是要和检索方式、问题类型、上下文长度、重排能力一起设计。

# 四 索引与检索

## 4.1 Embedding 检索

Embedding 检索会先把文本编码为稠密向量，再通过向量相似度找到语义最接近的内容。更多内容可见 [3 向量检索.md](./3%20向量检索.md)。

常见流程：

1. 使用 Embedding 模型把文档和查询编码为向量。
2. 将文档向量存入向量数据库或向量索引。
3. 查询时计算相似度，召回 Top-K 候选结果。

优点：

- 能处理同义表达和语义相近但字面不同的问题。
- 对自然语言问答通常比纯关键词检索更友好。

局限：

- 对精确术语、数字、缩写、实体名的匹配不一定稳定。
- 很难单靠向量相似度解决多跳推理和复杂约束问题。

## 4.2 稀疏检索与倒排召回

倒排索引（Inverted Index）是最常见的稀疏检索结构。系统会先对文档做分词，建立 term 到文档的映射；查询时再基于 BM25 等算法进行排序。

优点：

- 对关键词、专有名词、数字、精确匹配更稳定。
- 实现成熟，吞吐高，适合大规模检索。

局限：

- 对语义改写、同义表达和隐式语义理解能力较弱。

## 4.3 混合检索

混合检索通常把稀疏检索和稠密检索结合起来，先各自召回，再做结果融合。

常见融合方式：

1. RRF（Reciprocal Rank Fusion）：按各路结果的排名做融合，简单、稳定、鲁棒。
2. 加权融合：对 BM25 分数、向量相似度分数做归一化后加权。
3. 路由检索：根据 query 类型决定优先使用哪类检索器。

混合检索的价值在于：同时利用关键词匹配能力和语义匹配能力，通常是企业 RAG 的默认选择。

## 4.4 检索增强策略

为了提升初始召回质量，常见还会加入以下增强策略：

- Query Rewrite：把口语化问题改写成更适合检索的表达。
- Multi-Query：从一个问题生成多个子查询并行检索。
- Metadata Filter：按时间、作者、部门、文档类型等元数据过滤。
- HyDE：先生成“假设答案”或“假设文档”，再据此做向量检索。
- Reverse HyDE：从文档反向生成假设查询，提升文档与 query 的匹配机会。

# 五 重排与上下文构造

初召回拿到的文档“可能相关”，但不一定“足够回答问题”。因此多数 RAG 系统需要在召回后再做重排、去噪和压缩。

## 5.1 为什么需要重排

常见问题包括：

- 召回结果看起来相似，但并不回答当前问题。
- 候选文档之间有大量重复片段。
- 真正有用的信息排位靠后，未能进入最终上下文。

![](https://www.dailydoseofds.com/content/images/2025/02/image-3.png)

## 5.2 Cross-Encoder Reranker

Cross-Encoder 会把 Query 和候选文档拼接后一起输入模型，直接输出相关性分数。

- 优点：精度高，适合做 Top-K 精排。
- 缺点：速度慢，无法像双塔模型那样高效大规模召回。
- 适用：召回后的二阶段精排。

## 5.3 ColBERT Reranker

ColBERT 保留 query 和文档中每个 token 的向量，在检索或重排阶段做 Late Interaction（后期交互）。

- 优点：通常比双塔检索更精细，比 Cross-Encoder 更容易兼顾效率。
- 缺点：实现和存储成本更高。
- 适用：需要较高语义精度的检索与精排场景。

## 5.4 Graph-Based Reranking

把候选文档、段落或句子构造成图，再利用 PageRank、HITS、GNN 等方法综合考虑相关性和整体一致性。

- 优点：适合强调关系结构和全局一致性的任务。
- 缺点：复杂度高，工程实现成本大。
- 适用：多跳推理、图谱增强、复杂知识问答。

## 5.5 上下文压缩

重排之后，通常还会做上下文压缩，包括：

- 去重：移除重复段落和近似片段。
- 摘要：把长段落压缩成回答问题所需的关键信息。
- 结构化提取：只保留结论、定义、时间、实体、步骤等核心字段。

上下文构造的目标不是“塞入越多文本越好”，而是“让最终上下文尽量短、尽量准、尽量可回答”。

# 六 常见问题与设计权衡

## 6.1 为什么需要 RAG

如果只依赖模型参数，通常会面临以下问题：

- 知识滞后：模型无法天然知道训练后出现的新信息。
- 幻觉风险：模型可能生成看似合理但缺乏证据的答案。
- 专业领域局限：垂直领域知识覆盖不足，细节不可靠。
- 企业私有知识不可见：内部文档、流程和数据不在公开训练语料中。

RAG 的价值是把“模型能力”与“外部知识”解耦，让知识可以单独更新和维护。

## 6.2 检索结果是否真的有用

检索命中并不等于问题可回答。一个结果可能在语义上接近 query，但缺少真正回答问题所需的事实、约束或上下文。

因此评估 RAG 时不能只看“检索到了没有”，还要看：

- 召回结果是否包含支持答案的证据。
- 排名前几条是否已经足够回答问题。
- 最终答案是否忠实于检索证据。

## 6.3 分块多大才合适

块太小，容易丢失关键上下文；块太大，又会引入大量无关信息，稀释生成效果。

![](https://www.dailydoseofds.com/content/images/2025/02/image-2.png)

经验上可以这样理解：

- 事实型问答：倾向更小、更聚焦的 chunk。
- 解释型问答：需要更完整的上下文，可适当增大 chunk。
- 长文档检索：常结合分层分块或 Parent Document 策略。

## 6.4 RAG 不是万能解法

RAG 更擅长“找信息并基于证据回答”，但它并不天然擅长：

- 复杂规划和长链路决策。
- 没有外部证据时的纯推理问题。
- 依赖强操作反馈的任务执行。

这类问题通常需要继续引入工作流编排、工具调用或 [10 Agentic RAG.md](./10%20Agentic%20RAG.md) 中的 Agent 能力。
