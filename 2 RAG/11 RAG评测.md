# 十一 RAG 评测

RAG 评测不能只看最终答案。  
因为一次回答失败，可能出在：

- 数据接入与清洗
- chunk 切分
- 索引构建
- 初召回
- 重排
- 上下文构造
- 生成模型没有正确使用证据

所以，RAG Eval 的本质不是只评“回答对不对”，而是评估：

**整条“检索到生成”的知识利用链路是否可靠。**

## 11.1 为什么 RAG 评测比普通问答更复杂

普通问答模型的评测，很多时候是：

`Question -> Answer -> 对比标准答案`

但 RAG 系统更像：

`Question -> Retrieval -> Ranking -> Context -> Generation -> Answer`

这意味着它比普通问答多了几个复杂性来源。

### 11.1.1 错误来源更多

回答错误时，原因可能完全不同：

- 没检索到正确文档
- 检索到了但没排到前面
- 排到前面了但没进入最终上下文
- 进入上下文了但模型没正确使用
- 模型用到了错误证据或自行补全

如果只看最终答案，很难判断问题到底出在哪一层。

### 11.1.2 成功不代表链路健康

有些答案虽然看起来正确，但其实：

- 模型没依赖检索结果，凭参数知识答对了
- 检索结果里证据不足，模型靠猜补全
- 引用与答案并不严格对应

对于 RAG 来说，这种“结果碰巧对了”不算理想成功。

### 11.1.3 评测目标不止一个

RAG 至少要同时评估几个问题：

- 是否找到真正相关的信息
- 是否把最有用的证据放进上下文
- 生成结果是否忠实于证据
- 系统是否在延迟、成本和稳定性上可接受

因此，RAG 评测天然是分层评测，而不是单点打分。

## 11.2 RAG 评测要分哪些层

一个比较完整的 RAG Eval，通常至少要分成三层。

### 11.2.1 检索层

检索层关注：

**正确证据有没有被找回来。**

这一层通常不关心最终语言表达，而关心召回和排序质量。

### 11.2.2 生成层

生成层关注：

**模型有没有基于检索结果生成正确、忠实、可用的回答。**

这一层最关键的是 grounding，而不是单纯语言流畅。

### 11.2.3 端到端层

端到端层关注：

**整个系统作为产品能力是否可用。**

这一层通常会同时评估：

- 最终任务完成率
- 用户体验
- 成本
- 延迟
- 稳定性

如果不分层，很多问题会被混在一起，导致优化方向失真。

## 11.3 检索层指标

检索层的核心问题是：系统有没有把“应该用于回答问题的证据”找回来。

### 11.3.1 Recall@K

Recall@K 回答的是：

**前 K 个候选中，是否包含正确证据。**

它主要用于判断有没有漏召回。  
如果 Recall@K 很低，说明系统还没到“精排”阶段，先要解决召回问题。

### 11.3.2 Precision@K

Precision@K 回答的是：

**前 K 个结果里，真正相关的比例高不高。**

它反映的是噪音水平。  
如果 Precision 很低，后续 reranker 和生成阶段会被大量无关内容干扰。

### 11.3.3 MRR

MRR（Mean Reciprocal Rank）更关注：

**第一个正确结果排得靠不靠前。**

它很适合评估“用户最早能不能看到关键证据”这一问题。

### 11.3.4 nDCG

nDCG 更适合处理“相关性不是简单二元”的情况。  
例如：

- 某些文档高度相关
- 某些文档部分相关
- 某些文档只是背景材料

这比简单判断“对/错”更贴近真实检索质量。

### 11.3.5 检索层不止评初召回

实际系统里，检索层至少应拆成几个节点来评：

- 初始召回结果
- 融合后结果
- 重排后结果
- 最终进入 prompt 的上下文集合

因为很多问题并不是“根本没召回”，而是：

- 召回了但没排上来
- 排上来了但没被选进上下文

## 11.4 生成层指标

生成层的重点不是“文风好不好”，而是“回答是否真正基于证据”。

### 11.4.1 正确性

回答是否解决了用户问题，结论是否正确，是否存在明显错误或遗漏。

### 11.4.2 忠实性

忠实性关注：

**答案是否忠实于检索证据，而不是模型自己编出来。**

这是 RAG 评测里最关键的维度之一。  
因为 RAG 的目标本来就是“基于外部证据回答”。

### 11.4.3 完整性

回答即使不错误，也可能不完整。  
例如只覆盖了部分步骤、部分条件、部分限制。

因此生成层还应关注：

- 是否覆盖关键要点
- 是否遗漏必要约束
- 是否只回答了表面问题

### 11.4.4 引用质量

如果系统要求返回引用，还要额外评估：

- 引用是否真实支撑答案
- 是否存在错引
- 是否漏引
- 是否引用了与答案无关的段落

很多“看起来带引用”的答案，实际上引用和结论并不严格对应。

## 11.5 端到端指标

当 RAG 进入产品环境后，只看离线效果分已经不够。

端到端评测关注的是：

**系统是否真的可交付、可运营。**

常见指标包括：

- 任务完成率
- 用户满意度
- 首响应延迟
- 完整响应时间
- 每次请求成本
- 错误率
- 超时率
- 高峰期稳定性

一个检索质量更好的方案，如果把延迟从 1 秒拉到 8 秒，把成本提高数倍，在很多业务场景下并不一定值得。

所以端到端评测的价值在于：  
防止团队只盯着离线分数，却忽略整体系统表现。

## 11.6 人工评测与自动评测

RAG Eval 很少能完全依赖单一方式，通常要把自动评测和人工评测结合起来。

### 11.6.1 自动评测

自动评测的优点是：

- 可批量运行
- 成本低
- 适合版本对比
- 适合回归测试

它适合做：

- Recall@K / Precision@K / MRR / nDCG
- 日常回归
- 参数调优对比
- 线上指标持续监控

但局限也明显：

- 很难判断复杂回答是否真的有帮助
- 很难识别微妙幻觉
- 很难判断证据是否真正足够

### 11.6.2 人工评测

人工评测更适合判断：

- 回答是否真正解决问题
- 引用是否合理
- 多跳问题是否解释充分
- 回答是否存在隐蔽幻觉

缺点是：

- 成本高
- 速度慢
- 标准容易漂移

### 11.6.3 更实用的组合方式

比较实用的做法通常是：

- 用自动评测做日常回归
- 用人工评测做关键版本确认
- 用人工抽检覆盖高价值场景和疑难案例

## 11.7 构建 RAG 测试集的方法

RAG 测试集的质量，往往比指标本身更重要。  
如果测试集设计得过于理想化，评测结果很容易高估系统能力。

### 11.7.1 问题类型要覆盖

不要只收集简单事实题。一个更有价值的测试集，通常应覆盖：

- 精确实体问答
- 长文档定位
- 多文档综合
- 多跳推理
- 带时间变化的问题
- 带版本变化的问题
- 含噪音、歧义或模糊表达的问题

### 11.7.2 标准答案之外，还要标标准证据

除了标准答案，最好还能标出：

- 正确证据来自哪些文档
- 哪些 chunk 应被召回
- 哪些证据足以支撑回答

否则很难区分问题到底出在检索还是生成。

### 11.7.3 尽量来自真实业务查询

如果测试集全靠人工想象，往往会过于“干净”和“标准化”。  
真实用户的问题通常更口语、更模糊、更有噪音。

所以更可靠的做法是尽量从：

- 搜索日志
- 客服问题
- 工单历史
- 知识库使用记录

中抽取真实问题样本。

### 11.7.4 控制数据泄漏

如果评测问题、答案或高度相似文本本来就在模型训练数据中广泛出现，那么系统效果会被高估。  
因为模型可能不依赖检索也能答出来。

因此，评测时应尽量关注：

- 私有数据场景
- 最新文档场景
- 训练截止时间后的知识

## 11.8 一个可落地的 RAG Eval 流程

如果团队刚开始建立 RAG 评测体系，可以先从一套简单但完整的流程做起。

### 11.8.1 先分层看问题

不要一上来只看最终答案。  
先把评测拆成：

- 检索是否找回
- 重排是否排准
- 生成是否忠实
- 端到端是否可用

### 11.8.2 建立固定测试集

至少保留一套稳定测试集，用于：

- 版本回归
- 参数对比
- Prompt、chunking、reranker 调整前后比较

### 11.8.3 记录中间链路结果

要保存：

- query
- 召回结果
- 重排结果
- 最终上下文
- 模型回答
- 引用
- token 与延迟

否则出了问题很难复盘。

### 11.8.4 把离线和线上分开看

离线评测适合比较方案，线上评测适合观察真实效果。  
两者不能互相替代。

### 11.8.5 建失败样本库

把典型失败案例沉淀下来，按类型归档，例如：

- 没召回
- 召回了但没排上来
- 排上来了但没进上下文
- 模型忽略证据
- 引用错误
- 回答不忠实

这类失败样本库通常比单个总分更能驱动系统优化。

## 11.9 常见误区

RAG 评测里有几个非常常见的误区。

### 11.9.1 只看最终答案

这会掩盖整条链路的问题。  
回答对了时，不知道是否真的用到了检索；回答错了时，也不知道错在哪一层。

### 11.9.2 只看 Recall，不看排序和上下文选择

召回到了不等于用上了。  
很多系统的问题其实出在排序和上下文构造，而不是初始检索。

### 11.9.3 把语言流畅误当成 grounding

答案写得自然，不代表它忠实于证据。  
对 RAG 来说，grounding 和 faithfulness 比文风更重要。

### 11.9.4 用过于简单的问题做评测

如果测试集几乎全是单跳事实题，系统可能会被高估。  
一旦进入长文档、多文档、复杂问题场景，性能往往会明显下降。

### 11.9.5 只做离线评测，不看线上表现

离线分数提升，不一定代表线上用户体验提升。  
线上环境还会受到：

- 延迟
- 成本
- 用户提问噪音
- 数据更新
- 流量分布变化

等因素影响。

## 11.10 总结

RAG 评测的核心，不是简单判断“答案对不对”，而是分层判断整条知识利用链路是否可靠：

1. 相关信息有没有被找到
2. 找到的信息有没有被正确排序
3. 模型有没有忠实使用这些信息
4. 整个系统是否在成本、延迟和稳定性上可接受

可以用一句话概括：

**RAG Eval 不是只评答案，而是评“检索到生成”整条系统链路是否可信、可控、可上线。**
