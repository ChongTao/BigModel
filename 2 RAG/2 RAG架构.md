# 一 RAG 架构分类

如果说 [1 RAG概述.md](./1%20RAG概述.md) 关注的是 RAG 链路中的关键组件，那么本文关注的是另一件事：**这些组件是如何被组织成不同架构范式的**。

常见的 RAG 架构可以粗略分为五类：

1. Naive RAG：最基础的索引-检索-生成链路。
2. Advanced RAG：在检索、重排、压缩、生成等环节做系统优化。
3. Modular RAG：把各环节模块化、可插拔化。
4. Graph RAG：把知识组织为图结构，强调关系检索和多跳推理。
5. Agentic RAG：引入 Agent 的规划、决策和动态执行能力。

它们不是严格互斥的关系，而更像是逐步增强的能力层级。一个真实系统往往会同时具备多种特征，例如“Modular + Advanced”或“Graph + Agentic”。

# 二 Naive RAG

Naive RAG 是最基础的检索增强生成架构，采用“索引-检索-生成”的经典流程。

![naive](https://miro.medium.com/v2/resize:fit:4800/format:webp/0*Ko_ihY8ecAukf2g1.png)

## 2.1 核心机制

典型流程通常包括 4 个步骤：

1. **索引阶段（Indexing）**：将知识库文档切分为 chunks，并通过 Embedding 模型编码为向量，存入向量数据库或检索引擎。
2. **检索阶段（Retrieval）**：将用户 query 编码后，在索引中召回最相似的文档片段。
3. **提示构造（Prompt Construction）**：把召回结果拼接进 Prompt。
4. **生成阶段（Generation）**：将完整 Prompt 输入模型，生成最终回答。

## 2.2 适用场景

- FAQ 查询
- 企业产品手册检索
- 简单知识问答
- 基础客服场景

## 2.3 优势

- 架构简单，容易落地。
- 工程成本低，适合快速验证。
- 对事实型、单跳问题通常已经有效。

## 2.4 局限

- 召回噪声较大，结果质量高度依赖初召回。
- 上下文容易碎片化，缺乏更精细的筛选与压缩。
- 对复杂语义、多跳推理、约束型问题支持较弱。

## 2.5 与下一阶段的差异

Naive RAG 的核心问题不是“不能检索”，而是“检索到了也未必好用”。Advanced RAG 就是在这个基础上，系统性优化检索前、检索中和检索后的各个环节。

# 三 Advanced RAG

Advanced RAG 是在 Naive RAG 基础上的增强型方案。它并不改变 RAG 的基本链路，而是针对召回质量、上下文质量和生成质量做系统优化。

![advanced](https://miro.medium.com/v2/resize:fit:2000/format:webp/0*Gr_JqzdpHu7enWG9.png)

## 3.1 核心机制

优化通常集中在三个层面：

1. 检索层：改进检索算法、增强语义匹配、融合多种召回策略。
2. 知识选择层：做重排、去重、压缩，提升上下文质量。
3. 生成层：加入反思、验证、约束输出等机制。

常见增强策略：

- 检索前：查询改写、子查询分解、元数据过滤、Query Routing
- 检索中：混合检索、图数据库增强、多路召回
- 检索后：重排序、上下文压缩、引用筛选、答案验证

## 3.2 适用场景

- 法律合同问答
- 医疗知识辅助检索
- 金融、风控、审计等高准确率场景
- 专业知识密集型问答系统

## 3.3 优势

- 解决 Naive RAG 中“召回有了但不够准”的问题。
- 更适合复杂语义对齐和高精度问答。
- 可以显著提升最终答案的可用性和稳定性。

## 3.4 局限

- 系统链路更长，调参复杂度更高。
- 组件增多后，诊断问题和评估成本更高。
- 仍然主要是“增强检索”，不是“自主规划”。

## 3.5 与下一阶段的差异

Advanced RAG 关注的是“把每个环节做得更好”，而 Modular RAG 更关注“把整个系统设计成可替换、可扩展、可组合的架构”。

# 四 Modular RAG

**Modular RAG（模块化检索增强生成）** 是一种组件化、可插拔、可组合的架构设计思想。它不是某个单一算法，而是一种系统工程范式。

> 核心理念：把 RAG 的各个环节拆分成独立模块，例如检索器、重排器、压缩器、路由器、生成器、验证器，再根据业务场景自由组合。

![](https://miro.medium.com/v2/resize:fit:1100/format:webp/1*C1bMxjC5g0SDGSGOZocyuA.png)

## 4.1 核心机制

模块化架构通常具备以下特点：

- 检索器可替换：BM25、向量检索、混合检索、图检索可以并存。
- 重排器可替换：Cross-Encoder、ColBERT、规则打分可以按需切换。
- 路由能力：不同 query 可以进入不同的处理链路。
- 工作流可编排：根据业务需求接入过滤、验证、缓存、引用生成等模块。

## 4.2 适用场景

- 企业级知识管理平台
- 多源异构数据整合场景
- 需要按业务线定制检索链路的系统
- 长期演进、多人协作维护的 RAG 平台

## 4.3 优势

- 组件解耦，便于维护和扩展。
- 可以针对不同场景灵活组合检索、重排、压缩和生成能力。
- 更适合中大型系统持续迭代。

## 4.4 局限

- 初始设计成本更高，需要更强的系统抽象能力。
- 模块边界和接口设计不合理时，容易增加整体复杂度。
- 模块越多，越需要完善的评估和观测体系。

## 4.5 与下一阶段的差异

Modular RAG 的重点仍然是“模块怎么组织”，Graph RAG 则把重点放在“知识本身如何组织”，即从文档片段检索转向关系结构检索。

# 五 Graph RAG

**Graph RAG（Graph-based Retrieval-Augmented Generation）** 是一种在传统 RAG 基础上，引入知识图谱或语义图的增强型架构。

> 核心思想：把知识组织为图结构，由节点表示实体、概念或事件，由边表示关系，从而让系统不只基于文本相似度检索，还能基于关系与结构检索。

![](https://graphrag.com/_astro/graphrag-diagram.gzIXlJ0V_Z1168rD.svg)

## 5.1 核心机制

Graph RAG 通常包含以下关键环节：

1. 图构建：从文档中抽取实体、关系、事件，构建知识图谱或语义图。
2. 图存储：将节点、边及其属性存入图数据库或图索引。
3. 图检索：根据 query 检索相关节点、关系或子图。
4. 上下文生成：把子图信息转写为适合模型输入的上下文。

## 5.2 适用场景

- 多跳推理问题
- 实体关系密集的知识问答
- 因果分析、依赖分析、事件链分析
- 需要全局一致性和关系解释的场景

## 5.3 优势

- 显式表示实体和关系，结构化程度更高。
- 更适合处理跨文档关联、关系推理和全局一致性问题。
- 通常比纯向量检索更容易解释“为什么召回这些内容”。

## 5.4 局限

- 图构建成本高，依赖信息抽取质量。
- 数据更新和图维护比普通文档索引复杂得多。
- 如果问题本身不依赖关系结构，图方案未必比普通 RAG 更划算。

## 5.5 与下一阶段的差异

Graph RAG 解决的是“如何利用结构化关系更好检索”，而 Agentic RAG 解决的是“系统如何自主规划、迭代检索和调用工具完成任务”。

# 六 Agentic RAG

Agentic RAG 是在传统 RAG 框架上加入 **自主决策（Agent Reasoning）** 和 **动态执行（Action Execution）** 能力的系统。一些综述资料，例如 `Agentic Retrieval-Augmented Generation: A Survey on Agentic RAG`，会把它视为 RAG 从“静态问答链路”演进到“动态任务工作流”的代表方向。

> 可以粗略理解为：传统 RAG 更像一次性的检索增强问答，而 Agentic RAG 更像一个会思考“下一步该查什么、要不要继续查、是否需要拆解问题或调用工具”的智能体系统。

![](https://substackcdn.com/image/fetch/f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F65bf97c3-6f3d-4fd8-810a-b335f8795239_1160x796.gif)

## 6.1 核心机制

Agentic RAG 往往具备这些特征：

- 自主决策：根据 query 复杂度选择检索策略和工具。
- 任务分解：把复杂问题拆成多个子问题逐步处理。
- 迭代改进：根据中间结果持续改写 query、补充检索或修正方向。
- 工具调用：结合搜索、数据库、API、代码执行等能力完成任务。
- 结果验证：对答案进行校验、反思或多轮确认。

它的出发点不是简单替换检索器，而是把“会做决策、会反思、会调用工具、会协作”的 Agent 嵌入 RAG 流程，让检索与生成从静态流水线变成可迭代、可路由、可规划的动态工作流。

## 6.2 设计模式

Agentic RAG 常见有四类基础设计模式：

### 6.2.1 Reflection

反思（Reflection）让系统能够迭代地评估和改进自己的中间结果或最终输出。

- 典型做法：让模型检查答案是否正确、是否遗漏证据、是否满足格式要求。
- 常见增强：结合单元测试、搜索、规则校验器等外部工具验证结果。
- 价值：降低错误传播，提升答案质量和稳定性。
- 风险：增加调用次数和延迟，如果反馈标准不清晰，也可能陷入无效循环。

### 6.2.2 Planning

规划（Planning）让系统把复杂任务分解为多个子任务，再决定执行顺序和依赖关系。

- 典型做法：先拆问题，再分别检索、汇总、验证。
- 价值：适合多跳推理、复杂调研和开放任务。
- 风险：规划链路越长，行为越难预测，出错点也越多。

### 6.2.3 Tool Use

工具使用（Tool Use）让系统把检索之外的外部能力纳入工作流，例如 API、数据库、代码执行、搜索和业务系统接口。

- 典型做法：一边检索文档，一边查询实时数据或调用计算工具。
- 价值：突破模型预训练知识边界，增强实时性和可操作性。
- 风险：权限控制、失败兜底和结果校验会变得更重要。

### 6.2.4 Multi-Agent

多代理协作（Multi-Agent Collaboration）通过多个角色分工协作来完成复杂任务。

- 典型做法：一个代理负责规划，一个代理负责检索，一个代理负责验证或批判。
- 价值：提升任务专业化程度，适合并行处理复杂问题。
- 风险：系统编排更复杂，代理之间的上下文同步和结果一致性更难控制。

## 6.3 适用场景

- 复杂业务分析
- 多步骤调研与决策支持
- 需要跨系统、跨工具获取信息的任务
- 高复杂度企业助手或研究型助手

## 6.4 与传统 RAG 的差异

与 Naive、Advanced、Modular、Graph 等更偏“检索系统设计”的架构不同，Agentic RAG 更强调运行时决策能力：

- 前者重点是把知识“找对、排好、喂给模型”。
- 后者重点是让系统知道“下一步该做什么”。

因此，Agentic RAG 的核心不只是回答质量提升，还包括任务分解能力、动态执行能力和失败恢复能力。

## 6.5 优势

- 能处理更复杂的任务分解和多步推理。
- 对动态场景和开放任务的适应性更强。
- 更容易把检索、工具、执行和验证整合成闭环。

## 6.6 局限

- 成本更高，延迟更长，系统行为更难预测。
- 需要更强的权限控制、观测能力和失败兜底。
- 如果问题本身简单，Agent 化往往是过度设计。

## 6.7 与传统 RAG 的差异总结

传统 RAG 更偏“静态检索增强”，而 Agentic RAG 更偏“动态知识利用与任务执行”。前者核心是提升回答质量，后者核心是提升任务完成能力。

# 七 如何选择合适的 RAG 架构

可以用一个简单的判断逻辑：

- 如果只是快速做知识问答原型，先用 Naive RAG。
- 如果已经上线并且主要问题是召回不准、上下文噪声大，用 Advanced RAG。
- 如果系统要长期演进、要支持多业务线，优先考虑 Modular RAG。
- 如果问题大量依赖实体关系、多跳推理和全局一致性，考虑 Graph RAG。
- 如果任务本质上是多步骤分析、规划、执行和验证，考虑 Agentic RAG。

实际工程中，更常见的不是单选，而是组合：

- `Advanced + Modular`：企业知识平台的常见形态。
- `Graph + Advanced`：关系密集型问答系统的常见形态。
- `Agentic + Modular`：复杂智能助手系统的常见形态。

# 八 总结

RAG 架构的演进，本质上是在回答三个问题：

1. 如何更好地检索到信息。
2. 如何更好地组织和筛选信息。
3. 如何让系统从“回答问题”走向“完成任务”。

Naive RAG 回答的是第一个问题的最小可行方案；Advanced 和 Modular RAG 让系统更准、更稳、更可扩展；Graph 和 Agentic RAG 则分别沿着“结构化知识”与“自主决策执行”两条路线继续扩展能力边界。
