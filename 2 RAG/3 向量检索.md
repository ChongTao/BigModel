# 一 向量检索在 RAG 中的作用

向量检索是 RAG 中最常见的召回方式之一。它的目标不是做通用的数学表征介绍，而是解决一个更具体的问题：**当用户的问题和原文表述不完全一致时，系统如何仍然召回相关内容。**

在 RAG 链路里，向量检索通常出现在以下位置：

1. 文档切分：把原始文档切成适合检索的 chunks。
2. 向量化：使用 Embedding 模型把 chunk 编码成向量。
3. 建索引：把向量写入向量数据库或检索引擎。
4. 查询编码：把用户 query 编码成查询向量。
5. 相似度召回：找到与 query 向量最接近的 Top-K 文档块。
6. 重排与压缩：对候选结果再做 rerank、去重和上下文压缩。
7. 生成回答：把高质量上下文交给大模型生成最终答案。

如果你想看 Embedding 本身的基础概念，可以继续阅读 [4 LLM/10 大模型原理/2 Embedding.md](../4%20LLM/10%20%E5%A4%A7%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%90%86/2%20Embedding.md)。

# 二 为什么 RAG 需要向量检索

传统关键词检索擅长处理“词面必须命中”的问题，但在下面这些场景里会明显吃力：

- 用户用了同义表达，和文档原句不一致。
- 问题是自然语言描述，而文档是术语化写法。
- 用户只描述意图，没有给出精确关键词。

向量检索的核心价值在于：把 query 和文档映射到同一个语义空间中，让系统即使在“字面不一样”的情况下，也能找到语义接近的内容。

典型收益：

- 提升自然语言问答场景下的召回率。
- 更适合 FAQ、知识库问答、企业搜索、RAG 助手。
- 能补足 BM25 在同义词、近义表达上的不足。

典型局限：

- 对专有名词、错误码、型号、精确数字不一定稳定。
- 单靠向量相似度无法天然解决多跳推理和复杂约束问题。
- 如果 chunk 切分不好，向量检索也很难救回来。

# 三 RAG 中的三类检索方式

## 3.1 稀疏检索

稀疏检索在 RAG 中通常指基于倒排索引的关键词召回，例如 TF-IDF、BM25。

它的特点是：

- 强项：专有名词、产品名、接口名、报错信息、数字、缩写。
- 弱项：同义改写、语义理解、自然语言问法。

在 RAG 实践中，稀疏检索经常承担第一阶段召回的一部分，因为它成本低、速度快、可解释性强。

## 3.2 稠密检索

稠密检索会先把 query 和文档都编码成稠密向量，再通过相似度搜索找到语义最接近的内容。

它的特点是：

- 强项：语义相近、措辞不同、自然语言表达。
- 弱项：精确词面匹配、可解释性、算力与索引成本。

在 RAG 中，稠密检索通常是“语义召回”的核心能力。

## 3.3 混合检索

混合检索会把稀疏检索和稠密检索结合起来，再对结果做融合。

在 RAG 中最常见的组合就是：

1. BM25 召回：覆盖关键词精确匹配强的内容。
2. 向量召回：覆盖语义相近但词面不同的内容。
3. 结果融合：使用 RRF、加权融合，或者交给 reranker 精排。

混合检索的价值在于：两种召回方式的误差模式不同，组合后通常比单一路径更稳。

# 四 向量数据库在 RAG 中做什么

向量数据库或支持向量索引的检索引擎，本质上是为了高效完成两件事：

1. 存储大量文档向量及其元数据。
2. 在查询时快速完成近似最近邻搜索。

![](https://www.dailydoseofds.com/content/images/2024/02/image-160.png)

如果只是少量数据，暴力比对所有向量也能工作；但在真实 RAG 系统中，知识库往往包含几十万到数百万个 chunks，逐条计算相似度的成本通常不可接受。

因此，向量数据库的核心价值不只是“存向量”，更是“在大规模场景下可接受地检索向量”。

常见能力包括：

- 向量索引：支持 ANN 检索。
- 元数据过滤：按时间、来源、部门、文档类型做过滤。
- 混合检索：同时支持关键词和向量能力。
- 多租户与权限控制：企业场景下常见。
- 结果返回：除了文本片段，还要返回来源、分数、元数据。

# 五 ANN 为什么重要

当知识库规模很大时，RAG 系统不会真的把 query 和每个向量都比一遍，而是用 ANN（Approximate Nearest Neighbor，近似最近邻）来缩小搜索空间。

![](https://www.dailydoseofds.com/content/images/2024/11/image.png)

ANN 的核心思想是：**不追求绝对精确的全量最近邻，而是在可接受误差下显著提升检索速度。**

![](https://www.dailydoseofds.com/content/images/size/w1000/2024/02/image-187.png)

对 RAG 来说，这种取舍通常是值得的，因为：

- 初召回本来就不是最终答案，还会经过 rerank。
- 召回速度会直接影响系统整体延迟。
- 在多数知识问答场景中，“足够好且足够快”比“理论最优但很慢”更有价值。

# 六 RAG 中的关键设计点

向量检索效果通常不只取决于模型本身，更取决于整条链路设计。

## 6.1 Chunk 切分

chunk 太小，可能语义不完整；chunk 太大，又会把无关信息一起带进来，稀释召回质量。

因此向量检索设计必须和 chunking 一起看，不能孤立优化。

## 6.2 Embedding 模型选择

Embedding 模型决定了 query 和文档被映射到什么语义空间。

选择时通常关注：

- 是否适合中文或多语言
- 是否适合检索而不是分类
- 向量维度与成本
- 是否支持领域数据微调

## 6.3 Top-K 与召回深度

Top-K 过小，容易漏召回；过大，则会把太多噪声送给后续 rerank 或生成阶段。

常见做法是：

- 初召回取较大的 K
- 交给 reranker 再缩成更小的上下文集合

## 6.4 元数据过滤

很多时候问题不是“没有召回”，而是“召回范围太大”。加入时间、权限、文档类型、部门等过滤条件，往往比单纯调相似度阈值更有效。

## 6.5 重排

向量检索负责找“可能相关”的内容，但真正送入 Prompt 的内容，通常还要再做重排。

因此在工程上，更合理的理解是：

- 向量检索解决第一阶段语义召回。
- Reranker 解决第二阶段精确排序。

# 七 在 RAG 中如何选择

可以用一个简单判断：

- 如果文档里有大量专有名词、错误码、接口名，不能只用向量检索。
- 如果用户提问更自然、更口语化，向量检索通常很重要。
- 如果系统要求高稳定性，优先考虑混合检索而不是单一路径。
- 如果知识库规模很大，向量数据库和 ANN 基本是必需项。

实际工程里，最常见的方案不是“纯向量检索”，而是：

`Chunking + Embedding + 向量召回 + BM25 召回 + Rerank + Prompt 构造`

# 八 总结

从 RAG 视角看，向量检索不是一个独立主题，而是召回链路中的核心组件。

它解决的是“如何找到语义相关内容”，但不单独保证最终答案质量。真正决定效果的，通常是这些因素共同作用：

- 文档切分质量
- Embedding 模型
- 向量索引与 ANN
- 元数据过滤
- 混合检索策略
- Rerank 和上下文构造

所以在 RAG 系统里，向量检索最好被理解成“语义召回基础设施”，而不是“只要上了 embedding 就能解决所有检索问题”。
