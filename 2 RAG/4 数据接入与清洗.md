# 四 数据接入与清洗

很多 RAG 项目一开始看起来是在“调模型”或“调检索”，但真正最容易卡住的地方，往往是数据接入与清洗。

原因很简单：

- 模型再强，也无法理解脏数据
- 检索再强，也召不回没有被正确入库的内容
- 重排再强，也很难修复源数据本身的结构缺陷

从工程视角看，RAG 的上限通常不是先被模型限制，而是先被数据质量限制。

## 4.1 RAG 为什么先卡在数据而不是模型

很多团队做 RAG 时，第一反应是：

- 换更好的 Embedding
- 换更强的 Reranker
- 换更大的 LLM

但如果底层数据有问题，这些优化通常收益有限。

常见问题包括：

- 文档根本没有被接入
- PDF 解析后顺序错乱
- 表格、标题、代码块被打散
- 元数据缺失，无法过滤
- 重复文档过多，污染召回
- 历史旧版本和新版本混在一起
- 文本切分前没有清洗，导致 chunk 质量很差

RAG 检索本质上是在“已有数据”里找内容。  
如果数据没有被正确抽取、清洗、切分和标注，那么后面的检索、重排和生成都会建立在不稳定地基之上。

## 4.2 数据接入阶段要处理什么

数据接入不是简单“把文件丢进向量库”，而是一条完整的数据管道。

通常要处理下面几类问题。

### 4.2.1 数据源识别

首先要明确知识从哪里来。常见来源包括：

- 本地文件系统
- 企业网盘
- Wiki / Confluence / Notion
- Git 仓库
- 数据库
- 工单系统
- FAQ / CRM / 工单评论
- 邮件、聊天记录、客服对话
- 外部网页和 API

不同来源的数据结构完全不同，接入方式也不同。

### 4.2.2 拉取与同步

接入阶段要解决的不是“一次导入”，而通常是“持续同步”。

常见同步方式包括：

- 全量导入
- 增量同步
- 定时批处理
- 事件驱动更新
- 手工触发重建

如果同步机制设计不好，就会出现：

- 知识库内容滞后
- 已删除内容仍被检索到
- 新旧版本同时存在
- 部分更新成功、部分更新失败

### 4.2.3 格式解析

接入后的第一步通常不是建索引，而是解析原始格式。

典型格式包括：

- PDF
- Word
- Excel
- Markdown
- HTML
- JSON / CSV
- 代码仓库文件
- 图片 OCR 文档

格式解析的目标不是“把它转成一串文本”，而是尽量保留原始结构信息，例如：

- 标题层级
- 段落
- 列表
- 表格
- 代码块
- 页面与章节关系

如果只得到一坨扁平文本，后续 chunking 和检索效果通常会显著变差。

### 4.2.4 权限与可见性

企业场景里，数据接入还必须考虑权限边界，例如：

- 哪些文档能被入库
- 哪些元数据可以暴露
- 哪些用户能检索哪些文档
- 是否需要按部门、项目、角色做隔离

如果接入阶段不处理权限，后续很容易出现“能检索到不该看到的文档”。

## 4.3 清洗阶段的关键问题

数据清洗的目标不是让文本“更好看”，而是让数据变得更可检索、更可切分、更可用于生成。

### 4.3.1 去除无效噪音

原始文档里经常包含大量对检索无帮助的内容，例如：

- 页眉页脚
- 页码
- 导航栏
- 版权声明
- 重复目录
- 模板化按钮文本
- OCR 产生的杂乱字符

这些噪音如果不清除，会直接污染 chunk，影响 Embedding 和召回质量。

### 4.3.2 保留语义结构

清洗不等于“把所有格式都抹平”。  
对 RAG 来说，更重要的是保留文档结构，例如：

- 一级标题、二级标题
- 段落边界
- 表格行列关系
- 列表项归属
- 代码块边界
- 引用块

结构一旦丢失，后续很难恢复。  
因此，清洗应该是“去噪但不去语义结构”。

### 4.3.3 统一编码与文本规范

不同来源的数据，常见会出现：

- 编码不一致
- 全角半角混乱
- 换行符不一致
- 空格、制表符异常
- 标点与数字格式不统一

这些问题单看很小，但在大规模索引时会不断累积，最终影响切分、去重、过滤和精确匹配。

### 4.3.4 版本与重复问题

企业知识库里，最常见的脏数据之一就是重复和过时版本：

- 同一份文档多个副本
- 新版旧版同时存在
- 导出文件和源文件都入库
- 相同段落被重复切成多份

如果不处理，会导致：

- 检索结果高度重复
- 新旧结论冲突
- Rerank 资源浪费在重复候选上
- 最终上下文被无效内容占满

### 4.3.5 结构化字段补全

很多数据在原始形态里并不适合直接检索。  
例如工单、FAQ、知识条目、合同记录，通常更适合被拆成结构化字段，例如：

- 标题
- 正文
- 作者
- 时间
- 类型
- 标签
- 所属系统
- 版本号

这会显著提升后续过滤、路由和召回质量。

## 4.4 结构化与非结构化数据的差异

RAG 接入的数据，不全是“文档文本”。  
工程里通常至少要区分结构化数据和非结构化数据。

### 4.4.1 非结构化数据

非结构化数据通常指：

- PDF
- 网页正文
- Word 文档
- Markdown 文档
- 会议纪要
- 对话记录

它们的特点是自然语言多、结构弱、解析成本高。

对这类数据，重点通常是：

- 解析结构
- 去噪
- 保留标题层级
- 做合理 chunking

### 4.4.2 结构化数据

结构化数据通常指：

- SQL 表
- CSV
- 业务记录
- 工单字段
- 商品属性
- 用户资料
- 监控指标

这类数据不一定适合直接做普通文本向量检索。  
更常见做法是：

- 先做字段映射
- 转成可检索文档
- 保留 schema 和字段含义
- 混合使用结构化查询与语义检索

### 4.4.3 半结构化数据

很多企业数据其实是半结构化，例如：

- HTML 页面
- JSON 文档
- YAML 配置
- API 返回
- Jira / GitLab / CRM 记录

这类数据通常最适合在接入阶段先抽成：

- 主体文本
- 关键字段
- 元数据

再进入统一索引流程。

## 4.5 元数据为什么重要

很多团队前期只关注正文内容，忽略了 metadata。  
但在实际 RAG 系统里，元数据往往决定了检索是否“可控”。

常见元数据包括：

- 文档标题
- 来源系统
- 文档类型
- 创建时间 / 更新时间
- 作者 / 部门
- 标签
- 权限范围
- 版本号
- 语言
- 章节路径

元数据的价值主要体现在几个方面。

### 4.5.1 过滤

例如：

- 只查最近半年文档
- 只查某个系统的知识
- 只查当前用户有权限访问的文档
- 只查中文或英文内容

没有元数据，很多检索只能做全库盲搜。

### 4.5.2 排序与融合

即便向量相似度接近，也可以基于元数据做额外排序，例如：

- 新版本优先
- 权威来源优先
- 官方文档优先
- 高频使用文档优先

### 4.5.3 上下文解释

生成答案时，元数据还能帮助模型和用户理解来源，例如：

- 这段内容来自哪个文档
- 是哪个章节
- 更新时间是什么时候
- 是否来自正式制度还是社区帖子

这对可信度和可解释性很重要。

## 4.6 数据质量对下游检索的影响

RAG 里的很多“检索问题”，其实根源都在数据质量。

### 4.6.1 对 Embedding 的影响

如果 chunk 里混入了大量噪音，例如页码、导航、乱码、重复模板，那么 Embedding 学到的语义中心就会偏移。

结果是：

- 向量表示不稳定
- 相似内容无法聚在一起
- 无关内容反而相似

### 4.6.2 对 Chunking 的影响

切分策略依赖输入文本的结构质量。  
如果解析后的文本顺序已经错乱，那么再好的 chunking 也很难救回来。

例如：

- 标题和正文分离
- 表格被打散成无意义短句
- 代码块和说明混在一起
- 多栏 PDF 顺序错位

这些都会直接影响 chunk 可读性和召回精度。

### 4.6.3 对检索结果的影响

脏数据常见会造成：

- 检索结果重复
- 错误版本被召回
- 召回内容缺关键上下文
- 明明有答案却召不回来

从用户视角看，就是“知识库明明有，系统却答不好”。

### 4.6.4 对生成结果的影响

生成阶段依赖的是召回结果。  
如果召回内容脏、旧、重复或互相冲突，模型很可能会：

- 给出模糊答案
- 混合多个版本的信息
- 错引来源
- 产生幻觉式补全

所以，很多生成问题其实不该先去改 Prompt，而应该先检查数据链路。

## 4.7 一个常见的数据入库链路

一个比较常见、也比较可落地的数据入库流程，大致如下：

1. 识别数据源
2. 拉取原始数据
3. 解析格式并提取结构
4. 做清洗去噪
5. 提取或补全元数据
6. 做去重、版本归并和有效性判断
7. 切分 chunk
8. 生成 Embedding
9. 写入索引与存储
10. 建立增量更新与回溯机制

如果写成更直观的链路，可以表示为：

`Source -> Parse -> Clean -> Normalize -> Metadata -> Dedup -> Chunk -> Embed -> Index`

工程上，通常还应补上几个保障环节：

- 失败重试
- 变更审计
- 数据质量抽检
- 样本回放
- 重新入库能力

## 4.8 一个更实用的清洗检查清单

如果你要快速判断一套 RAG 数据链路是否靠谱，可以先检查下面这些问题：

- 文档是否能稳定增量同步
- 同一份文档是否会重复入库
- 是否能区分新旧版本
- PDF / HTML / 表格 / 代码块是否被正确解析
- 标题层级是否被保留
- 是否有足够可用的元数据
- 是否支持按权限过滤
- chunk 是否可读、可解释、可追溯
- 检索结果是否频繁重复
- 删除或下线文档后是否能及时从索引中移除

如果这些基础问题没解决，通常不应该太早进入模型优化阶段。

## 4.9 设计数据接入与清洗时的常见权衡

数据管道设计里，也有几组非常典型的权衡。

### 4.9.1 全量重建 vs 增量更新

全量重建更简单、更一致，但成本高；增量更新更高效，但更容易出现状态不一致和漏更新。

### 4.9.2 强清洗 vs 保守保留

清洗过重，会误删信息；清洗过轻，又会把噪音带进索引。  
关键不是“清得越多越好”，而是保留对检索有价值的结构和内容。

### 4.9.3 统一格式 vs 保留源特性

全部转成统一纯文本更方便处理，但会丢失结构；保留源格式特性则更利于高质量检索，但管道复杂度更高。

### 4.9.4 通用处理 vs 场景定制

统一的接入链路便于维护，但不同数据源的最佳处理方式常常并不相同。  
高价值数据源通常值得做专门解析与清洗策略。

## 4.10 总结

RAG 的数据接入与清洗，本质上是在回答一个问题：

**如何把真实世界里杂乱、异构、不断变化的数据，变成可检索、可过滤、可追溯、可用于生成的知识单元。**

它直接决定了：

- 检索能不能召回来
- 召回内容准不准
- 上下文是否干净
- 结果是否可信

可以把它理解成 RAG 的地基工程：

- 数据接入负责把知识接进来
- 数据清洗负责把噪音清出去
- 元数据和结构化处理负责让知识变得可控

如果地基不稳，后面的向量检索、混合检索、Reranker 和生成模型都会被一起拖累。
