# 七 Hybrid Retrieval

Hybrid Retrieval（混合检索）通常指把**稀疏检索**和**稠密检索**组合起来做召回，再对结果做融合或重排。

在真实 RAG 系统中，它往往不是“高级可选项”，而是默认方案。  
原因很简单：

- 只靠关键词检索，容易错过语义相近但词面不同的内容
- 只靠向量检索，又容易漏掉错误码、型号、专有名词、精确数字这类强词面信号

混合检索的本质，是用两种不同误差模式的召回器互相补短板。

## 7.1 为什么只靠向量检索不够

向量检索在自然语言问答里很强，但它不是万能解。

### 7.1.1 向量检索擅长语义相近

例如用户问：

`怎么限制接口请求频率`

而文档写的是：

`Rate limiting configuration`

这种词面不同但语义接近的场景，向量检索通常比 BM25 更有优势。

### 7.1.2 向量检索不擅长精确匹配

但如果问题里包含下面这些内容，向量检索就不一定稳：

- 报错码
- 产品型号
- API 名称
- 版本号
- 精确数字
- 缩写
- 特定表名、字段名、函数名

例如：

- `ERR_CONNECTION_RESET`
- `gpt-4.1-mini`
- `user_id`
- `k8s 1.30`

这些内容往往更依赖关键词检索或倒排索引。

### 7.1.3 只用向量检索的常见问题

工程里常见的失败模式包括：

- 明明知识库里有精确答案，但向量没召回来
- 召回到语义相近但不是同一个概念的内容
- 新旧版本文档混淆
- 过度偏好“话题相近”而不是“事实匹配”

所以，从 RAG 工程角度看，只靠向量检索通常不够稳。

## 7.2 稀疏检索与稠密检索各自擅长什么

混合检索的前提，是理解两类检索器的能力边界。

### 7.2.1 稀疏检索擅长什么

稀疏检索通常指：

- 倒排索引
- TF-IDF
- BM25

它更擅长：

- 精确关键词命中
- 专有名词
- 报错码
- 版本号
- 数字和缩写
- 明确实体名

它的优点是：

- 快
- 稳
- 可解释
- 对精确词面很强

但它的问题也很明显：

- 同义改写不友好
- 口语问题不友好
- 语义泛化能力弱

### 7.2.2 稠密检索擅长什么

稠密检索通常指基于 Embedding 的向量检索。

它更擅长：

- 同义表达
- 自然语言问法
- 语义近似
- 主题相近内容召回

它的优点是：

- 更接近用户真实问法
- 更适合问答型 RAG
- 能覆盖词面不一致的内容

但它的问题是：

- 精确匹配不稳定
- 可解释性弱
- 容易召回“相似但不对”的内容

### 7.2.3 为什么两者组合更稳

因为它们的长处和短板并不重合：

- 稀疏检索负责守住“词面精确命中”
- 稠密检索负责补足“语义相近召回”

这也是混合检索比单一路径更稳定的根本原因。

## 7.3 Hybrid Retrieval 的基本思路

混合检索的基本流程通常是：

1. 用户输入 query
2. 稀疏检索召回一批候选
3. 稠密检索召回一批候选
4. 对两路结果做融合
5. 交给 reranker 或直接进入上下文构造

如果写成简化链路，大致是：

`Query -> BM25 Recall + Vector Recall -> Fusion -> Rerank -> Prompt`

工程上，混合检索不要求两路检索器完全对等。  
常见设计包括：

- 两路并行召回后融合
- 先 BM25，再用向量扩召回
- 先向量召回，再用关键词补精确命中
- 根据 query 类型路由到不同组合

重点不在于流程形式，而在于：

**让最终候选集合同时包含词面强相关和语义强相关的内容。**

## 7.4 常见融合方式

混合检索的关键不只是“召回两路结果”，更重要的是“怎么融合”。

### 7.4.1 RRF

RRF（Reciprocal Rank Fusion）是工程里最常见的融合方法之一。

它的思路很简单：

- 不直接比较不同检索器的原始分数
- 只看各自的排名
- 按排名位置给分，再把多路分数相加

优点是：

- 简单
- 稳定
- 不需要分数校准
- 对不同检索器鲁棒

很多系统里，RRF 是混合检索的默认首选。

### 7.4.2 加权分数融合

另一种常见方式是对不同召回器的分数归一化后加权，例如：

`final_score = a * bm25_score + b * vector_score`

优点是：

- 可控
- 便于做精细调参

缺点是：

- 不同检索器的分数空间往往不可直接比较
- 归一化和权重非常依赖数据集

如果没有较好的验证集，分数融合往往不如 RRF 稳。

### 7.4.3 去重后拼接

有些系统会更简单：

- 取 BM25 Top-K
- 取 Vector Top-K
- 合并去重
- 直接交给 reranker

这种方式实现最简单，适合先快速验证混合检索是否有效。

### 7.4.4 路由式融合

更进一步的做法，是先判断 query 类型，再决定采用哪种融合策略。例如：

- 如果 query 包含报错码、型号、表名，偏向 BM25
- 如果 query 更自然语言、描述型问题，偏向向量检索
- 如果 query 同时包含实体和意图，两者并重

这类做法效果可能更好，但实现复杂度更高。

## 7.5 Hybrid Retrieval 适合什么场景

混合检索并不是所有系统都必须上，但下面这些场景通常非常适合。

### 7.5.1 企业知识库搜索

企业知识库通常混合了很多类型的数据：

- 流程文档
- Wiki
- API 文档
- FAQ
- 工单
- 邮件和聊天记录

这些内容既有自然语言描述，也有大量专有名词和版本信息。  
因此混合检索往往是默认选择。

### 7.5.2 技术文档问答

技术文档里经常同时存在：

- 口语化问题
- 精确接口名
- 配置项
- 错误码
- 代码片段

只靠一种检索路径通常很难兼顾。

### 7.5.3 客服与工单系统

客服问答中，用户问题往往很口语化，但知识库内容可能包含：

- 订单号
- 业务术语
- 产品名
- 错误提示
- 历史工单标题

这正是混合检索最容易体现价值的场景。

### 7.5.4 多语言或表达风格差异大的场景

如果用户和文档的表达风格差异很大，例如：

- 用户很口语
- 文档很术语化
- 同一概念有多个叫法

那么稠密检索负责语义桥接，稀疏检索负责精确兜底，通常会更稳。

## 7.6 设计时的关键参数

混合检索效果通常不是“开了就行”，而是由一系列参数共同决定。

### 7.6.1 两路召回深度

首先要决定：

- BM25 取多少条
- 向量召回取多少条

如果某一路取太少，就起不到补充作用；取太多，又会把太多噪音交给后续 reranker。

### 7.6.2 Chunk 粒度

Hybrid Retrieval 的效果仍然强依赖 chunking。  
如果 chunk 太大：

- BM25 会因为无关词过多而被稀释
- 向量检索也可能语义中心不清

如果 chunk 太小：

- 稀疏检索可能召回碎片
- 稠密检索也可能失去上下文完整性

所以，chunk 粒度仍然是混合检索的前置变量。

### 7.6.3 元数据过滤

很多时候，检索效果问题不是“检索器不够强”，而是搜索范围太大。  
加入元数据过滤，例如：

- 时间
- 部门
- 文档类型
- 权限范围
- 产品线

往往比单纯调融合参数更有效。

### 7.6.4 融合顺序

你需要明确系统是：

- 召回后立即融合
- 先融合再重排
- 各路分别重排后再融合

不同顺序会影响成本和精度。  
工程上最常见的依然是：

`多路召回 -> 融合去重 -> rerank`

### 7.6.5 是否接入 reranker

混合检索通常能显著提升召回质量，但它不一定能直接保证最终排序质量。  
如果任务精度要求较高，往往还需要：

- Cross-Encoder Reranker
- ColBERT 风格精排
- 任务定制化 rerank

也就是说，Hybrid Retrieval 通常解决的是“召回更全”，不是“排序自动最优”。

## 7.7 常见问题

混合检索效果不好时，问题不一定出在“混合”本身，往往出在链路细节。

### 7.7.1 两路结果高度重复

如果 BM25 和向量召回本来就大量重合，那么混合后的提升可能很有限。  
这时要检查：

- chunk 是否太粗
- query 是否过于简单
- 向量模型是否没有提供额外语义增益

### 7.7.2 分数不可比

BM25 分数和向量相似度通常不在同一尺度上。  
如果直接硬加，结果经常不稳定。

这也是为什么很多时候更推荐 RRF，而不是直接做分数加权。

### 7.7.3 召回变多，但质量没有提升

这是最常见的问题之一。原因通常包括：

- 候选更多，但噪音也更多
- 去重不充分
- reranker 不够强
- chunk 质量本身差

这说明 Hybrid Retrieval 提升的是“覆盖面”，但如果后处理跟不上，最终效果未必变好。

### 7.7.4 过度依赖某一路

有些系统名义上是混合检索，但实际配置里：

- BM25 权重几乎为零
- 向量召回深度极小
- 路由规则过于偏置

这会让系统重新退化成单一路径，只是形式上看起来是 hybrid。

### 7.7.5 忽略评测

混合检索不是“理论上更好”就一定对当前数据集更好。  
必须看真实评测，例如：

- Recall@K
- Precision@K
- MRR / nDCG
- 最终答案质量
- 成本与时延

否则很容易只是在增加复杂度。

## 7.8 总结

Hybrid Retrieval 的核心价值，不是让系统“更复杂”，而是让召回阶段更稳。

可以把它理解成：

- 稀疏检索负责精确命中
- 稠密检索负责语义召回
- 融合与重排负责把两者组织成更高质量的候选集

它尤其适合：

- 企业知识库
- 技术文档问答
- 客服与工单系统
- 同时存在自然语言问题和精确术语的场景

从工程经验看，很多生产级 RAG 最终都会走向类似方案：

`Chunking + Metadata Filter + BM25 + Vector Search + Fusion + Rerank`

因此，Hybrid Retrieval 更适合被理解成企业 RAG 的默认召回架构，而不是单一检索方法的补丁。
