# 九 Reranker

Reranker（重排器）是在**初召回之后**，对候选结果再次排序的模块。  
它解决的不是“能不能找到相关内容”，而是：

**在已经召回的一批候选里，谁最值得进入最终上下文。**

从工程角度看，RAG 中很多问题并不是“完全没召回”，而是“召回了很多看起来相关的内容，但真正最有用的没有排在前面”。  
Reranker 就是用来解决这个问题的。

## 9.1 为什么召回之后还要重排

初召回阶段的目标通常是：

- 快速
- 覆盖尽可能多的相关候选
- 控制计算成本

因此不管是 BM25、向量检索还是 Hybrid Retrieval，本质上都更像“粗筛”。

### 9.1.1 初召回常见问题

初召回后的候选结果，经常会出现下面几类问题：

- 语义相近，但并不真正回答当前问题
- 相关内容排位靠后
- 多个结果高度重复
- 召回到了正确文档，但错误段落排在前面
- 查询词命中了很多背景材料，却没有命中真正答案

例如用户问：

`怎么关闭某个接口的 rate limit`

初召回可能同时返回：

- rate limiting 的总体介绍
- 接口限流的设计原则
- 配置文件说明
- 真正的“关闭方式”

如果没有重排，真正最有用的段落不一定在最前。

### 9.1.2 召回和排序不是同一个问题

可以把两者简单区分为：

- Recall：尽量把可能有用的内容找回来
- Rank：把最有用的内容放到最前面

很多系统前期只关注召回率，但真正进入 Prompt 的内容是有限的。  
所以排序质量往往比“全库能否找到”更直接影响最终答案质量。

## 9.2 Reranker 在链路中的位置

Reranker 一般位于“初召回”和“上下文构造”之间。

一条典型链路大致如下：

`Query -> Recall -> Top-K Candidates -> Rerank -> Top-N Context -> LLM`

更具体一些，可以是：

`Query -> BM25 / Vector / Hybrid Recall -> Fusion -> Reranker -> Context Compression -> Prompt`

它在链路中的职责非常明确：

- 不负责全库搜索
- 不负责最终生成
- 只负责在候选集内部做更精细的相关性排序

从系统设计上看，Reranker 是典型的二阶段检索组件。

## 9.3 常见 Reranker 类型

Reranker 不止一种，常见可以分成几类。

### 9.3.1 Cross-Encoder Reranker

Cross-Encoder 是最常见的 Reranker 之一。  
它的做法是把 `query + document` 一起输入模型，直接输出一个相关性分数。

它的特点是：

- 能看到 query 和文档的完整交互
- 排序精度通常较高
- 很适合做 Top-K 精排

优点：

- 通常比双塔向量相似度更准
- 对细粒度相关性判断更强

缺点：

- 速度慢
- 成本高
- 不适合全库召回，只适合候选集重排

### 9.3.2 ColBERT / Late Interaction

这类方法保留 query 和文档的 token 级表示，在后期交互阶段计算更细粒度匹配。

相对 Cross-Encoder，它的特点是：

- 交互更细
- 排序通常比双塔更强
- 效率通常比 Cross-Encoder 好一些

适合：

- 对排序精度要求高
- 同时又希望延迟和吞吐不要过高

### 9.3.3 LLM-as-Reranker

有些系统会直接让 LLM 对候选结果打分、排序或选择最相关文档。

优点：

- 推理能力强
- 对复杂问题和长上下文理解更强
- 可以结合任务要求做更灵活判断

缺点：

- 成本高
- 延迟高
- 稳定性和可重复性通常不如专用 reranker

它更适合：

- 高价值、低吞吐场景
- 复杂推理型文档选择
- 研究或实验系统

### 9.3.4 规则型或轻量级重排

并不是所有重排都必须用大模型。  
很多系统还会叠加一些规则型排序，例如：

- 标题命中优先
- 新版本优先
- 权威来源优先
- 去重后保留信息量更高的 chunk
- 按 metadata 做重排序

这类方式精度上限不一定最高，但非常实用，尤其在企业场景里。

## 9.4 Reranker 提升效果的原理

Reranker 之所以有效，核心原因在于它比初召回阶段拥有更强的“精细判断能力”。

### 9.4.1 初召回偏粗粒度

无论是 BM25 还是向量检索，初召回通常都要在大规模数据上运行，所以必须牺牲一部分精度来换速度。

这意味着：

- BM25 更偏关键词匹配
- 向量检索更偏整体语义近似
- 混合检索更偏覆盖范围

但这些方法都未必能准确判断：

- 哪一段最直接回答当前问题
- 哪个结果只是背景材料
- 哪些结果虽然相关但不应该排前

### 9.4.2 Reranker 更关注 query-document 精匹配

Reranker 通常不是只看“主题是否相近”，而是更关注：

- 当前问题到底问的是什么
- 这段候选内容是否真的回答了这个问题
- 哪个候选更具体、更直接、更可用

也就是说，它把“相关性”从粗粒度主题相似，提升到更细粒度的问答匹配。

### 9.4.3 Reranker 改善的是 Top-N 质量

Reranker 不一定改变“候选集里有没有答案”，但它往往能显著改善：

- Top-3 质量
- Top-5 质量
- 最终进入 Prompt 的上下文质量

而在 RAG 中，这部分提升通常比提升很远位置上的候选更重要。

## 9.5 成本、延迟与效果的权衡

Reranker 很有用，但它不是没有代价。

### 9.5.1 成本增加

因为 Reranker 需要对每个 `query-candidate` 配对做额外计算，所以它会带来：

- 更多模型调用
- 更高推理成本
- 更长整体链路

候选越多，成本越高。

### 9.5.2 延迟增加

如果初召回拿回 50 个候选，而每个候选都要精排，那么整体响应时间会明显上升。

所以工程里常见做法是：

- 初召回取较大的 K
- Reranker 只重排较小范围
- 最终只保留更小的 Top-N 进入上下文

### 9.5.3 效果不是无限提升

Reranker 的前提是：

**正确答案已经在候选集里。**

如果初召回根本没把正确内容带回来，再强的 Reranker 也无能为力。

所以它更像“放大已有召回质量”，而不是“替代召回”。

### 9.5.4 常见权衡方式

工程里常见的平衡方式包括：

- 缩小 rerank 的候选数
- 只对高价值 query 启用 rerank
- 使用更轻量的 reranker
- 先规则过滤再做模型重排
- 对延迟敏感场景关闭深度重排

## 9.6 什么时候值得上 Reranker

并不是所有 RAG 系统都必须一开始就上 Reranker，但下面这些场景通常非常值得。

### 9.6.1 候选很多，但前排不准

如果你发现：

- 正确文档常常在 Top-20 里
- 但经常进不了最终上下文

这就是非常典型的 reranker 使用场景。

### 9.6.2 文档相似度高、容易混淆

例如：

- 多版本产品文档
- 一组主题接近的 FAQ
- 大量相似工单
- 同类接口说明

这种场景中，初召回容易“都看起来相关”，而 Reranker 更容易拉开细粒度差异。

### 9.6.3 Prompt 上下文预算有限

如果最终只能放入很少的几个 chunk，那么每个位置都很宝贵。  
这时重排质量就会非常重要。

### 9.6.4 用户问题更复杂

复杂问题通常不只是要“找同主题内容”，而是要找“真正回答这个问题的证据”。  
Reranker 在这类场景中通常比单纯提高 Top-K 更有效。

## 9.7 Reranker 与 Query Rewrite、Hybrid Retrieval 的关系

这几个模块经常一起出现，但职责并不相同。

### 9.7.1 Query Rewrite 解决什么

Query Rewrite 主要解决：

- 用户问题表述不利于检索
- query 太口语、太短、太模糊
- 需要改写成更适合召回的形式

它发生在检索之前。

### 9.7.2 Hybrid Retrieval 解决什么

Hybrid Retrieval 主要解决：

- 稀疏检索和稠密检索各有盲区
- 需要提升初召回覆盖面

它发生在召回阶段。

### 9.7.3 Reranker 解决什么

Reranker 主要解决：

- 候选已经召回来，但排序还不够好

它发生在召回之后。

### 9.7.4 三者的协作关系

可以把三者理解为不同层次的问题：

- Query Rewrite：让 query 更容易被找到
- Hybrid Retrieval：让候选更全
- Reranker：让前排更准

一条比较完整的链路可以是：

`Query -> Rewrite -> Hybrid Recall -> Rerank -> Compress -> Prompt`

这也是很多生产级 RAG 系统的常见形态。

## 9.8 常见问题

Reranker 效果不佳时，问题通常不只在模型本身。

### 9.8.1 候选集本身太差

如果正确答案根本没进候选集，Reranker 不可能把它排上来。  
这时问题应该回到：

- 数据质量
- chunking
- recall depth
- 混合检索策略

### 9.8.2 候选太多，成本失控

如果初召回拿得过多，Reranker 会变得很贵。  
这时通常需要先优化：

- 候选数量
- 元数据过滤
- 去重
- 粗排质量

### 9.8.3 重排更准了，但答案没提升

这也很常见。原因可能包括：

- 上下文构造阶段又把好结果稀释了
- 最终 Prompt 拼接方式不合理
- 生成模型没有忠实使用检索证据

说明 Reranker 只是整条链路的一环，不是最终答案质量的唯一决定因素。

### 9.8.4 过度依赖单一指标

只看 MRR 或 nDCG 提升，不一定代表最终问答体验就提升。  
还应结合看：

- 最终答案质量
- 忠实度
- 引用准确率
- 成本与时延

### 9.8.5 忽略重复和多样性

Reranker 有时会把一堆几乎重复的内容都排到前面。  
这会让上下文浪费在重复证据上。

所以实际系统里常常还要叠加：

- 去重
- 多样性约束
- Parent Document 合并
- 上下文压缩

## 9.9 总结

Reranker 的作用，可以概括成一句话：

**初召回负责把可能相关的内容找回来，Reranker 负责把最值得看的内容排到最前。**

它最适合解决的是：

- 候选已有，但排序不够准
- 上下文预算有限
- 文档相似度高，容易混淆
- 查询复杂，需要更细粒度相关性判断

从工程角度看，Reranker 不是召回的替代品，而是召回之后的精排器。  
很多高质量 RAG 系统，最终都会采用类似结构：

`Recall -> Fusion -> Rerank -> Compression -> Generation`

因此，Reranker 更适合被理解成生产级 RAG 的“精排层”，而不是单独的检索器。
