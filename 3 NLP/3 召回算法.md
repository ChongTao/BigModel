# 一 TF-IDF

词频-逆文档频率([TF-IDF](https://aandds.com/blog/tf-idf.html))是一种统计方法， **它能评估出语料库中某篇文档中某个词的重要程度。** 它是基本思想是：字词的重要随着它在文件中出现的次数成正比增加，但同时会随着它在语料库中出现的频率成反比下降。

TF-IDF 值是由两个主要部分构成的：**TF**（词频）和 **IDF**（逆文档频率），其计算公式为：

TF−IDF*(*t)=TF*(*t)×IDF*(*t)

- **TF**（词频，Term Frequency）：某个词在文档中出现的次数占文档总词数的比例。
- **IDF**（逆文档频率，Inverse Document Frequency）：衡量某个词在整个语料库中的普遍程度。出现越普遍的词（比如“的”“和”）权重越低，稀有的词权重越高



# 二 BM25 

BM25 的全称是 **Best Matching 25**，是一种基于概率检索模型的词语匹配算法，用于评估查询与文档之间的相关性。

BM25 是对 TF-IDF 的改进版，它考虑了两个关键因素：

1. **词频饱和（Term Frequency Saturation）**：词出现越多越重要，但增长会逐渐减弱；
2. **文档长度归一化（Document Length Normalization）**：较长的文档不应因包含更多词而被不公平地评分更高。

其中：

- `q_i`：查询中的第 `i` 个词项（term）
- `IDF(q_i)`：词项的逆文档频率。越少见、区分度越高的词，权重越大
- `f(q_i, D)`：词项 `q_i` 在文档 `D` 中出现的次数
- `|D|`：文档 `D` 的长度
- `avgdl`：语料库中文档的平均长度
- `k1`：控制词频饱和速度，常见默认值约为 `1.2 ~ 2.0`
- `b`：控制文档长度归一化强度，常见默认值约为 `0.75`

可以这样理解 BM25：

1. 查询词在文档里出现过，文档才有机会拿到分数。
2. 某个词在当前文档里出现次数越多，分数通常越高，但不是线性增长，而是逐渐趋于饱和。
3. 某个词在整个语料库里越罕见，它对排序的贡献越大。
4. 文档越长，通常会被做一定程度的长度惩罚，避免“长文天然占便宜”。

## 2.1 BM25 的原理

BM25 本质上是在回答一个问题：**对于给定 query，哪篇文档最相关，应该排在前面？**

它的核心思想可以拆成三步：

1. **先看是否命中查询词**
   如果文档里完全没有出现 query 里的关键词，这篇文档基本拿不到分数。
2. **再看命中词的重要程度**
   如果某个词在整个语料库里很少见，那么它更能区分文档，权重就更高，这部分由 `IDF` 体现。
3. **最后综合词频和文档长度**
   词在当前文档中出现得越多，通常说明越相关，但这个增长不会无限放大；同时过长的文档会被做长度归一化，防止因为“字多”而天然得高分。

换句话说，BM25 不是简单地“统计出现了多少次”，而是同时考虑：

- 这个词在当前文档里**出现了几次**
- 这个词在整个语料库里**常不常见**
- 当前文档相对平均文档来说**是不是过长**

因此 BM25 更适合搜索排序，而不只是关键词计数。

## 2.2 BM25 例子

假设用户查询是：`BM25 原理`

有三篇文档：

- 文档 A：`BM25 原理详解，包含公式推导与参数说明`
- 文档 B：`向量检索原理，介绍 embedding 和余弦相似度`
- 文档 C：`BM25 在 RAG 中的应用，BM25 适合关键词召回`

BM25 排序时，可以这样直观理解：

1. **文档 A**
   同时包含 `BM25` 和 `原理`，而且两个词都比较关键，因此得分通常会很高。
2. **文档 B**
   只包含 `原理`，没有 `BM25`，即使也讲“原理”，相关性通常仍低于文档 A。
3. **文档 C**
   包含 `BM25`，但没有直接出现 `原理`，得分通常会介于 A 和 B 之间，或者低于 A。

如果进一步细化：

- 假设 `BM25` 在整个语料库里出现很少，那么它的 `IDF` 会更高。
- 假设文档 A 中 `BM25` 和 `原理` 都出现 2 次，而文档 C 中 `BM25` 出现 8 次，BM25 不会让文档 C 的分数无限高于文档 A，因为存在**词频饱和**。
- 假设文档 C 是一篇超长文档，虽然包含很多次 `BM25`，但也会受到**文档长度归一化**的影响。

所以最终排序往往不是“谁出现次数最多谁第一”，而是“谁对 query 的关键词覆盖更完整、关键词更关键、长度也更合理，谁更靠前”。

## 2.3 为什么 BM25 比 TF-IDF 更常用

TF-IDF 更像是“给词打权重”的通用统计方法，而 BM25 更像是“为检索排序设计”的打分函数。相较于 TF-IDF，BM25 的优势主要在于：

- **词频不会无限放大**：一个词出现 10 次通常比出现 2 次更相关，但未必需要高出 5 倍分数。
- **对长文档更公平**：长文包含更多词，不代表一定更相关。
- **更适合搜索排序场景**：BM25 常直接作为倒排检索系统中的默认排序函数。

## 2.4 BM25 的优点

- **实现成熟**：Lucene、Elasticsearch、OpenSearch 等系统都原生支持。
- **可解释性强**：能明确说明“为什么这篇文档排在前面”。
- **对精确词匹配有效**：产品名、接口名、错误码、专有名词、时间地点等关键词检索效果通常很好。
- **作为第一阶段召回很高效**：适合在大规模语料中快速筛出候选文档。

## 2.5 BM25 的局限

- **无法很好处理语义相近但词面不同的表达**：例如“退款流程”和“退货怎么申请”可能语义相近，但关键词重合较少。
- **依赖分词与文本预处理**：分词效果、停用词、同义词处理会明显影响结果。
- **对上下文理解有限**：BM25 主要关注词项匹配及统计特征，不真正理解句子语义。
- **对长查询、口语化查询不总是稳定**：自然语言问句中可能包含大量噪声词。

## 2.6 BM25 与向量检索的关系

BM25 和向量检索并不是替代关系，更常见的是互补关系：

- **BM25** 擅长精确词匹配，适合召回包含明确关键词的文档。
- **向量检索** 擅长语义匹配，适合同义改写、自然语言问答、上下文关联等场景。

在 RAG 中，常见做法是：

1. 用 **BM25** 召回一批关键词匹配强的候选文档。
2. 用 **向量检索** 再召回一批语义相关的候选文档。
3. 对两路结果做融合，例如 **RRF**、加权融合，或者交给 **reranker** 精排。

因此，BM25 依然是很多 RAG 系统里的重要基础能力，尤其适合做**第一阶段召回**。

## 2.7 什么时候优先考虑 BM25

下面这些场景通常适合优先使用 BM25，或者至少保留 BM25 这一路召回：

- 搜索目标包含明显关键词，如报错码、函数名、类名、表名、接口路径。
- 用户会输入较短、较准的查询词。
- 业务更强调可解释性和低成本。
- 语料规模较大，需要先做高效粗召回。

如果查询更偏自然语言表达、同义改写很多、语义理解要求高，则通常要结合向量检索一起使用。
