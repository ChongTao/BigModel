# 一 Harness 介绍

Harness 可以理解为大模型之外、围绕模型运行的一整套“控制与执行层”。它负责把模型接到真实的软件环境中，让模型不仅能“回答”，还能“做事”。

如果只给一个 LLM 文本输入和输出接口，它更像一个会推理的生成器；而当你为它补上上下文管理、工具调用、任务分解、状态保存、结果校验和安全约束后，它才更接近一个真正可落地的 Agent。

可以用一句话概括：

**Agent = LLM + Harness**

也可以进一步理解为：

- **LLM** 决定“怎么思考、怎么生成”
- **Harness** 决定“看什么、能做什么、按什么规则做、做完如何验证”

所以，Harness 不是某一个单独组件，而是一整套围绕模型构建的运行机制。它包含所有不属于模型本体的代码、配置和执行逻辑。一个裸模型不是智能体，只有当 Harness 为它提供状态、工具执行、反馈回路和可执行的约束时，它才真正成为智能体。

## 1.1 为什么 Harness 很重要

在传统 AI 应用里，很多人关注点放在“换更强的模型”。但在 Agent 场景里，模型能力只是上限的一部分，真正影响稳定性和产出质量的，往往是 Harness 设计。

原因很直接：

- 模型并不了解你的项目结构、团队规范和业务目标
- 模型本身不会天然拥有文件系统、终端、浏览器、数据库等执行能力
- 模型容易在长任务中丢失上下文，或者偏离目标
- 模型可能会“自信地出错”，没有外部校验就很难发现问题
- 模型缺少边界约束时，容易做出高风险操作

因此，Harness 的核心任务不是替代模型推理，而是把模型放进一个“可工作、可约束、可验证、可恢复”的环境里。

## 1.2 Harness 解决的核心问题

从工程角度看，Harness 主要解决下面几类问题：

| 问题 | Harness 的作用 |
| :--- | :--- |
| 上下文不完整 | 按需加载项目规则、文档、历史状态 |
| 无法执行真实任务 | 提供终端、文件系统、浏览器、外部工具 |
| 长任务容易跑偏 | 拆解任务、记录进度、设置阶段目标 |
| 结果真假难辨 | 用测试、Lint、运行结果、截图等做验证 |
| 操作风险过高 | 加入权限、审批、沙箱、回滚与限制 |

## 1.3 Harness 与 Prompt、Framework、MCP 的关系

Harness 经常和 Prompt Engineering、Agent Framework、MCP 放在一起讨论，但它们并不等价。

- **Prompt** 更关注“如何告诉模型做事”
- **Framework** 更关注“如何组织 Agent 应用的流程与组件”
- **MCP** 更关注“如何标准化接入工具和数据”
- **Harness** 更关注“如何把模型放进一个真实可执行、可验证、可控的工作环境”

可以把 Harness 看成比 Prompt 更大、比单一 Tool Calling 更完整的一层。它往往会使用 Prompt、Framework、MCP、脚本、规则文件、CI、测试系统等多种机制共同构建。

## 1.4 Prompt、Context、Harness 的区别与关系

这三个概念很容易混在一起，但如果从工程视角理解，它们更适合被看作一组**递进关系**，而不是完全平级的概念：

**Prompt -> Context -> Harness**

也就是说：

- Prompt 是最内层，解决“这一轮具体怎么和模型说”
- Context 是外一层，解决“这一轮除了 Prompt 之外，还给模型看哪些信息”
- Harness 是最外层，解决“模型如何在真实环境中持续工作，并管理 Prompt 与 Context”

三者的范围是逐层扩大的。Prompt 包含在 Context 中，Context 又由 Harness 组织、筛选和调度。

### 1.4.1 Prompt 是“你怎么对模型说话”

Prompt 指的是你发给模型的指令文本，包括：

- 角色设定
- 任务描述
- 输出格式要求
- 示例
- 限制条件

例如“你是一名资深后端工程师，请检查这段代码中的性能问题，并按问题清单输出”，这就是 Prompt。

Prompt 的重点是：**告诉模型当前要做什么、怎么做、输出成什么样子。**

如果只看单轮交互，Prompt 往往是最直接、最显眼的一层。

### 1.4.2 Context 是“模型当前能看到什么”

Context 指模型在当前推理时可访问的信息集合，不只是用户这一次输入的内容，还可能包括：

- 系统提示词
- 对话历史
- 项目规则文件
- 当前代码片段
- 检索到的文档
- 工具调用返回结果
- 历史任务状态

可以把 Context 理解为模型当前回合的“工作记忆”。Prompt 也属于 Context 的一部分，但 Context 的范围更大。

Prompt 解决的是“怎么说”，Context 解决的是“给它看什么”。

所以，Context 可以看作 Prompt 的上位层。只有 Prompt，没有足够 Context，模型通常只能在信息不足的情况下工作。

### 1.4.3 Harness 是“模型如何在环境中工作”

Harness 则更外层一些。它不仅决定给模型什么 Prompt、什么 Context，还决定：

- 模型可以调用哪些工具
- 调用工具的权限边界是什么
- 长任务如何拆解和续跑
- 结果如何验证
- 失败后如何恢复
- 哪些操作需要审批

所以 Harness 不是一段提示词，也不是一段上下文，而是一整套运行机制。

Harness 可以看作 Context 的上位层。它决定 Context 从哪里来、加载多少、什么时候更新、何时丢弃，以及模型完成一步操作后下一步该进入什么状态。

### 1.4.4 三者之间的关系

可以用一个递进公式理解：

- **Prompt**：任务指令层
- **Context**：信息供给层，包含 Prompt
- **Harness**：运行控制层，管理 Context，也管理 Prompt 的生成、装配和更新

三者关系可以概括为：

- Prompt 决定当前这一轮具体怎么执行
- Context 决定当前这一轮模型基于哪些信息执行
- Harness 决定整个任务周期内，Prompt 和 Context 如何被持续组织、刷新、验证和约束

如果用层级来表达，可以直接写成：

```text
Harness
  └─ Context
       └─ Prompt
```

或者换一种更工程化的理解：

- Prompt 是一次调用的指令
- Context 是一次调用的输入环境
- Harness 是多次调用组成任务闭环时的总控制系统

如果类比软件系统：

- Prompt 像“当前操作说明书”
- Context 像“当前工作台上摊开的所有资料”
- Harness 像“整个操作系统和执行环境”

### 1.4.5 为什么只优化 Prompt 不够

很多 Agent 系统在早期容易陷入一个误区：认为只要 Prompt 写得更强，模型就能稳定完成复杂任务。

但实际上：

- 没有足够 Context，Prompt 再好也会因为信息不足而出错
- 没有 Harness，模型即使理解任务，也无法稳定执行、校验和纠偏
- 任务越长、环境越复杂，Harness 的重要性越高

所以在简单问答场景中，Prompt 往往最显眼；在需要更多资料时，问题会上升到 Context 设计；而在真实工程场景中，最终决定系统上限的，通常是 Harness 对 Prompt 和 Context 的组织能力。

# 二 Harness 的 5 个核心维度

当前关于 Harness 的讨论里，一个很常见的拆解方式是从 5 个维度看它：上下文管理、执行能力、任务编排、反馈机制、架构护栏。现有总结中的要点也可以统一放到这 5 个维度下理解。

## 2.1 上下文管理

Harness 在这一层的目标，是让模型始终拿到“足够完成当前任务、但不过载”的上下文。

一个常见做法是把规则文件化、结构化，而不是把所有要求一次性塞进系统提示词。

生产实践里常见三层设计：

### 2.1.1 第一层：项目级总规则

项目根目录的 `AGENTS.md`、`CLAUDE.md` 这类文件，相当于项目地图。通常会在每次新对话时优先加载，内容一般包括：

- 技术栈
- 目录结构
- 修改约束
- 禁止事项
- 提交前检查项
- UI 或代码风格边界

这类文件通常不宜过长，重点是高密度、强约束、便于快速加载。

### 2.1.2 第二层：主题化详细规则

把规则继续拆到 `docs/` 或其他目录下，例如：

- `frontend.md`
- `security.md`
- `api-design.md`
- `release.md`

模型先从总规则中知道“去哪找什么信息”，然后在需要时按需读取，而不是一次性灌入全部细节。这能明显降低上下文浪费。

### 2.1.3 第三层：轻量记忆与历史索引

除了静态规则，Harness 还会维护一些轻量级记忆，例如：

- 当前任务进度
- 已尝试方案
- 常见失败原因
- 项目特定偏好
- 上一轮变更产生的中间产物

这类设计的本质，是在“AI 能看见全貌”和“上下文不被塞爆”之间取得平衡。

## 2.2 执行能力

没有执行能力，Agent 只能停留在“建议模式”；有了 Harness，Agent 才能进入“操作模式”。

### 2.2.1 基础执行层

最常见的基础能力有三类：

- **终端**：跑命令、装依赖、执行测试、看日志、构建项目
- **文件系统**：读代码、改文件、生成文档、落盘中间结果
- **浏览器**：访问页面、点击元素、截图、校验真实 UI 效果

这三类能力一旦打通，模型就不只是写建议，而是能闭环完成很多软件任务。

### 2.2.2 外部能力接入

更进一步，Harness 往往还需要把外部系统能力接给模型，例如：

- 数据库
- 搜索系统
- 爬虫
- 设计工具
- 监控平台
- 发布系统
- 工单系统

这里最常见的标准化方式之一就是 **MCP（Model Context Protocol）**。MCP 的价值在于，它把工具和资源接入做成统一协议，便于不同模型、不同 Agent 运行时复用。

### 2.2.3 Skills 与工作流封装

再往上走一层，不是只暴露原子工具，而是把多步骤工作流封装成可复用能力包，例如：

- 自动生成周报
- 从指定网站抓取竞品信息
- 完成一次固定结构的代码 Review
- 根据模板生成技术方案文档

这样模型遇到相似任务时，就不需要每次重新发明流程。

**核心原则：合理选择每一个工具，多不如精。**

## 2.3 任务编排

Harness 在这一层做的是把长任务结构化，让模型从“单轮回答”升级为“多阶段执行”。

典型能力包括：

- 将大目标拆成多个子任务
- 为每一步设置输入、输出和完成条件
- 记录当前处于哪个阶段
- 在失败后从中间状态恢复
- 支持并行执行部分独立任务

在长任务中，进度文件和版本控制非常关键。实践中常见做法包括：

- 用 `progress.md` 记录当前状态、已完成项、待办和阻塞
- 用 Git commit 作为阶段性存档点

这背后的核心原则是：

**`progress.md` 和 Git commit 是 AI 的存档点，存了档才能放心做长任务。**

## 2.4 反馈机制

反馈机制决定了 Agent 是“做完就算”，还是“做完后自证正确”。

Harness 的重点不是让模型口头承诺“已经修好”，而是要求它给出可验证证据。常见反馈信号包括：

- 单元测试
- 集成测试
- E2E 测试
- Lint / 类型检查
- 页面截图
- 日志检查
- Benchmark 结果
- 人工审批结果

理想状态下，模型做出修改后能立即触发校验，并根据校验结果继续修正，形成闭环。

**核心原则：AI 嘴上说修好没有用，跑通测试才算完成。**

## 2.5 架构护栏

如果说前四个维度决定“能不能做成事”，那架构护栏决定“会不会把事情做坏”。

常见护栏包括：

- 权限控制
- 沙箱隔离
- 危险命令审批
- 只允许修改指定目录
- 生产环境只读
- 变更范围限制
- 敏感信息扫描
- 回滚机制

对于企业场景，护栏几乎不是可选项，而是 Harness 的基础组成部分。因为 Agent 一旦拥有终端和文件系统能力，就已经具备了真实执行风险。

# 三 一个典型的 Harness 工作流

一个完整的 Harness 通常会把 Agent 工作流程组织为下面几步：

1. 读取项目总规则，例如 `AGENTS.md`
2. 分析任务目标，识别需要哪些上下文
3. 按需读取相关文档、代码、历史记录
4. 制定执行计划并拆解子任务
5. 调用终端、文件系统、浏览器或 MCP 工具开始执行
6. 在关键节点记录进度和中间结果
7. 运行测试、检查日志、对比截图或进行其他验证
8. 若失败则根据反馈继续修复
9. 输出最终结果、风险说明和后续建议

这说明 Harness 不是单点能力，而是一条完整的“感知 -> 规划 -> 执行 -> 校验 -> 迭代”链路。

# 四 如何理解 Harness 的工程价值

Harness 的工程价值主要体现在四个方面：

- **稳定性**：让模型在复杂任务中更少跑偏
- **可扩展性**：同一个模型可以接入更多工具和流程
- **可控性**：可以定义边界、权限、审核和回滚策略
- **可复用性**：把一次成功经验沉淀为规则、Skill 或自动化流程

从这个角度看，Harness 的本质不是“给模型加几个工具”，而是把人类团队已有的工程方法，转译成模型可消费、可执行、可验证的系统环境。

# 五 落地 Harness 的常见实践

如果要在团队里逐步搭建 Harness，可以先从轻量版本开始：

## 5.1 最小可用 Harness

- 一个项目级规则文件，如 `AGENTS.md`
- 可读写代码仓库的文件系统能力
- 可执行命令和测试的终端能力
- 一套基础验证流程，如 `lint + test`

## 5.2 进阶 Harness

- 按主题拆分规则文件
- 引入浏览器能力做真实页面校验
- 接入 MCP，把数据库、搜索、设计平台等接给 Agent
- 为常见任务封装 Skills
- 用 `progress.md`、任务状态文件保存长任务上下文

## 5.3 企业级 Harness

- 增加权限分级与审批流
- 加入敏感信息检测和安全扫描
- 支持任务审计、日志追踪、失败恢复
- 建立更系统的评估体系，如成功率、回滚率、人工接管率

# 六 总结

Harness 可以看作 Agent 的“操作系统”或“驾驶舱”。模型负责生成与决策，Harness 负责提供环境、规则、工具、反馈和边界。

如果只关注模型能力，往往只能得到一个“会说但不一定会做”的系统；而当 Harness 设计足够完善时，模型才真正具备稳定完成长任务、复杂任务和真实工程任务的能力。

用本文最核心的一句话收尾：

**一个裸模型不是智能体。只有当 Harness 为它提供状态、工具执行、反馈回路和可执行约束时，它才真正成为智能体。**
