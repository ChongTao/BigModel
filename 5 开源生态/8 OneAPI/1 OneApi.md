# 1 OneAPI

`OneAPI` 更准确的定位不是“大模型原理”或“提示词技术”，而是一个 **大模型 API 网关 / 代理层 / 管理平台**。

它对上提供统一接口，对下接入多个模型供应商或本地推理服务，让业务只对接一套 API，就能在 OpenAI、Claude、Gemini、通义、智谱，甚至本地模型之间切换。

## 1.1 它属于什么类型

从系统分层看，OneAPI 更接近：

- 模型网关
- API 代理层
- 多模型统一接入平台
- 鉴权、计费、限流与路由控制层

所以它更适合放在“开源生态 / 开源软件”这一类目录里，而不是放在“LLM 原理”或“杂项技术”里。

## 1.2 它解决什么问题

如果业务直接分别对接 OpenAI、Azure OpenAI、Anthropic、Gemini、通义、智谱等接口，通常会遇到下面这些问题：

| 问题 | 描述 |
| :--- | :--- |
| 多平台对接复杂 | 不同厂商的认证、模型名、参数格式、返回结构并不完全一致 |
| Key 管理混乱 | 多个团队、多个项目、多个供应商的密钥难以统一管理 |
| 成本不可控 | 很难按用户、团队或项目维度统计与限额 |
| 服务不稳定 | 单一供应商抖动时，缺少统一切换与兜底层 |
| 缺少监控 | 业务层往往看不到谁在调用、消耗多少 Token、成功率如何 |

OneAPI 的价值就是在业务与模型供应商之间加一层统一控制面。

## 1.3 核心架构

```text
┌──────────────┐
│   前端 / 应用  │
│ (Web / App)  │
└──────┬───────┘
       │ OpenAI 兼容接口
┌──────▼──────────────────┐
│       OneAPI 网关        │
│────────────────────────│
│ 认证鉴权 | 限流 | 配额    │
│ 路由策略 | 协议转换 | 监控 │
└──────┬──────────────────┘
       │
┌──────▼──────────────────────────┐
│      各类模型服务 / 推理后端      │
│ OpenAI / Claude / Gemini / GLM │
│ vLLM / llama.cpp / Ollama       │
└─────────────────────────────────┘
```

## 1.4 核心能力

- 统一 API 接入
  - 常见形态是兼容 OpenAI 风格接口，例如 `/v1/chat/completions`、`/v1/embeddings`
- 多模型接入与路由
  - 可以同时接多个云厂商和本地推理后端
- Key 与权限管理
  - 支持用户级、项目级、渠道级密钥与访问控制
- 配额、计费与统计
  - 统计 Token、请求量、成功率、并发与额度消耗
- 限流与风控
  - 支持 QPS、并发、额度等限制
- 协议适配
  - 把下游模型差异收敛为统一上游接口

## 1.5 与 Ollama 的区别

`OneAPI` 和 `Ollama` 经常会一起出现，但两者不在同一层。

| 维度 | OneAPI | Ollama |
| :--- | :--- | :--- |
| 本质 | 模型网关 / API 管理层 | 本地模型运行时 |
| 主要职责 | 统一接入、鉴权、限流、路由、统计 | 拉取模型、加载模型、本地推理 |
| 面向对象 | 团队、平台、服务端系统 | 单机开发者、本地推理环境 |
| 上游 | 应用系统 / Agent / RAG 服务 | 本地 CLI / 本地服务 |
| 下游 | 云模型、本地模型、推理服务 | 本地模型文件与推理引擎 |

可以把两者理解成：

- `Ollama` 负责“把模型跑起来”
- `OneAPI` 负责“把多个模型统一接进来并管起来”

在实际系统里，`Ollama` 甚至可以作为 `OneAPI` 的一个下游后端。

## 1.6 适用场景

OneAPI 比较适合：

- 一个系统要接多个大模型供应商
- 需要统一 OpenAI 风格接口，降低业务接入复杂度
- 需要对不同团队、用户、项目做配额与密钥管理
- 需要在模型切换、灰度、降级、兜底之间做统一控制
- RAG、Agent 或企业内部平台要做集中化模型接入

如果只是本地单机运行一个模型，通常 `Ollama`、`llama.cpp`、`vLLM` 这类运行时更直接；如果是团队级模型入口，OneAPI 更合适。

## 1.7 常见的 OneAPI 类开源项目

这里的 “OneAPI” 更像一种产品形态，不只指单一仓库。

| 项目 | 定位 | 核心技术栈 | OpenAI API 兼容 | 多模型支持 | 鉴权 / Key 管理 | 计费 / Token 统计 | UI 管理台 | 适合场景 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `songquanpeng/one-api` | 完整 OneAPI 平台 | Go + Gin + GORM | 高 | 高 | 高 | 中高 | 有 | 企业 / RAG / Agent |
| `openai-proxy` | 轻量代理 | Go / Node | 中 | 低 | 低 | 低 | 无 | Key 隐藏 / 小规模 |
| `llm-proxy` | LLM 转发层 | Node / Go | 中 | 低 | 低 | 低 | 无 | 快速转发 |
| `Helicone` | LLM 可观测性 | Node + DB | 中高 | 中 | 中 | 高 | 有 | 用量分析 / 审计 |
| `Kong + LLM 插件` | API Gateway 扩展 | Kong / Lua | 中低 | 高 | 高 | 中 | 有 | 大型平台 |
| `APISIX + AI 插件` | 云原生网关 | APISIX / Lua | 中低 | 高 | 高 | 中 | 有 | K8s / 企业网关 |

## 1.8 一句话总结

OneAPI 不是模型本身，也不是训练框架，而是位于业务系统与模型服务之间的 **统一网关层**。  
如果 `Ollama` 解决的是“模型怎么跑”，那 `OneAPI` 解决的就是“多个模型怎么统一接、统一管、统一计量”。
