# 1 推理框架介绍

在大模型领域，**推理框架（Inference Framework）** 指的是：

> 用于加载训练完成的模型权重，并在服务环境中高效执行前向计算的一整套运行时系统。

它关注的是**模型怎么跑得更快、更稳、更省资源**，而不是模型怎么训练出来。

## 1.1 推理框架解决什么问题

一个模型训练完成后，直接拿训练代码上线，通常会遇到这些问题：

- 显存占用高
- 首 token 延迟高
- 吞吐低
- 并发能力差
- KV Cache 利用率低
- 不容易做量化和多卡部署
- 很难直接对外提供稳定 API 服务

推理框架的目标就是把这些问题系统化解决掉。

| 目标 | 说明 |
| :--- | :--- |
| 提升吞吐 | 每秒生成更多 token |
| 降低延迟 | 缩短首 token 和整体响应时间 |
| 降低显存压力 | 通过量化、KV Cache 管理和显存复用减少资源占用 |
| 支持并发 | 支持动态批处理、请求排队、连续调度 |
| 支持服务化 | 提供 HTTP、gRPC 或 OpenAI 兼容接口 |
| 支持多卡扩展 | 支持张量并行、流水线并行或专家并行 |

## 1.2 推理框架的核心组成

一个完整的推理框架，通常由下面几部分组成：

### 1.2.1 模型加载层

- 加载 FP16、BF16、INT8、INT4 等不同格式权重
- 兼容 Hugging Face、GGUF 或框架自定义格式

### 1.2.2 执行引擎

- 优化 Attention 和 FFN 计算
- 使用 fused kernel、FlashAttention 等算子优化
- 调整算子执行路径以提升 GPU 利用率

### 1.2.3 KV Cache 管理

- 管理上下文缓存
- 复用显存页或块
- 减少内存碎片
- 支持长上下文和多请求并发

### 1.2.4 调度与批处理

- 动态 batching
- 连续批处理
- Token 级调度
- 请求队列管理

### 1.2.5 并行与部署能力

- Tensor Parallel
- Pipeline Parallel
- Expert Parallel（MoE）
- 多卡、多机或 Kubernetes 环境部署

### 1.2.6 服务接口层

- REST API
- OpenAI 兼容接口
- 流式输出
- 监控、日志与健康检查

## 1.3 主流推理框架

### 1.3.1 vLLM

目前最常见的大模型在线推理框架之一。

特点：

- 以 `PagedAttention` 为代表的 KV Cache 管理优化
- 高并发、连续批处理能力强
- OpenAI 兼容接口成熟
- 很适合在线推理服务

适用场景：

- Chat 服务
- Agent 后端
- RAG 生成层
- SaaS 模型服务

### 1.3.2 TensorRT-LLM

NVIDIA 生态里的高性能推理框架。

特点：

- 对 NVIDIA GPU 优化很深
- 支持 INT4、FP8 等低精度推理
- 多卡并行能力强
- 更偏生产级性能优化

适用场景：

- NVIDIA GPU 集群
- 高吞吐生产环境
- 对性能和延迟要求非常高的服务

### 1.3.3 TGI

Hugging Face 的 Text Generation Inference。

特点：

- 开箱即用
- 与 Hugging Face 生态结合紧
- 服务化部署较方便
- 对容器化、Kubernetes 场景友好

适用场景：

- 标准化部署
- Hugging Face 模型上线
- 需要快速搭服务的团队

### 1.3.4 DeepSpeed Inference

微软 DeepSpeed 在推理侧的能力延伸。

特点：

- 和训练体系联系紧密
- 更强调大模型分布式推理
- 适合已有 DeepSpeed 体系的团队

### 1.3.5 llama.cpp

面向本地和轻量部署的推理框架。

特点：

- 支持 CPU 推理
- 常见于 GGUF 模型生态
- Windows、macOS、Linux 支持较好
- 适合本地设备和边缘环境

适用场景：

- 本地单机运行
- 边缘设备
- 资源有限环境

## 1.4 推理框架怎么分类

### 按运行环境

- GPU 推理：如 vLLM、TensorRT-LLM
- CPU 推理：如 llama.cpp
- 边缘推理：移动端或嵌入式设备上的轻量推理运行时

### 按产品形态

- 纯执行引擎
- 带完整 API 服务的推理服务
- 云厂商托管推理平台

### 按目标场景

- 在线高并发推理
- 本地开发与桌面推理
- 企业私有化部署
- 多卡/多机大模型服务

## 1.5 推理框架 vs 训练框架

| 维度 | 训练框架 | 推理框架 |
| :--- | :--- | :--- |
| 代表 | PyTorch / TensorFlow | vLLM / TensorRT-LLM / TGI |
| 核心目标 | 训练参数 | 执行前向推理 |
| 优化重点 | 反向传播、梯度同步 | KV Cache、并发调度、延迟与吞吐 |
| 显存使用 | 更关注训练过程 | 更关注推理压缩和缓存复用 |
| 并发能力 | 通常不是重点 | 是重点能力 |

一句话理解：

> 训练框架负责“造模型”，推理框架负责“跑模型”。

## 1.6 推理框架在系统架构中的位置

在典型系统里，推理框架通常位于模型服务层：

```text
客户端 / 应用
   ↓
API 网关 / 统一接入层
   ↓
推理服务（vLLM / TGI / TensorRT-LLM）
   ↓
GPU / CPU / 加速卡
   ↓
模型权重
```

如果放进 RAG 或 Agent 架构中，可以理解为：

```text
用户请求
   ↓
检索 / 工具调用 / 上下文拼装
   ↓
推理框架
   ↓
生成结果
```

它是生成阶段真正执行模型前向计算的“发动机”。

## 1.7 与 Ollama、OneAPI、Harness 的区别

这几个概念经常被混在一起，但它们不在同一层。

| 名称 | 主要定位 | 更接近什么 |
| :--- | :--- | :--- |
| 推理框架 | 模型运行时与服务引擎 | 执行层 |
| Ollama | 本地模型运行与分发工具 | 单机运行时 / 产品化封装 |
| OneAPI | 多模型统一接入与网关层 | API 网关 / 管理层 |
| Harness | 围绕模型运行的控制环境 | 工程运行外壳 |

可以这样理解：

- `vLLM / TGI / TensorRT-LLM / llama.cpp` 解决“模型如何高效执行”
- `Ollama` 更强调“本地怎么把模型跑起来、拉下来、管起来”
- `OneAPI` 解决“多个模型接口怎么统一接入、统一计量和统一路由”
- `Harness` 解决“模型运行时的权限、工具执行、上下文装配、状态管理”

因此：

- 推理框架适合归入 `5 开源生态`
- 不适合放入 `0 LLM`
- 也不应和 `OneAPI`、`Harness` 混成同一概念

## 1.8 一句话总结

推理框架不是训练框架，也不是网关层，而是位于模型服务内部、负责把模型高效跑起来的核心运行时系统。
