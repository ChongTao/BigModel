# 2 vLLM

`vLLM` 是目前最常见的大模型在线推理框架之一，核心定位是：

> 让大模型在服务场景下以更高吞吐、更低延迟和更高显存利用率运行起来。

它最出名的点不是“也能跑模型”，而是围绕 **KV Cache 管理、连续批处理和服务化接口** 做了系统化优化，因此很适合做在线推理服务。

## 2.1 它解决什么问题

传统推理服务在多请求并发时，常见瓶颈主要有：

- KV Cache 占用大，显存碎片严重
- 请求长度不同，普通 batch 利用率低
- 首 token 延迟高
- GPU 利用率不稳定
- 很难同时兼顾吞吐和响应速度

vLLM 的目标就是把这些问题尽量收敛掉。

## 2.2 核心创新

### 2.2.1 PagedAttention

这是 vLLM 最有代表性的设计。

传统 KV Cache 通常按连续内存大块分配，容易出现：

- 长请求占着大量显存
- 请求结束后留下很多碎片
- 并发一上来，显存很快变得难以复用

vLLM 的思路更像“分页内存”：

- 把 KV Cache 切成较小的块
- 通过映射关系组织这些块
- 请求只按需占用和复用对应页面

这样做的结果是：

- 显存利用率更高
- 碎片更少
- 更适合高并发场景

### 2.2.2 Continuous Batching

普通静态 batch 往往要等一组请求凑齐，再一起跑。

vLLM 更强调连续批处理：

- 新请求可以持续插入
- 已完成请求可以及时移出
- 调度粒度更细

这使得 GPU 不需要频繁等“整批结束”，整体吞吐更高。

### 2.2.3 面向服务的接口设计

vLLM 不只是一个底层执行引擎，它也提供了比较成熟的服务化能力：

- OpenAI 兼容接口
- 流式输出
- 多模型服务部署
- 与常见应用框架集成

因此它非常适合作为在线服务后端，而不只是本地实验工具。

## 2.3 系统里它处在什么位置

在典型系统中，vLLM 一般位于模型执行层：

```text
应用 / Agent / RAG
   ↓
API 网关 / 统一接入层
   ↓
vLLM
   ↓
GPU
   ↓
模型权重
```

如果和仓库里其他概念对比：

- `OneAPI` 更像模型网关和统一接入层
- `vLLM` 是真正执行推理的后端引擎
- `Harness` 是围绕模型运行的工程控制环境

## 2.4 它适合什么场景

vLLM 比较适合：

- 在线聊天服务
- Agent 后端推理
- RAG 生成层
- 需要 OpenAI 兼容 API 的自建服务
- 多用户并发访问的模型服务

如果只是本地单机测试一个小模型，`Ollama` 或 `llama.cpp` 往往更直接；如果要做服务端高并发生成，vLLM 通常更合适。

## 2.5 与其他方案的区别

| 对比项 | vLLM | Ollama | OneAPI | TGI |
| :--- | :--- | :--- | :--- | :--- |
| 核心定位 | 推理框架 | 本地运行时产品 | 模型网关 | 推理服务框架 |
| 重点 | 高吞吐、KV Cache、连续批处理 | 本地运行与模型管理 | 多模型统一接入与计量 | Hugging Face 生态服务化 |
| 典型场景 | 服务端在线推理 | 本地单机使用 | 团队统一模型入口 | 标准化模型部署 |

简单理解：

- `Ollama` 更像“本地把模型跑起来”
- `OneAPI` 更像“把多个模型统一接进来”
- `vLLM` 更像“把模型高效跑成在线服务”

## 2.6 优势与限制

优势：

- 高并发能力强
- 显存利用率高
- OpenAI 兼容接口成熟
- 适合生产侧在线推理

限制：

- 更适合 GPU 服务端，不是最轻量的本地工具
- 关注点在推理效率，不负责统一网关治理
- 真正的最佳效果依赖具体模型、硬件和部署方式

## 2.7 一句话总结

vLLM 的核心价值，不只是“支持大模型推理”，而是通过 `PagedAttention` 和连续批处理，把大模型在线推理的资源利用率和并发能力做到了工程可用的水平。
