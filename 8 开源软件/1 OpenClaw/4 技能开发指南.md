# OpenClaw 技能开发指南

本文聚焦 Skill 的开发方式、运行边界和发布流程，并和 Tool、Plugin 的职责做清晰区分。

> 重要说明：
> 本文中的 Skill，主要采用当前仓库内部的“业务能力封装”视角来组织内容，用于说明一个面向任务场景的能力单元如何设计、开发和发布。
>
> 这与 OpenClaw 官网当前定义的 `SKILL.md` 型 Skill 并不完全相同。官网语义下，Skill 更接近一个以 `SKILL.md` 为核心的指令包 / workflow pack，通常以目录形式存在，并通过 `SKILL.md` 描述触发条件、操作步骤和工具使用规则。
>
> 因此阅读本文时需要明确区分：
> - 如果你关注的是**官网当前的 Skill 标准形态**，应优先参考官网的 `SKILL.md` 体系。
> - 如果你关注的是**本仓库当前文档语境下的 Skill 抽象**，则本文仍然有效，但不要直接将其等同于官网当前定义。

## 1 技能系统概览

### 1.1 技能是什么

OpenClaw **Skills** 是面向用户任务的**业务级能力封装**。一个 Skill 通常会围绕一类目标场景组织意图识别、参数处理、权限判断、外部调用和结果返回，例如：

- GitHub Issue 管理
- Notion 文档读写
- 多步骤内容生成工作流
- 数据查询与分析

Skill 的重点不是暴露单个原子动作，而是把一组相关动作组织成用户可直接使用的能力单元。

### 1.2 Skill、Tool、Plugin 的关系

| 维度 | Skill | Tool | Plugin |
|------|------|------|------|
| 角色 | 面向用户任务的业务能力 | Agent 可直接调用的原子能力 | 扩展平台底层能力的机制 |
| 关注点 | 意图、流程、权限、结果组织 | 文件、浏览器、消息、记忆等基础动作 | 渠道、模型、记忆后端、工具、Hook 扩展 |
| 粒度 | 高 | 低 | 平台级 |
| 典型例子 | `github-issues`、`notion` | `message-tool`、`memory-tool` | Channel Plugin、Provider Plugin |

可以简单理解为：

- **Tool** 是积木。
- **Skill** 是用积木搭出来的业务能力。
- **Plugin** 是给整个平台增加新的积木、接入点或扩展槽位。

### 1.3 Skill 的实现方式说明

当前文档示例沿用 `@openclaw/plugin-sdk` 暴露的 `defineSkill` 能力来描述 Skill 开发入口。也就是说，**Skill 与 Plugin 在工程层面可能复用同一套 SDK**，但在职责上仍应区分：

- Skill 负责用户任务。
- Plugin 负责平台扩展。

如果后续 OpenClaw 提供独立的 Skill SDK，应以实际官方 SDK 为准。

## 2 技能项目结构

### 2.1 常见目录结构

```text
my-awesome-skill/
├── src/
│   ├── index.ts                 # 技能入口
│   ├── handlers/                # 意图处理器
│   │   ├── create.ts
│   │   ├── list.ts
│   │   └── close.ts
│   ├── clients/                 # 外部服务客户端封装
│   │   └── github.ts
│   ├── schema/                  # 输入/配置校验
│   │   └── issue.ts
│   └── types.ts                 # 类型定义
├── package.json
├── skill.manifest.json
├── README.md
└── CHANGELOG.md
```

推荐把“外部 API 调用”与“意图处理逻辑”分开，不要把所有逻辑都堆在单个 handler 文件里。

### 2.2 `package.json` 基础配置

```json
{
  "name": "@openclaw/github-issues",
  "version": "1.0.0",
  "description": "Manage GitHub Issues from OpenClaw",
  "main": "dist/index.js",
  "types": "dist/index.d.ts",
  "files": ["dist", "skill.manifest.json"],
  "scripts": {
    "build": "tsc",
    "dev": "tsc --watch",
    "test": "vitest",
    "lint": "eslint src/",
    "format": "prettier --write src/"
  },
  "dependencies": {
    "@openclaw/plugin-sdk": "^1.x.x",
    "@octokit/rest": "^20.x.x"
  },
  "devDependencies": {
    "typescript": "^5.x.x",
    "vitest": "^1.x.x",
    "@types/node": "^20.x.x"
  }
}
```

这里继续使用 `@openclaw/plugin-sdk` 是因为本文示例假设 Skill 入口能力由同一套 SDK 提供。

### 2.3 `skill.manifest.json` 的职责

`skill.manifest.json` 主要描述：

- Skill 身份信息
- 兼容版本
- 意图清单
- 配置项
- 示例语句

Agent 会基于这些元数据判断“什么时候该选中这个 Skill”。

## 3 开发环境与脚手架

### 3.1 前置要求

```bash
# Node.js v22 或更高
node --version  # v22.0.0+

# npm 或 pnpm
npm --version   # v10+
pnpm --version  # v8+
```

### 3.2 CLI 命令约定

为和当前仓库其他文档保持一致，本文统一使用 `openclaw skills ...` 命令前缀来描述 Skill 的管理操作，例如：

- `openclaw skills list`
- `openclaw skills search`
- `openclaw skills install`
- `openclaw skills info`

如果你本地 CLI 版本仍使用单数命令（如 `openclaw skill create`、`openclaw skill publish`），请以本地帮助信息为准。

### 3.3 快速创建项目

```bash
# 如果当前 CLI 提供脚手架命令，可直接使用
openclaw skills create my-awesome-skill

# 如果当前版本没有 create 子命令，可手动初始化
mkdir my-awesome-skill
cd my-awesome-skill
npm init -y
npm install @openclaw/plugin-sdk
```

### 3.4 本地加载方式

本地开发时，Skill 通常以“本地扩展包”的方式被 OpenClaw 发现。当前文档约定下，可以将 Skill 包放入或软链到：

```text
~/.openclaw/extensions/
```

这里的 `extensions/` 是本地扩展发现目录，**在工程形态上可能同时承载 Plugin 和 Skill**；但在语义上，Skill 仍然是面向任务的业务能力，不等同于 Plugin。

## 4 开发技能步骤

### 4.1 步骤 1：定义技能元数据

**`skill.manifest.json`**

```json
{
  "id": "github-issues",
  "name": "GitHub Issues Manager",
  "version": "1.0.0",
  "description": "Create, list, and manage GitHub Issues",
  "author": "OpenClaw Community",
  "license": "MIT",
  "keywords": ["github", "issues", "development"],
  "compatible": "^1.0.0",
  "intents": [
    {
      "name": "create_issue",
      "description": "Create a new GitHub issue",
      "examples": [
        "Create a GitHub issue titled 'Fix login bug'",
        "Open a new issue about performance improvements"
      ]
    },
    {
      "name": "list_issues",
      "description": "List issues from a repository",
      "examples": [
        "Show me open issues in myrepo",
        "List all GitHub issues assigned to me"
      ]
    }
  ],
  "configuration": {
    "required": ["github_token"],
    "optional": ["github_org"],
    "schema": {
      "github_token": {
        "type": "string",
        "description": "Personal access token for GitHub API",
        "sensitive": true
      },
      "github_org": {
        "type": "string",
        "description": "Default GitHub organization"
      }
    }
  }
}
```

### 4.2 步骤 2：实现技能入口

**`src/index.ts`**

```typescript
import { defineSkill } from '@openclaw/plugin-sdk'
import { createIssueHandler } from './handlers/create'
import { listIssuesHandler } from './handlers/list'

export default defineSkill({
  id: 'github-issues',
  displayName: 'GitHub Issues Manager',
  description: 'Manage GitHub Issues',

  async initialize(runtime) {
    const token = runtime.config.get('github_token')
    if (!token) {
      throw new Error('GitHub token is required.')
    }

    const logger = runtime.logging.createLogger('github-issues')
    logger.info('GitHub Issues skill initialized')
  },

  handlers: {
    create_issue: createIssueHandler,
    list_issues: listIssuesHandler
  },

  async authorize(runtime, context) {
    const userPerms = context.userPermissions || []
    return userPerms.includes('github:write')
  }
})
```

### 4.3 步骤 3：实现处理器

Skill 处理器可以：

- 直接调用外部 API
- 调用已有 Tool
- 组合多个步骤形成工作流

它**不要求**必须建立一层虚构的“GitHub 专用工具集合”后再开发。只要职责清晰，直接封装业务逻辑也是合理的。

**`src/handlers/create.ts`**

```typescript
import { Octokit } from '@octokit/rest'

export async function createIssueHandler(input, runtime) {
  if (!input.title) {
    return {
      success: false,
      error: 'MISSING_TITLE',
      message: 'Issue title is required'
    }
  }

  const token = runtime.config.get('github_token')
  const octokit = new Octokit({ auth: token })
  const logger = runtime.logging.createLogger('github-issues:create')

  try {
    const response = await octokit.issues.create({
      owner: input.owner,
      repo: input.repo,
      title: input.title,
      body: input.description
    })

    logger.info(`Issue created: ${response.data.number}`)

    return {
      success: true,
      issueNumber: response.data.number,
      issueUrl: response.data.html_url,
      message: `Issue #${response.data.number} created successfully`
    }
  } catch (error) {
    logger.error(`Failed to create issue: ${error.message}`)

    return {
      success: false,
      error: error.message,
      message: `Failed to create issue: ${error.message}`
    }
  }
}
```

**`src/handlers/list.ts`**

```typescript
import { Octokit } from '@octokit/rest'

export async function listIssuesHandler(input, runtime) {
  const token = runtime.config.get('github_token')
  const octokit = new Octokit({ auth: token })
  const logger = runtime.logging.createLogger('github-issues:list')

  try {
    const response = await octokit.issues.listForRepo({
      owner: input.owner,
      repo: input.repo,
      state: input.state || 'open',
      per_page: input.limit || 10
    })

    return {
      success: true,
      count: response.data.length,
      issues: response.data.map((issue) => ({
        number: issue.number,
        title: issue.title,
        state: issue.state,
        url: issue.html_url
      }))
    }
  } catch (error) {
    logger.error(`Failed to list issues: ${error.message}`)

    return {
      success: false,
      error: error.message,
      message: `Failed to list issues: ${error.message}`
    }
  }
}
```

## 5 技能元数据与发现机制

### 5.1 Intents 的作用

`intents` 的本质是告诉 Agent：

- 这个 Skill 能做什么
- 用户会怎么表达这类需求
- 需要哪些参数

示例：

```json
{
  "intents": [
    {
      "name": "create_issue",
      "description": "Create a new GitHub issue",
      "parameters": {
        "owner": "Repository owner",
        "repo": "Repository name",
        "title": "Issue title"
      },
      "examples": [
        "Create a GitHub issue titled 'Fix login bug'",
        "Open a new issue about performance improvements"
      ]
    }
  ]
}
```

### 5.2 自动注册到 `AGENTS.md`

Skill 安装后，OpenClaw 会自动更新 `~/.openclaw/workspace/AGENTS.md`，让 Agent 知道当前可用的 Skill 清单与能力范围。

```markdown
## 已安装的技能

### GitHub Issues Manager (v1.0.0)
- ID: github-issues
- 描述: Create, list, and manage GitHub Issues
- 意图:
  - create_issue
  - list_issues
```

### 5.3 配置项设计建议

配置项应尽量满足三点：

- 必需项与可选项分开
- 敏感信息显式标注
- 默认值和作用写清楚

像 `github_token` 这类配置，应始终通过配置系统或环境变量注入，不要硬编码在代码中。

## 6 常见技能类型示例

### 6.1 集成类 Skill

目标是接入第三方服务，如 GitHub、Notion、Slack。

```typescript
export default defineSkill({
  id: 'notion-docs',
  displayName: 'Notion Documents Manager',
  handlers: {
    list_databases: async (input, runtime) => {
      // 调用 Notion API
    },
    create_page: async (input, runtime) => {
      // 创建 Notion 页面
    }
  }
})
```

### 6.2 工作流类 Skill

目标是把多步任务编排成单个业务能力。

```typescript
export default defineSkill({
  id: 'content-workflow',
  displayName: 'Content Creation Workflow',
  handlers: {
    draft_article: async (input, runtime) => {
      // 1. 收集信息
      // 2. 生成大纲
      // 3. 编写内容
      // 4. 审核与返回
      return { success: true }
    }
  }
})
```

### 6.3 分析类 Skill

目标是数据提取、分析和结果组织。

```typescript
export default defineSkill({
  id: 'analytics',
  displayName: 'Data Analytics',
  handlers: {
    analyze_metrics: async (input, runtime) => {
      // 查询数据
      // 统计分析
      // 组织结果
      return { summary: 'analysis result' }
    }
  }
})
```

## 7 测试与本地调试

### 7.1 测试原则

Skill 测试不要直接依赖真实外部服务。更稳妥的方式是：

- mock 外部 API 客户端
- 先测输入校验
- 再测成功/失败分支
- 把网络调用留给集成测试

### 7.2 单元测试示例

下面示例只验证“缺少标题时直接返回错误”，避免真实访问 GitHub：

```typescript
import { describe, it, expect, beforeEach, vi } from 'vitest'
import { createIssueHandler } from '../handlers/create'

vi.mock('@octokit/rest', () => ({
  Octokit: vi.fn(() => ({
    issues: {
      create: vi.fn()
    }
  }))
}))

describe('Create Issue Handler', () => {
  let mockRuntime: any

  beforeEach(() => {
    mockRuntime = {
      config: {
        get: vi.fn().mockReturnValue('fake-token')
      },
      logging: {
        createLogger: vi.fn(() => ({
          info: vi.fn(),
          error: vi.fn()
        }))
      }
    }
  })

  it('should reject when title is missing', async () => {
    const result = await createIssueHandler({ owner: 'myorg', repo: 'myrepo' }, mockRuntime)

    expect(result.success).toBe(false)
    expect(result.error).toBe('MISSING_TITLE')
  })
})
```

### 7.3 在 OpenClaw 中本地调试

```bash
# 进入技能目录
cd my-awesome-skill

# 编译
npm run build

# 将技能软链到本地扩展目录
ln -s $(pwd) ~/.openclaw/extensions/my-awesome-skill

# 重启 Gateway，让系统重新发现本地扩展
openclaw gateway restart
```

如果你的运行环境不支持软链，也可以直接复制到 `~/.openclaw/extensions/`。

## 9 最佳实践

### 9.1 清晰定义意图

意图名、描述和示例语句要让 Agent 能明显区分不同 Skill 的触发场景。

### 9.2 隔离外部依赖

把 GitHub、Notion、数据库等外部客户端封装到独立模块，别把网络调用散落在多个 handler 里。

### 9.3 记录必要日志

```typescript
const logger = runtime.logging.createLogger('github-issues:create')

logger.info(`Creating issue in ${input.owner}/${input.repo}`)
logger.error(`Failed: ${error.message}`)
```

避免把完整 token、密码、用户隐私内容直接写入日志。

### 9.4 做好配置验证

```typescript
async function initialize(runtime) {
  const token = runtime.config.get('github_token')

  if (!token) {
    throw new Error('GitHub token is required.')
  }
}
```

### 9.5 文档保持完整

README 至少应包含：

- 安装方式
- 配置方法
- 用法示例
- 常见错误排查

## 10 常见问题

### Q: 如何让 Agent 自动选择我的 Skill？

**A:** 关键在 `skill.manifest.json` 里的 `intents`、`description` 和 `examples`。它们越清晰，Agent 越容易把用户请求路由到正确的 Skill。

### Q: Skill 和 Tool 有什么区别？

**A:** Tool 是原子能力，Skill 是业务能力。Tool 更像底层指令，Skill 更像完成一类任务的封装。

### Q: Skill 和 Plugin 有什么区别？

**A:** Skill 面向任务，Plugin 面向平台扩展。新增业务场景优先考虑 Skill；扩展渠道、模型、记忆或底层能力时再考虑 Plugin。

### Q: Skill 可以调用其他 Skill 吗？

**A:** 不建议把 Skill 之间做强依赖。更稳妥的做法是把共享逻辑抽成公共模块，或者由 Agent/工作流层做编排。

### Q: 如何处理认证信息？

**A:** 通过 `runtime.config.get()` 或环境变量读取，不要硬编码到代码或文档示例之外的真实配置文件中。

### Q: Skill 一定支持流式响应吗？

**A:** 不应默认假设所有 Skill 都有统一的流式 API。是否支持流式返回，取决于当前运行时能力和渠道能力，开发时应以实际 SDK/Runtime 为准。

### Q: 如何调试异步问题？

**A:** 在关键步骤打日志，并使用本地扩展加载方式重复验证初始化、配置读取和 handler 执行链路。
