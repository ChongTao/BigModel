# 1 上下文窗口基础

## 1.1 Claude Code 上下文窗口

Claude Code 的上下文窗口大小是 **1,000,000 tokens**。它容纳的是 Claude 在当前会话中“能看到的一切内容”，主要包括：

| 内容来源 | 加载时机 | Token 占用 |
| :--- | :--- | :--- |
| 系统提示词 | 每次会话启动 | 固定占用 |
| `CLAUDE.md` 文件 | 启动时加载 | 取决于文件大小 |
| 自动记忆（Auto memory） | 启动时加载 | 有上限 |
| 对话历史 | 持续累积 | 会不断增长 |
| 工具结果 | 每次工具调用后追加 | 日志和大文件消耗很快 |
| MCP 工具描述、Skills 描述 | 启动时加载 | 中等 |

每个 Claude Code 会话都是从一个新的上下文窗口开始的。真正能跨会话保留下来的内容，主要靠两类机制：

- `CLAUDE.md`：你手动编写的规则、约束和项目说明
- 自动记忆：Claude 在工作中自己提炼出的偏好、习惯和经验

两者都会在新会话开始时被带入上下文，但定位不同：

- `CLAUDE.md` 更像显式规则
- 自动记忆更像 Claude 为当前项目维护的工作笔记

### 1.1.2 会话持久化与本地保存

虽然每个 Claude Code 会话都会从一个新的上下文窗口开始，但会话本身并不是“关掉终端就消失”。

Claude Code 会把当前工作中的会话历史保存在本地，常见会包含：

- 对话消息
- 工具调用记录
- 工具返回结果

这也是为什么下面这些能力能够成立：

- `claude --continue`：继续最近一次会话
- `claude --resume <id>`：恢复指定会话
- `/rewind`：回退到某个检查点

一个实用理解是：

- **新会话**：新的上下文窗口，从头开始处理当前任务
- **恢复旧会话**：继续读取之前已保存的同一段会话历史
- **跨会话延续偏好**：主要依赖 `CLAUDE.md` 和自动记忆，而不是把所有旧会话都拼接进来

另外，在 Claude 准备修改文件之前，通常还会对受影响文件生成快照。  
这也是为什么回退操作不只是在“撤销聊天”，还可能同时恢复代码改动。

### 1.1.1 `CLAUDE.md` 与自动记忆的区别

| 维度 | `CLAUDE.md` | 自动记忆 |
| :--- | :--- | :--- |
| 谁编写 | 你或团队 | Claude |
| 内容类型 | 指令、约束、规范 | 偏好、经验、常用做法 |
| 作用范围 | 项目、用户或组织 | 项目级工作流 |
| 加载方式 | 每次会话加载 | 每次会话加载，且有大小上限 |
| 适合存什么 | 强规则、硬约束、项目背景 | 常用命令、调试经验、偏好习惯 |

## 1.2 上下文衰退（Context Rot）

官方明确指出：随着上下文越来越满，**LLM 的表现会下降**。原因不是窗口“装不下”，而是注意力会被大量旧信息分散，导致 Claude 对当前任务的聚焦能力变差。

常见症状包括：

- 前后矛盾，忘记之前已经确认的决策
- 回答变得模糊、重复或绕圈
- 反复询问已经回答过的内容
- 同一个问题在同一会话中被纠正多次

出现这些迹象时，不要硬撑着在当前会话继续纠偏。更合适的做法通常是：

- 用 `/compact` 压缩会话摘要，继续当前任务
- 用 `/clear` 清空对话历史，切换到全新任务

## 1.3 上下文处理命令

### 1.3.1 `/compact`：压缩当前会话

`/compact` 会把当前对话历史压缩成摘要，释放上下文空间，同时尽量保留当前任务继续所需的关键信息。

```sh
/compact
```

也可以附带压缩指令，告诉 Claude 摘要时重点保留哪些内容：

```sh
/compact 重点保留数据库 schema 决策和已完成接口列表
```

当上下文使用率超过阈值时，Claude Code 也可能自动触发压缩。可以在 `settings.json` 中配置：

```json
{
  "autoCompactThreshold": 0.85
}
```

### 1.3.2 `/clear`：清空当前会话

`/clear` 会完全清空当前会话的对话历史，相当于从零开始一个新会话。

```sh
/clear
```

可以简单理解为：

- `/compact`：保留摘要，适合继续同一个任务
- `/clear`：彻底清空，适合切换到新任务

## 1.4 自动记忆（Auto Memory）

自动记忆是 Claude Code 的跨会话记忆机制，默认开启。它的目标不是记录完整聊天记录，而是提炼出“未来还值得带回来的信息”。

Claude 通常会把下面这类内容记成自动记忆：

- 你经常使用的构建、测试、调试命令
- 你反复强调的代码风格或修复偏好
- 项目里常见的坑点和已验证过的处理方式
- 某些本地运行依赖，例如 Redis、特定启动顺序等

可以把自动记忆理解为：**Claude 为当前项目自己维护的一份工作笔记**。

### 1.4.1 记忆存储在哪里

自动记忆通常是 **按项目 / 按 git 仓库隔离** 的。也就是说：

- 每个仓库有自己的记忆
- 同一仓库的多个 worktree 通常共享同一份记忆
- 不同仓库之间的记忆不会混在一起

当前文档里采用的默认路径口径是：

```text
~/.claude/projects/<项目路径>/memory/
```

也可以在 `~/.claude/settings.json` 中指定自动记忆目录：

```json
{
  "autoMemoryDirectory": "~/.claude/memory/"
}
```

从使用者角度看，更重要的不是具体文件名，而是下面三点：

- 底层存储是项目级隔离的
- ` /memory ` 展示的是聚合后的记忆视图
- Claude 会持续维护这份记忆，而不是要求你每次手工整理

### 1.4.2 记忆通常存什么，不存什么

自动记忆适合保存的是“未来还有复用价值”的内容，例如：

- 常用命令：`pnpm test`、`pytest -k auth`、本地启动脚本
- 偏好习惯：更偏向最小改动、先写测试再修复、统一错误返回结构
- 调试经验：某个服务启动前必须先拉起依赖、某个目录需要单独构建
- 项目惯例：某类改动优先走哪条路径、某模块常见问题怎么排查

自动记忆不适合保存的是：

- 只对当前一次任务有用的临时信息
- 大段日志、完整命令输出、整段代码
- 本该写进 `CLAUDE.md` 的团队规则或强约束
- 你不希望跨会话保留的临时决定

### 1.4.3 Claude 如何处理记忆

Claude 并不是每次对话都会写记忆。它会先判断信息是否值得跨会话保留。可以把处理流程理解成四步：

1. **观察信号**
   Claude 会关注一些明显信号，例如：
   - 你直接说“记住”
   - 你反复纠正同一类行为
   - 某条命令或工作流在多个任务中重复出现

2. **判断是否有长期价值**
   如果信息只对当前任务有用，通常不会写入。只有 Claude 判断“以后大概率还用得上”时，才会进入候选记忆。

3. **提炼与压缩**
   Claude 不会把原始对话整段存进去，而是会压缩成简短、可加载、可检索的记忆条目。

4. **去重、合并或替换**
   如果新记忆和旧记忆重复或冲突，Claude 通常会重写、合并或替换，而不是无限叠加。

写入时，终端常会显示：

```text
Writing memory
```

这表示 Claude 正在把当前会话中的某条信息提炼成记忆。

### 1.4.4 记忆如何被使用

自动记忆的使用大致分成三层。

#### 1.4.4.1 会话启动时预加载

每次新会话开始时，Claude Code 会把自动记忆和 `CLAUDE.md` 一起带入上下文。

当前文档使用的口径是：

- 自动记忆加载前 200 行或 25KB

这也是为什么记忆必须被压缩成简洁条目，而不能变成大段原始历史。

#### 1.4.4.2 任务相关时被召回

当当前任务和某些旧记忆相关时，Claude 会把相关条目拉回当前推理中使用。终端常会显示：

```text
Recalled memory
```

这表示 Claude 认为某条旧记忆对当前任务有帮助，例如：

- 你们项目常用的测试命令
- 某个模块以往的调试经验
- 你反复强调过的风格偏好

#### 1.4.4.3 影响回答和工具选择

记忆不会像权限配置那样“硬控制” Claude，但会明显影响它的行为倾向。例如：

- 优先尝试你们常用的构建命令
- 遇到类似问题时先采用你以前认可过的修复方式
- 生成代码和文档时更贴近你反复强调的风格

要注意这一点：**自动记忆是上下文，不是强制配置**。  
它会影响 Claude，但不会像 `settings.json` 那样形成硬限制，也不会覆盖你在当前会话里给出的更新、更明确的指令。

### 1.4.5 查看与编辑记忆（`/memory`）

可以用 `/memory` 查看和编辑当前可见的记忆内容，包括各级 `CLAUDE.md` 和自动记忆：

```sh
/memory
```

这个界面更像是一个“记忆管理视图”，而不是单一底层文件。你可以用它来：

- 查看 Claude 当前会带入哪些记忆
- 删除错误、过时或不希望继续保留的条目
- 判断某条内容应该保留在自动记忆里，还是升级为 `CLAUDE.md`

如果想临时禁用自动记忆，可以用环境变量：

```sh
CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 claude
```

### 1.4.6 主动让 Claude 记住内容

你可以直接在对话中告诉 Claude 把某条信息写进自动记忆，例如：

```markdown
记住：我们的 API 测试需要本地运行一个 Redis 实例
以后请记住：处理错误时，统一返回 { code, message } 结构
```

如果你希望这不是“偏好”而是“明确规则”，更合适的表达方式通常是：

```markdown
把这条规则写进 CLAUDE.md
```

可以简单区分为：

- 自动记忆：偏好、习惯、常用方式
- `CLAUDE.md`：团队规则、项目约束、强制工作流

# 2 上下文管理实践

## 2.1 控制 Token 消耗

大文件读取和长命令输出是 token 的主要消耗来源。下面这些做法通常更省上下文：

| 场景 | 推荐做法 |
| :--- | :--- |
| 读取大型日志 | 只看关键片段，如最后 50 行错误日志 |
| 多文件批量操作 | 拆成多个独立会话，每次只处理相关文件 |
| 长时间调同一个问题 | 出现上下文衰退迹象时及时 `/compact` |
| 任务切换 | 新任务开始前执行 `/clear` |
| `CLAUDE.md` 过长 | 超过 100 行时考虑拆到 `.claude/rules/` |

## 2.2 会话管理策略

### 2.2.1 单个会话只专注一个任务

把多个不相关任务塞进同一个长会话，会让大量旧信息持续占用注意力，降低 Claude 对当前问题的聚焦能力。

### 2.2.2 新会话开始时补必要背景

`/clear` 之后，Claude 不再记得刚才那段对话历史。这时可以在新会话开始时补一句非常短的背景：

```text
我们正在实现用户认证模块，已完成 JWT 生成，现在需要实现 refresh token 逻辑。
```

### 2.2.3 把重要决策固化到 `CLAUDE.md`

如果某条规则是项目长期约束，而不是个人偏好，就不要只依赖自动记忆。  
更稳妥的做法是要求 Claude 把它写进 `CLAUDE.md`，这样新会话和新成员也能继承。

## 2.3 Claude 的任务循环

从使用效果看，Claude Code 处理一项任务通常不是“一次回答完”，而是一个持续收敛的循环：

1. 接收目标，理解你要它完成什么
2. 读取约束，加载系统提示、`CLAUDE.md`、相关记忆和必要说明
3. 收集证据，读文件、搜代码、运行必要命令
4. 形成方案，尽量选择最小可行改动路径
5. 执行与验证，修改文件、运行检查或测试
6. 失败后调整，缩小范围、换工具、重试或标记阻塞
7. 输出结果，说明改了什么、还有什么风险或未完成项

理解这个循环很有用，因为它解释了 Claude Code 为什么经常会先读文件、再动手、最后验证，而不是一上来就直接改。

### 2.3.1 失败恢复与收敛

Claude Code 的很多表现，本质上都来自这个循环里的“失败后收敛”机制。例如：

- **工具失败**：会先读错误信息，再重试、换路径或缩小查询范围
- **上下文过载**：会建议你 `/compact` 或新开会话
- **改动风险过高**：会倾向先做更小的改动，而不是直接大重构
- **结果不确定**：会标注假设、未验证项和需要你确认的决策点
