# 1 上下文窗口基础

## 1.1 Claude Code上下文窗口

Claude Code 的上下文窗口大小为 **1,000,000 tokens（100 万 token）**。它容纳了 Claude 在当前会话中"看到"的一切内容：

| 内容来源                           | 加载时机                       | Token 占用          |
| :--------------------------------- | :----------------------------- | :------------------ |
| 系统提示词（Claude Code 行为规范） | 每次会话启动                   | 固定占用            |
| CLAUDE.md 文件                     | 启动时完整加载                 | 取决于文件大小      |
| 自动记忆（Auto memory）            | 启动时加载（前 200 行或 25KB） | 有上限              |
| 对话历史                           | 累积增长                       | 随会话持续增加      |
| 工具结果（文件读取、命令输出等）   | 每次工具调用后追加             | 日志/大文件消耗极快 |
| MCP 工具名称、Skills 描述等        | 启动时加载                     | 中等                |

每个 Claude Code 会话都从一个全新的上下文窗口开始。两种机制可以跨会话传递知识：

- **CLAUDE.md 文件**：你编写的指令，为 Claude 提供持久上下文
- **自动记忆**：Claude 根据你的更正和偏好自己编写的笔记

Claude Code 有两个互补的记忆系统。两者都在每次对话开始时加载。Claude 将它们视为上下文，而不是强制配置。

CLAUDE.md 与自动记忆区别：

|              | CLAUDE.md 文件             | 自动记忆                              |
| :----------- | :------------------------- | :------------------------------------ |
| **谁编写**   | 你                         | Claude                                |
| **包含内容** | 指令和规则                 | 学习和模式                            |
| **范围**     | 项目、用户或组织           | 每个工作树                            |
| **加载到**   | 每个会话                   | 每个会话（前 200 行或 25KB）          |
| **用于**     | 编码标准、工作流、项目架构 | 构建命令、调试见解、Claude 发现的偏好 |



## 1.2 上下文衰退（Context rot）

官方明确指出：随着上下文填满，**LLM 性能会下降**，因为注意力被分散到更多 token 上，旧的、不相关的内容开始干扰当前任务。具体症状包括：

- Claude 开始前后矛盾，忘记之前达成的决策
- 响应变得模糊、笼统，细节减少
- 对同一问题反复询问已经回答过的内容
- 在同一个会话里对同一个问题纠正了两次以上

出现以上症状说明上下文已经"污染"，此时应主动使用 `/compact` 或 `/clear` 处理，而不是继续在同一会话中纠正。

## 1.3 上下文处理命令

### 1.3.1 /compact -- 压缩上下文

`/compact` 将当前会话的对话历史**压缩为摘要**，释放 token 空间，同时保留关键信息继续当前任务。适合在任务进行到一半、上下文已经很长但不想重新开始时使用。

```sh
/compact
```

也可以附加自定义摘要指令，告诉 Claude 压缩时重点保留哪些内容：

```sh
/compact 重点保留数据库 schema 设计决策和已完成的接口列表
```

**自动压缩**：当上下文使用量超过阈值时，Claude Code 会自动触发压缩，终端会显示提示。也可在 settings.json 中配置：

```json
{
  "autoCompactThreshold": 0.85
}
```

### 1.3.2 /clear -- 清空上下文

`/clear` 完全清空当前会话的对话历史，从零开始一个全新会话。适合任务已经完成、准备切换到完全不同的任务时使用。

```sh
/clear
```

> `/compact` 保留摘要继续当前任务；`/clear` 彻底清空，适合切换任务。

## 1.4 自动记忆（Auto memory）

自动记忆是 Claude Code v2.1.59 及以上版本内置的跨会话记忆系统，**默认开启**，无需任何配置。它让 Claude 在工作过程中自主记录发现的构建命令、调试经验、代码风格偏好等信息。

### 1.4.1 工作机制

- Claude 不会每次会话都写记忆，它会判断信息是否值得保留（在未来对话中是否有用）
- 记忆按 git 仓库路径隔离，存储于 `~/.claude/projects/<项目路径>/memory/`
- 同一仓库的所有工作树共享一份自动记忆
- 写入记忆时，终端会显示 "Writing memory"；下次检索时显示 "Recalled memory"

### 1.4.2 查看与编辑记忆（/memory）

使用 `/memory` 命令查看和编辑所有记忆内容，包括各级 CLAUDE.md 和自动记忆：

```sh
/memory
```

在编辑器中可以删除不准确的条目，或使用 Auto memory 开关禁用自动记忆功能。也可以通过环境变量临时禁用：

```sh
CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 claude
```

### 1.4.3 主动让 Claude 记住内容

对话中可以直接让 Claude 写入记忆：

```markdown
记住：我们的 API 测试需要本地运行一个 Redis 实例
以后请记住：处理错误时，统一返回 { code, message } 结构
```

如果想升级为强制规范而非只是偏好记忆，可以说："把这条规则写进 CLAUDE.md"，Claude 会将其写入对应的 CLAUDE.md 文件

在 `~/.claude/settings.json` 中指定记忆存储目录

```json
{
  "autoMemoryDirectory": "~/.claude/memory/"
}
```

# 2 上下文管理实践

## 2.1 控制 Token 消耗

大文件读取和命令输出是 token 的主要消耗来源，以下策略可有效降低占用：

| 场景                   | 推荐做法                                       |
| :--------------------- | :--------------------------------------------- |
| 读取大型日志/输出      | 只请求关键片段，如"显示最后 50 行错误日志"     |
| 多文件批量操作         | 拆分为多个独立会话，每次只处理相关文件         |
| 长时间调试同一问题     | 出现上下文衰退症状时立即 `/compact`            |
| 任务切换               | 新任务开始前执行 `/clear`                      |
| CLAUDE.md 过长         | 超过 100 行时拆分到 `.claude/rules/` 模块化管理 |

## 2.2 会话管理策略

**单个会话专注单一任务**：跨越多个不同任务的长会话会引入大量无关上下文，降低 Claude 对当前任务的专注度。

**新会话时补充必要背景**：`/clear` 后 Claude 不记得之前的决策，可以在新会话开头简短说明：

```
我们正在实现用户认证模块，已完成 JWT 生成，现在需要实现 refresh token 逻辑。
```

**将重要决策固化到 CLAUDE.md**：对话中确定的架构决策、编码约定，让 Claude 写入 CLAUDE.md，确保新会话也能继承。
