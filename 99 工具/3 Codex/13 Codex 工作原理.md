# 13 Codex 工作原理

如果从工程视角看，Codex 不是单纯“问答模型”，而是一套带工具、规则和执行环境的代理系统。

可以粗略理解成：

```text
用户目标
  ↓
规则加载（AGENTS / config / Skills / Memories）
  ↓
上下文收集（文件、目录、选择内容、外部工具）
  ↓
计划与执行（读文件、改代码、跑命令、调用工具）
  ↓
验证与汇总
  ↓
输出结果
```

## 13.1 它为什么比普通聊天更像代理

因为它不只是生成文本，还会：

- 读取文件
- 修改文件
- 执行命令
- 根据权限策略请求审批
- 调用外部工具
- 在复杂任务里开子代理

## 13.2 本地客户端共享什么

Codex 的 CLI、IDE 扩展和 App，本质上共享同一套核心代理能力，差异主要在交互界面和工作流承载形式。

可以理解为：

- CLI：终端协作入口
- IDE：编辑器协作入口
- App：桌面工作台入口

## 13.3 为什么要有这么多自定义层

因为真实工程环境里，不同问题属于不同层：

- `AGENTS.md`：规则
- `Memories`：记忆
- `Skills`：方法
- `MCP`：外部工具
- `Subagents`：并行执行

这些层叠加起来，才让 Codex 不只是一个“会聊天的模型”。

## 13.4 Codex 的基本取舍

Codex 的效果通常来自三个因素共同作用：

- 模型能力
- 上下文组织
- 工程执行环境

所以很多时候，问题不是“模型差”，而是：

- 规则没写清楚
- 上下文给乱了
- 任务没拆对
- 验证链路没建立
