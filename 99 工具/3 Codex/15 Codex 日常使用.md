# 15 Codex 日常使用

Codex 真正好不好用，通常不取决于单次提示词，而取决于你是否把它纳入稳定工作流。

## 15.1 一个顺手的日常流程

### 开始任务前

- 确认当前目录正确
- 看仓库里有没有 `AGENTS.md`
- 明确这次是否允许改文件、跑命令、联网

### 开始任务时

- 先让它总结现状
- 再让它给最小方案
- 需要的话直接执行修改

### 修改后

- 让它做最小验证
- 让它总结风险和剩余问题
- 如果发现高频规则缺失，更新 `AGENTS.md`

## 15.2 提示方式建议

更稳的提示方式通常是：

```text
目标 + 范围 + 约束 + 验证要求
```

例如：

```text
修复这个接口的空指针问题，只改相关模块，改完跑最小测试，并总结风险。
```

## 15.3 高频习惯

- 先读，再改
- 先小改，再扩
- 大任务拆分
- 结果要它自检
- 重复问题沉淀到 AGENTS 或 Skill

## 15.4 常见误区

- 把所有长期规则都写进临时 prompt
- 任务太大却不拆
- 不要求验证
- 一次会话里塞太多无关上下文
- 想让它同时做分析、实现、评审、重构，却没有明确优先级

## 15.5 一句话建议

把 Codex 当成“可配置的工程同事”，而不是一次性问答工具，效果通常会稳定很多。
